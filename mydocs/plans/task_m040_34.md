# Homebrew formula local tap smoke PoC 수행계획서

GitHub Issue: [#34](https://github.com/postmelee/hyper-waterfall/issues/34)
마일스톤: M040

## 목적

publish 완료된 `hyper-waterfall@0.2.0` npm tarball을 기준으로 Homebrew formula 후보를 실제 macOS local tap에서 검증한다. 검증 결과는 public tap 배포 전 위험과 보류 조건을 판단할 수 있을 정도로 재현 가능하게 남긴다.

이번 task는 M040의 두 번째 실행 항목이다. #33에서 npm publish와 post-publish smoke가 완료됐으므로, Homebrew wrapper가 `hyper-waterfall --version`과 `hyper-waterfall doctor --repo .`를 설치 경로에서 실행할 수 있는지 확인한다.

## 배경

Task #23에서 `docs/homebrew-formula-tap-poc.md`에 Homebrew formula/tap PoC 설계와 책임 경계가 정리됐다. 해당 문서는 npm package wrapper를 1순위 접근으로 보고, public tap 배포 전 npm tarball SHA256, local tap install, `--version`, `doctor`, `brew test` 검증을 승인 게이트로 남겼다.

Task #33에서 `hyper-waterfall@0.2.0`이 npm registry에 publish됐고, registry tarball URL은 `https://registry.npmjs.org/hyper-waterfall/-/hyper-waterfall-0.2.0.tgz`로 확인됐다. 이제 이 tarball의 SHA256과 Homebrew local tap 동작을 실제 검증한다.

## 설계 방향

- formula는 npm registry tarball URL, SHA256, `depends_on "node"`, `std_npm_args` 기반 설치를 사용한다.
- formula는 CLI executable 설치만 담당하고, GitHub Release/tag, `templates/manifest.json`, migration guide, 적용 저장소 승인 게이트를 재정의하지 않는다.
- local tap과 tarball 검증 산출물은 `/private/tmp` 아래에 만들고, repository에는 재현 가능한 명령과 결과만 남긴다.
- Homebrew 명령은 사용자의 로컬 Homebrew tap/install 상태를 변경할 수 있으므로, 구현계획서 승인 후 Stage 실행 시 필요한 명령 범위와 정리 방침을 다시 명시한다.
- 실제 smoke 결과는 최종 보고서에 남기고, #35 public tap 배포 판단에 바로 재사용할 수 있도록 별도 검증 문서 또는 기존 PoC 문서 갱신을 검토한다.

## 잠정 단계

- **Stage 1 — npm tarball 기준과 로컬 환경 확인**
  - `npm view`와 tarball download 기준으로 version, tarball URL, SHA256을 확인한다.
  - `brew --version`, `brew config`, `node --version` 등 local Homebrew smoke 가능 조건을 확인한다.
  - #33 publish 결과와 실제 registry metadata가 충돌하지 않는지 확인한다.
- **Stage 2 — Homebrew formula 후보와 local tap 준비**
  - npm tarball wrapper formula를 작성하고 `url`, `sha256`, `depends_on`, `install`, `test do` 범위를 고정한다.
  - `/private/tmp` 아래 local tap workspace를 만들고 public tap과 분리된 이름으로 검증한다.
  - formula가 manifest나 migration guide를 복제하지 않는지 검토한다.
- **Stage 3 — Homebrew audit/install/smoke 검증**
  - 적용 가능한 `brew audit` 명령을 실행한다.
  - local tap에서 `brew install --build-from-source`를 실행한다.
  - 설치된 `hyper-waterfall --version`, `hyper-waterfall doctor --repo .`, `brew test`를 확인한다.
  - smoke 전후 `git status --short`로 repository 파일 변경 없음 여부를 확인한다.
- **Stage 4 — 결과 문서화와 public tap readiness 판단**
  - SHA256, formula 후보, audit/install/smoke 결과, 실패 또는 보류 항목을 기록한다.
  - public tap 배포 가능 여부와 #35로 넘길 조건을 정리한다.
  - 오늘할일, 단계 보고서, 최종 보고서를 정리한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `npm view hyper-waterfall@0.2.0 version`
  - `npm view hyper-waterfall@0.2.0 dist.tarball`
  - `curl -L https://registry.npmjs.org/hyper-waterfall/-/hyper-waterfall-0.2.0.tgz -o /private/tmp/.../hyper-waterfall-0.2.0.tgz`
  - `shasum -a 256 /private/tmp/.../hyper-waterfall-0.2.0.tgz`
  - `brew --version`
  - `brew config`
- Stage 2
  - `test -f /private/tmp/.../Formula/hyper-waterfall.rb`
  - `ruby -c /private/tmp/.../Formula/hyper-waterfall.rb`
  - formula 수동 검토: URL, SHA256, `depends_on "node"`, `std_npm_args`, `test do`
- Stage 3
  - `brew audit --new --formula ...` 또는 적용 가능한 대체 audit 명령
  - `brew install --build-from-source ...`
  - `hyper-waterfall --version`
  - `hyper-waterfall doctor --repo .`
  - `brew test ...`
  - `git status --short`
- Stage 4
  - `rg -n 'Homebrew|formula|tap|SHA256|brew audit|brew install|doctor|public tap' docs mydocs/tech mydocs/report mydocs/working`
  - `git diff --check`
  - `git status --short --branch`

### 통합 검증

- npm tarball SHA256이 재현 가능하게 기록된다.
- local tap formula가 `hyper-waterfall@0.2.0` CLI를 설치한다.
- `hyper-waterfall --version`이 `0.2.0`을 출력한다.
- `hyper-waterfall doctor --repo .`가 실행되고 repository 파일을 수정하지 않는다.
- formula가 manifest, migration guide, checksum 정책을 복제하지 않는다.
- public tap 배포 가능 여부와 보류 조건이 최종 보고서에 정리된다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **Homebrew 로컬 상태 변경**: `brew tap`, `brew install`, `brew test`는 로컬 Homebrew prefix와 tap 목록을 변경할 수 있다. Stage 실행 전 명령 범위와 정리 방침을 구현계획서에 고정한다.
- **network/registry 변동성**: npm tarball download, Homebrew audit, dependency fetch는 네트워크 상태에 영향을 받을 수 있다. 실패 시 명령, 시각, stderr 요약과 재시도 여부를 기록한다.
- **formula layout 불일치**: npm package의 `bin` layout과 Homebrew `std_npm_args` 설치 결과가 다를 수 있다. install 후 실제 executable 경로와 `brew test`로 확인한다.
- **doctor exit code와 출력 해석**: 빈 임시 저장소 또는 현재 framework 저장소에서 diagnostic이 나올 수 있다. smoke의 핵심은 command 실행과 read-only 동작이며, lifecycle 정합성 평가는 별도 범위로 둔다.
- **public tap 오인**: 이번 task의 local tap은 public 배포가 아니다. README 설치 안내, tap 저장소 push, Homebrew core 제출은 #35 이후 별도 승인으로만 진행한다.
