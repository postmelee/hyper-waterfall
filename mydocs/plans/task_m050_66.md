# Task M050 #66 수행 계획서

GitHub Issue: [#66](https://github.com/postmelee/hyper-waterfall/issues/66)
마일스톤: M050

## 목적

README를 다국어 사용자가 Hyper-Waterfall을 이해하고 바로 적용할 수 있는 진입 문서로 재구성한다. 기본 README는 영어(`en`)로 전환하고, 기존 한국어 내용은 `README.ko.md`로 보존·개편하며, 중국어 간체 사용자를 위한 `README.zh-CN.md`를 제공한다.

이번 task의 핵심은 단순 번역이 아니라 적용자 중심의 정보 구조 재정렬이다. README는 먼저 "무엇인지", "바로 적용하는 방법", "언제 쓰면 좋은지", "도입 후 무엇이 바뀌는지"를 짧게 보여주고, 철학·프롬프트 가이드·dogfooding 사례는 하단 appendix 성격으로 둔다.

## 배경

#65에서 M050 다국어 정책을 정리했고, README와 사용자 진입 프롬프트는 locale 대상이라고 확정했다. 현재 README는 586줄이며 방법론 설명, 가치 제안, 적용 절차, lifecycle, CLI, Homebrew, 프롬프트 가이드가 한 문서에 모두 들어 있다.

사전 분석 결과, 현재 README는 "적용하려는 사람" 기준으로 다음 문제가 있다.

- `바로 적용하기`와 `새 저장소에 빠르게 적용하기`가 같은 적용 프롬프트를 반복한다.
- `적용하면 작업 방식이 이렇게 바뀝니다`, `적용하면 바로 달라지는 것`, `왜 이 저장소를 써야 하는가`가 승인 게이트, 작업 기억, PR 리뷰, 컨텍스트 경량화를 반복 설명한다.
- `타스크 진행 절차`와 `타스크 사이클`이 같은 흐름을 두 방식으로 길게 설명한다.
- 두 번째 H1인 `# postmelee/hyper-waterfall`이 문서 outline을 끊는다.
- 프롬프트 가이드 상세는 이미 `<details>`로 접혀 있으므로 유지하되, 하단 appendix 성격을 분명히 하는 편이 좋다.

사용자와 합의한 개편 방향은 아래 순서다.

1. What: Hyper-Waterfall이 무엇인지 5줄
2. Quick Start: 적용 프롬프트 1개, "AI가 먼저 보고할 것" 3개
3. When to Use: 이 방법론이 맞는 경우 / 과한 경우
4. What Changes: Issue, plan, stage report, PR 흐름 요약
5. Generated Structure: 대상 저장소에 생기는 파일 축약 tree
6. Maintainer Details: lifecycle, update, CLI, Homebrew는 접기 또는 별도 문서 링크
7. Philosophy / Prompt Guide / Dogfooding: 하단 appendix

## 범위

### 포함

- 루트 `README.md`를 영어 기본 문서로 전환
- `README.ko.md`에 기존 한국어 README의 핵심 내용을 보존하면서 적용자 중심 구조로 재구성
- `README.zh-CN.md` 중국어 간체 문서 작성
- 세 README 상단에 언어 링크 추가
- 각 README에 해당 언어 사용자가 그대로 복사할 수 있는 적용 시작 프롬프트 제공
- README 내부 링크, heading, 주요 anchor 검증
- `docs/localization.md`의 `en`, `ko`, `zh-CN`, placeholder 보존, canonical 기준과 충돌하지 않게 정렬

### 제외

- `templates/` locale pack 전체 작성
- `templates/manifest.json` locale 확장 구현
- CLI locale 선택 기능 구현
- manual 또는 Skill 전체 번역
- 릴리스 생성
- #70에서 다룰 신규 적용/update workflow의 locale 저장·전환 로직 구현

## 설계 방향

- README의 주 독자는 "내 저장소에 이 방법론을 적용할지 판단하는 사용자"로 둔다. 내부 유지보수자용 세부 기준은 `docs/`, `templates/mydocs/manual/`, `mydocs/` 문서로 넘긴다.
- 루트 `README.md`는 기본 locale인 `en`으로 작성한다.
- 기존 한국어 본문은 버리지 않고 `README.ko.md`에 재구성해 보존한다. 한국어 문서는 영어 README의 단순 번역이 아니라, 기존 dogfooding 맥락과 설명 밀도를 유지하되 중복을 줄인다.
- `README.zh-CN.md`는 영어 기본 문서와 한국어 원문을 함께 참조해 중국어 간체로 작성한다. `Issue`, `PR`, `branch`, `milestone`, placeholder, 명령어는 구조적 계약으로 보존한다.
- 모든 README는 같은 상위 섹션 구조를 공유한다. 단, 문장 길이와 예시는 각 언어 독자가 자연스럽게 이해할 수 있게 조정한다.
- `Philosophy / Prompt Guide / Dogfooding`은 README 하단 appendix에 둔다. 프롬프트 가이드 상세는 `<details>`로 유지하되, 특정 모델명처럼 빠르게 낡는 표현은 가능한 한 모델 중립적으로 정리한다.
- `docs/agent-entrypoint.md`는 agent-facing lifecycle 진입점으로 유지한다. 사용자가 복사할 적용 시작 프롬프트는 각 README에 locale별로 제공한다. `docs/agent-entrypoint.md` 본문 개편이 필요하면 최소 링크나 안내만 검토하고, locale 선택 workflow 자체는 #70으로 넘긴다.
- README 본문에서 npm CLI, Homebrew, plugin 등 배포 채널 설명은 Quick Start 뒤의 핵심 흐름을 방해하지 않도록 접기나 별도 문서 링크 중심으로 정리한다.

## 잠정 단계

- **Stage 1 — 한국어 README 구조 재정렬**
  - 기존 `README.md`의 핵심 내용을 중복 제거 기준으로 분류하고, `README.ko.md`를 사용자 적용 흐름 중심 구조로 작성한다.
  - 프롬프트 가이드와 철학/dogfooding은 하단 appendix로 유지한다.
  - 검증 관점: 기존 한국어 README의 핵심 메시지, 적용 프롬프트, 운영 흐름, dogfooding 링크가 손실되지 않아야 한다.
- **Stage 2 — 영어 기본 README 전환**
  - `README.ko.md` 구조를 기준으로 루트 `README.md`를 영어 기본 문서로 작성하고, 언어 링크를 추가한다.
  - 모델명·배포 채널·lifecycle 설명은 시간에 덜 취약한 표현으로 조정한다.
  - 검증 관점: `README.md`가 영어 기본 문서로 단독 진입 가능해야 한다.
- **Stage 3 — 중국어 간체 README 작성**
  - `README.zh-CN.md`를 작성하고 `README.md`, `README.ko.md`, `README.zh-CN.md` 간 섹션 구조와 적용 프롬프트 의미를 맞춘다.
  - 구조적 계약, placeholder, 명령어, 파일명은 번역하지 않는다.
  - 검증 관점: 중국어 간체 사용자가 README만으로 적용 시작 프롬프트를 복사할 수 있어야 한다.
- **Stage 4 — 링크, anchor, 진입 프롬프트 정합성 검증**
  - 세 README의 언어 링크, 내부 heading, 적용 프롬프트, `docs/agent-entrypoint.md` 링크를 점검한다.
  - 필요 시 `docs/agent-entrypoint.md`에 README 언어 진입 안내를 최소 추가한다.
  - 검증 관점: #66 수용 기준과 #65 localization 정책이 충돌하지 않아야 한다.

### 통합 검증

- `README.md`가 영어 기본 문서로 동작한다.
- `README.ko.md`, `README.zh-CN.md`가 루트 README 언어 링크로 접근 가능하다.
- 각 README의 적용 프롬프트가 해당 언어 사용자에게 바로 사용할 수 있는 형태다.
- 기존 한국어 README의 핵심 내용이 `README.ko.md`에 보존되어 있다.
- `docs/localization.md`의 locale 정책, fallback, placeholder 보존 기준과 충돌하지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **README가 번역 문서 3개로 커져 유지 비용이 증가**: 세 README의 상위 섹션 구조를 맞추고, 후속 #71에서 smoke/migration 검증에 포함할 수 있게 한다.
- **영어 기본화 중 한국어 원문 손실**: Stage 1에서 먼저 `README.ko.md`를 작성해 한국어 핵심 내용을 보존한 뒤 영어/중국어 문서를 작성한다.
- **철학 설명이 다시 길어지는 문제**: 철학, prompt guide, dogfooding은 appendix로 제한하고 적용자의 첫 흐름을 방해하지 않게 한다.
- **#67/#70 범위 침범**: template locale pack 구조, manifest/update 구현, locale 선택 저장 방식은 이번 task에서 구현하지 않는다.
- **번역 중 구조적 계약 손상**: placeholder, branch, filename, command, GitHub vocabulary는 #65 정책에 따라 번역하지 않는다.
- **중국어 간체 품질 위험**: `zh-CN` 문서는 영어 기본 문서와 한국어 원문을 함께 대조하고, 의미가 흔들리는 용어는 Stage 보고서에 기록한다.

## 승인 요청 사항

- #66을 기존 열린 이슈로 사용하고 새 이슈를 만들지 않는 방향
- README를 적용자 중심 구조로 먼저 재구성한 뒤 영어 기본/한국어 보존/중국어 간체 문서로 확장하는 단계 구성
- `README.md`, `README.ko.md`, `README.zh-CN.md` sibling 구조와 상단 언어 링크 방식
- 프롬프트 가이드 상세는 삭제하지 않고 하단 appendix의 `<details>`로 유지하는 방향
- `docs/agent-entrypoint.md`는 이번 task에서 최소 링크/안내만 검토하고, locale 선택 workflow 구현은 #70으로 남기는 범위 제한

승인되면 `task_m050_66_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
