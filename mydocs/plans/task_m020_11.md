# task_m020_11.md - 추가 배포 채널 확장안 검토 수행계획서

GitHub Issue: [#11](https://github.com/postmelee/hyper-waterfall/issues/11)
마일스톤: M020

## 목적

Hyper-Waterfall의 canonical 배포 기준을 GitHub Release/tag, `templates/manifest.json`, migration guide로 유지하면서, npm CLI 이후에 고려할 수 있는 추가 배포 채널의 역할과 우선순위를 정리한다.

이번 작업의 결과는 Homebrew, Docker, Codex plugin, Claude plugin이 각각 어떤 사용자 문제를 해결하는지, 어떤 운영 비용을 요구하는지, 다음 마일스톤에서 무엇을 먼저 구현하거나 보류해야 하는지 판단할 수 있는 전략 문서다.

## 배경

#7에서 GitHub Release/tag, manifest, migration guide가 canonical 배포 단위로 정의됐고, #8과 #9에서 신규 적용/기존 업데이트 lifecycle과 update PR 전환 규칙이 정리됐다. #10에서는 npm CLI MVP가 같은 기준을 실행하기 쉽게 만드는 편의 채널로 추가됐다.

M020의 남은 판단은 npm 이후의 확장 채널이다. Homebrew, Docker, Codex plugin, Claude plugin은 설치 경험과 실행 환경을 넓힐 수 있지만, 각 채널이 canonical 원천처럼 보이거나 release/manifest/update protocol과 따로 움직이면 유지보수 비용이 커진다. 따라서 실제 구현보다 먼저 채널별 책임 경계와 우선순위를 문서로 고정한다.

## 범위

### 포함

- 추가 배포 채널 전략 문서 작성
- npm, GitHub Release, Homebrew, Docker, Codex plugin, Claude plugin의 목적과 비목표 비교
- 채널별 사용자 시나리오, 배포 단위, 운영 비용, 검증 부담, 리스크 정리
- 다음 마일스톤에서 먼저 구현할 후보와 보류할 후보 제안
- 추가 채널은 canonical protocol을 실행하는 수단이라는 경계 명시

### 제외

- Homebrew formula 구현
- Docker image 또는 registry publish 구현
- Codex/Claude plugin 실제 패키징과 배포
- npm publish 자동화 또는 release pipeline 구현
- GitHub Actions 기반 자동 릴리스 파이프라인 구현

## 설계 방향

- 새 전략 문서는 `docs/distribution-channels.md`에 둔다. `docs/migrations/`는 버전 간 적용 지침 폴더이므로 배포 채널 전략 문서를 넣지 않는다.
- README에는 canonical 기준과 추가 채널 전략 문서로 이어지는 짧은 설명만 추가하거나 보강한다.
- manual과 SKILL 본문은 이번 task에서 기본적으로 수정하지 않는다. 추가 채널 전략은 방법론 절차 변경이 아니라 배포 전략 판단 자료이기 때문이다.
- 모든 채널 설명은 "GitHub Release/tag + manifest + migration guide가 기준이고, 채널은 그 프로토콜을 실행하거나 발견하기 쉽게 하는 수단"이라는 문장을 기준으로 정렬한다.
- 우선순위는 사용자 가치, 구현 난이도, 운영 비용, 기존 npm CLI와의 중복, update protocol과의 정합성을 함께 비교해 제안한다.
- 실제 배포 구현은 다음 마일스톤 후보로만 남기고, 이번 작업에서는 구현 파일이나 릴리스 설정을 추가하지 않는다.

## 잠정 단계

- **Stage 1 - 기준과 비교 축 확정**
  - 기존 README, agent entrypoint, migration guide, npm CLI 산출물을 읽고 canonical 기준과 채널 비교 축을 확정한다.
  - 비교 축이 이슈 수용 기준의 키워드와 모두 연결되는지 확인한다.
- **Stage 2 - 추가 배포 채널 전략 문서 작성**
  - `docs/distribution-channels.md`를 작성해 Homebrew, Docker, Codex plugin, Claude plugin, npm, GitHub Release의 역할을 비교한다.
  - 채널별 목적, 비목표, 사용자 시나리오, 운영 비용, 검증 부담, 리스크를 표와 짧은 설명으로 정리한다.
- **Stage 3 - README 연결과 우선순위 검증**
  - README에 추가 채널 전략 문서 링크와 "추가 채널은 프로토콜 실행 수단" 경계를 짧게 반영한다.
  - 키워드, 우선순위, 후속 마일스톤 후보, canonical 기준이 문서에서 확인되는지 검증한다.

- README 또는 전략 문서에서 "추가 채널은 프로토콜 실행 수단"이라는 경계가 드러난다.
- Homebrew, Docker, Codex, Claude, npm, GitHub Release 키워드가 확인된다.

- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **채널이 canonical 원천처럼 보일 위험**: Homebrew나 Docker 설명이 release 원천처럼 읽히면 #7~#10의 배포 기준과 충돌한다. 모든 채널을 protocol 실행 수단으로 정렬한다.
- **전략 문서 과확장**: 실제 formula, image, plugin packaging 절차까지 쓰면 구현 범위처럼 보일 수 있다. 이번 task는 비교와 우선순위 판단에 머문다.
- **README 중복 설명 증가**: README에 긴 비교표를 넣으면 핵심 흐름이 흐려질 수 있다. 상세 비교는 전략 문서에 두고 README는 링크와 경계만 둔다.
- **플러그인 명칭 혼동**: Codex/Claude plugin은 실제 배포 형식이 도구별로 다를 수 있다. 이번 문서는 구현 명세가 아니라 후보 채널과 검증 질문을 정리한다.
