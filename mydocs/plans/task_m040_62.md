# Task M040 #62 수행 계획서

GitHub Issue: [#62](https://github.com/postmelee/hyper-waterfall/issues/62)
마일스톤: M040

## 목적

에이전트가 매번 읽는 지침과 하이퍼-워터폴 운영 매뉴얼을 더 작은 진실 원천으로 분산해, Codex와 Claude Code의 공식 권장 구조에 맞게 컨텍스트 부담과 문서 유지보수 비용을 줄인다.

이번 task는 조사된 우선 분산 후보를 실제 문서 구조에 반영한다. 특히 `document_structure_guide.md`의 폴더별 상세 규칙은 각 `templates/mydocs/` 하위 폴더의 `README.md`로 이동하고, PR/타스크/Git/lifecycle 관련 규칙은 역할별 문서로 분리한다.

## 배경

사용자 요청은 “우선 분산 후보를 전부 진행해줘. 1.은 각 폴더의 README.md로 이동해줘.”였다. 직전 조사에서 `document_structure_guide.md`, `pr_process_guide.md`, `task_workflow_guide.md`, `git_workflow_guide.md`, `docs/agent-entrypoint.md`가 분산 후보로 확인됐다.

공식 기준도 같은 방향이다. Codex는 `AGENTS.md`가 큰 경우 중첩 디렉터리로 나누는 방식을 안내하고, Claude Code는 큰 `CLAUDE.md`를 path-scoped rules나 Skill로 분리하는 방식을 권장한다. Codex Skill도 progressive disclosure를 전제로 하므로 반복 절차는 항상 로드되는 지침보다 필요한 시점에 읽히는 문서와 Skill에 두는 편이 맞다.

현재 루트 `AGENTS.md` 자체는 약 60줄로 병목이 아니지만, `templates/mydocs/manual/document_structure_guide.md`는 384줄이고 폴더별 상세 규칙까지 모두 포함한다. 또한 lifecycle/release/update 규칙이 `docs/agent-entrypoint.md`, `task_workflow_guide.md`, `git_workflow_guide.md`, `document_structure_guide.md`에 일부 중복된다.

## 범위

### 포함

- `templates/mydocs/manual/document_structure_guide.md`의 폴더별 상세 규칙을 각 폴더 `README.md`로 이동
- `templates/mydocs/{_templates,orders,plans,working,report,feedback,tech,troubleshootings,pr,manual,skills}/README.md` 생성 또는 갱신
- `templates/mydocs/manual/pr_process_guide.md`를 내부 task PR, 외부 PR 검토, PR 생성 명령 규칙으로 분산
- `templates/mydocs/manual/task_workflow_guide.md`에서 framework lifecycle 규칙 분리
- `templates/mydocs/manual/git_workflow_guide.md`에서 release/tag/update protocol 분리
- `docs/agent-entrypoint.md`를 신규 적용, 기존 업데이트, 업데이트 PR 전환 규칙 중심의 하위 lifecycle 문서로 분산
- `README.md`, `templates/manifest.json`, 관련 매뉴얼의 링크와 설명 정합성 갱신

### 제외

- 제품 코드 변경
- npm/Homebrew 배포 로직 변경
- 새 Skill 추가
- GitHub label 또는 milestone 생성
- 사용자 또는 다른 작업자가 만든 미추적 파일 수정
- 수행계획서 승인 전 실제 매뉴얼 본문 변경

## 설계 방향

- 루트 `AGENTS.md`는 현재처럼 매 턴 로드되어야 하는 핵심 정책, 제약, 인덱스만 유지한다. 이번 task의 주 변경 대상은 큰 매뉴얼과 lifecycle 진입 문서다.
- `document_structure_guide.md`는 `mydocs/` 문서 체계의 entrypoint로 줄이고, 폴더별 세부 규칙은 각 폴더의 `README.md`에 둔다.
- `templates/mydocs/` 하위 폴더 README는 적용 저장소에도 복사되는 운영 기준이므로 placeholder를 보존하고, 실제 task 산출물과 혼동되지 않게 “폴더 규칙 문서” 성격을 명확히 한다.
- lifecycle 관련 규칙은 `docs/lifecycle/` 계열 문서에 단일 진실 원천을 만들고, manual 문서에서는 짧게 참조한다.
- PR 관련 규칙은 내부 task PR 작성, 외부 기여 PR 검토, PR 생성 명령/링크 규칙이 서로 다른 독자를 갖기 때문에 분리한다.
- `templates/manifest.json`은 새로 추가되는 적용 대상 문서를 누락하지 않도록 갱신한다. 디렉터리 단위 항목으로 이미 포함되는지 확인하되, 분산 문서의 update policy와 checksum 정책이 기존 원칙과 모순되지 않게 한다.

## 잠정 단계

- **Stage 1 — 폴더별 README 분산**
  - `document_structure_guide.md`의 폴더별 상세 규칙을 각 `templates/mydocs/` 하위 폴더 README로 이동
  - `document_structure_guide.md`는 요약 표와 각 README 링크 중심으로 축소
  - 폴더별 허용 파일명, 템플릿, 금지 내용, 다음 세션이 복원할 맥락을 유지
- **Stage 2 — PR 처리 가이드 분산**
  - 내부 task PR 작성 규칙, 외부 기여 PR 검토 절차, PR 생성 명령/링크 규칙을 별도 문서로 분리
  - 기존 `pr_process_guide.md`는 entrypoint와 관련 문서 링크 중심으로 정리
  - 외부 PR 검토 Skill과 문서 링크 정합성 확인
- **Stage 3 — task/Git/lifecycle 규칙 분산**
  - `task_workflow_guide.md`에서 framework lifecycle 규칙을 별도 문서로 이동
  - `git_workflow_guide.md`에서 release/tag/update protocol을 별도 문서로 이동
  - `docs/agent-entrypoint.md`를 lifecycle 하위 문서로 분산하고 entrypoint 역할만 유지
- **Stage 4 — README, manifest, 링크 정합성**
  - README의 문서 구조, 핵심 SKILL, 도입 후 작업 흐름 설명을 새 구조와 맞춤
  - `templates/manifest.json`에 새 문서가 적용 대상에 포함되는지 확인 및 필요 시 반영
  - `rg`와 `git diff --check`로 링크, placeholder, 중복 규칙 잔존 여부 검증

  - `git diff --check`

### 통합 검증

- 큰 매뉴얼이 entrypoint와 링크 중심으로 줄어든다.
- 각 `templates/mydocs/` 폴더 README가 해당 폴더의 목적, 허용 파일명, 템플릿, 금지 내용을 담는다.
- lifecycle/release/update 규칙의 단일 진실 원천과 참조 링크가 명확하다.
- `templates/` placeholder가 치환되거나 손상되지 않는다.
- README, manual, Skill 설명이 서로 모순되지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **분산 후 탐색성 저하**: 문서가 많아지면 어디를 읽어야 하는지 모호할 수 있다. entrypoint 문서는 짧은 요약과 링크 표를 유지한다.
- **중복 규칙 잔존**: 분산 후 원문 일부가 남아 서로 다른 기준처럼 보일 수 있다. `rg`로 핵심 키워드를 확인하고 중복은 요약 또는 링크로 바꾼다.
- **적용 manifest 누락**: 새 README나 lifecycle 문서가 release manifest에 반영되지 않으면 적용 저장소에 전달되지 않을 수 있다. manifest 파싱과 대상 경로 확인을 검증에 포함한다.
- **범위 과확장**: 문서 구조 변경 task가 배포 로직이나 새 Skill 추가로 확장될 수 있다. 이번 task는 문서 분산과 링크 정합성에 한정한다.

## 승인 요청 사항

- `document_structure_guide.md`의 폴더별 상세 규칙을 각 `templates/mydocs/*/README.md`로 이동하는 방향
- PR 처리 가이드를 내부 PR, 외부 PR 검토, PR 명령/링크 규칙으로 분리하는 방향
- task/Git 문서의 lifecycle/release/update 내용을 별도 매뉴얼 및 `docs/lifecycle/` 문서로 분리하는 방향
- README와 manifest를 새 문서 구조에 맞춰 갱신하는 범위

승인되면 `task_m040_62_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
