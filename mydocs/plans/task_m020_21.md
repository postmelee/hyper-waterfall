# task_m020_21.md - npm publish 준비와 배포 전 검증 정리 수행계획서

GitHub Issue: [#21](https://github.com/postmelee/hyper-waterfall/issues/21)
마일스톤: M020

## 목적

`hyper-waterfall@0.2.0`을 npm에 publish하기 전에 확인해야 할 package metadata, 포함 파일, smoke test, 문서 안내, 승인 게이트를 고정한다. npm CLI는 Hyper-Waterfall canonical 배포 기준을 대체하지 않고, GitHub Release/tag + `templates/manifest.json` + migration guide를 실행하기 쉽게 하는 편의 채널로 유지한다.

이번 task의 결과는 npm publish 전 검증 체크리스트와 로컬 tarball 기반 검증 절차다. 실제 `npm publish` 실행은 작업지시자가 별도로 명시 승인한 경우에만 수행한다.

## 배경

M020에서 npm CLI MVP가 구현됐고 현재 `package.json`은 `name: hyper-waterfall`, `version: 0.2.0`, `bin.hyper-waterfall: ./bin/hyper-waterfall.js`, `engines.node: >=18`, `scripts.test: node --test`를 포함한다. `files`에는 `bin`, `src`, `templates`, `docs`, `README.md`, `LICENSE`가 지정되어 있다.

README와 `docs/distribution-channels.md`는 `npx hyper-waterfall`을 편의 실행 채널로 설명한다. 다만 npm publish 전에는 사용자가 public registry에서 `npx hyper-waterfall`을 안정적으로 실행할 수 없으므로, publish 전/후 상태와 `v0.2.0` GitHub Release/tag 정합성을 문서에서 분리해야 한다.

선행 작업으로 `docs/releases/v0.2.0.md`가 release/tag 준비와 manifest checksum 정책을 정리했다. 이 task는 그 위에 npm package publish 준비만 얹고, tag 생성, GitHub Release publish, npm publish 실행은 승인 게이트로 남긴다. 메인 worktree에는 기존 미추적 파일 `DESIGN.md`, `README_draft.md`, `opus_report.md`가 있으므로 이번 task는 `/private/tmp/hyper-waterfall-task21` 분리 worktree에서 진행한다.

## 범위

### 포함

- `package.json` metadata, `files`, `bin`, `engines`, `scripts.test` 점검
- `npm pack --dry-run` 또는 동등한 tarball 구성 검증
- 로컬 tarball 기반 CLI smoke 검증 절차 정리
- `node bin/hyper-waterfall.js --version`, `--help` 실행 기준 확인
- README의 `npx hyper-waterfall` 예시와 실제 publish 상태 정합성 점검
- `v0.2.0` GitHub Release/tag와 `hyper-waterfall@0.2.0` publish 순서, 승인 게이트, rollback/republish 리스크 정리

### 제외

- 승인 없는 `npm publish` 실행
- 승인 없는 `git tag v0.2.0`, `git push origin v0.2.0`, `gh release create` 실행
- Homebrew formula 구현
- Docker image 구현
- Codex plugin 또는 Claude plugin 구현
- CLI 기능 확장
- 메인 worktree의 미추적 파일 수정, 이동, 삭제

## 설계 방향

- npm publish 전용 체크리스트는 `docs/releases/v0.2.0-npm-publish.md`로 분리해 `docs/releases/v0.2.0.md`의 GitHub Release/tag 기준 문서와 역할을 나눈다.
- `package.json`은 실제 publish에 필요한 metadata나 포함 범위가 부족할 때만 최소 수정한다. CLI 기능이나 command contract는 이번 task에서 넓히지 않는다.
- `npm pack --dry-run` 결과와 로컬 tarball smoke 절차는 문서에 재현 가능한 명령으로 남긴다. dry-run이 아닌 publish 명령은 실행 후보와 승인 조건으로만 적는다.
- README의 `npx` 예시는 publish 전 상태와 충돌하지 않도록 필요한 경우 "npm publish 이후 사용 가능" 또는 동등한 상태 안내를 보강한다.
- `v0.2.0` GitHub Release/tag가 canonical 기준이고 npm package는 실행 채널이라는 설명을 유지한다.

## 잠정 단계

- **Stage 1 — package metadata와 pack 구성 점검**
  - `package.json`의 publish metadata, `files`, `bin`, `engines`, test script를 확인하고 `npm pack --dry-run`으로 포함 파일을 대조한다.
  - 검증 관점: CLI 실행에 필요한 `bin`, `src`, `templates`, `docs`, README, LICENSE가 package에 포함되는지 확인한다.
- **Stage 2 — npm publish 전 체크리스트와 smoke 절차 작성**
  - `docs/releases/v0.2.0-npm-publish.md`를 신설해 tarball 구성, 로컬 tarball 기반 CLI smoke, publish 전 승인 조건, publish 후 확인, rollback/republish 리스크를 정리한다.
  - 검증 관점: 실제 `npm publish` 없이도 maintainer가 publish 직전 판단에 필요한 정보를 재현할 수 있는지 확인한다.
- **Stage 3 — README와 배포 채널 정합성 보강**
  - README, `docs/distribution-channels.md`, `docs/releases/v0.2.0.md`에서 `npx`, npm CLI, GitHub Release/tag, manifest, migration guide 설명을 대조하고 필요한 최소 문구를 보강한다.
  - 검증 관점: npm package가 canonical 기준처럼 보이지 않고, publish 전/후 사용 가능 상태가 문서에서 혼동되지 않는지 확인한다.
- **Stage 4 — 통합 검증과 publish 승인 경계 확정**
  - 이슈 검증 명령을 실행하고, `npm publish`와 release/tag 생성 명령이 실행되지 않았음을 최종 보고에 남긴다.
  - 검증 관점: `hyper-waterfall@0.2.0` publish 전 확인 항목과 별도 승인 요청 사항이 명확한지 확인한다.

### 통합 검증

- npm publish 전에 확인해야 할 package 포함 범위와 smoke test가 문서화되어 있다.
- `hyper-waterfall@0.2.0` publish가 `v0.2.0` GitHub Release/tag와 어떤 순서와 관계를 갖는지 정리되어 있다.
- README의 `npx` 예시가 실제 publish 상태와 충돌하지 않는다.
- `npm publish`는 작업지시자 별도 승인 전 실행되지 않는다.
- `git status --short --branch`에서 task 산출물과 승인된 변경만 표시된다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **npm publish 되돌리기 한계**: npm은 동일 version 재배포가 제한되고 unpublish 정책도 제약이 크다. publish 전 tarball smoke와 승인 게이트를 문서화해 실수를 줄인다.
- **GitHub Release/tag와 npm version drift**: `v0.2.0` release/tag와 `package.json` version이 어긋나면 사용자가 어떤 기준을 따라야 하는지 혼란이 생긴다. publish 순서와 정합성 검증을 체크리스트에 둔다.
- **package 포함 파일 누락**: `files` 범위가 좁으면 `npx` 실행 시 CLI가 manifest, migration guide, templates를 찾지 못할 수 있다. `npm pack --dry-run`과 로컬 tarball smoke로 확인한다.
- **README 상태 오인**: publish 전에도 `npx` 예시가 안정 경로처럼 읽힐 수 있다. 필요하면 README에 publish 상태와 canonical 기준을 분리해 적는다.
- **승인 없는 배포 실행**: 이번 task에서는 `npm publish`, tag 생성, GitHub Release publish를 실행하지 않는다.
