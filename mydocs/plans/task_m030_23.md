# Task #23 수행계획서 - Homebrew formula/tap PoC

GitHub Issue: [#23](https://github.com/postmelee/hyper-waterfall/issues/23)
마일스톤: M030

## 목적

Homebrew formula/tap PoC의 최소 범위와 검증 경로를 문서로 확정한다. macOS 사용자가 `brew install` 또는 tap 기반 흐름으로 `hyper-waterfall` CLI를 설치하고 `--version`, `doctor` smoke를 실행할 수 있는지 검토하되, Homebrew가 canonical 배포 기준을 대체하지 않는다는 경계를 분명히 한다.

이번 task의 결과는 실제 공개 tap 배포나 release pipeline 구현이 아니라, 후속 구현 또는 보류 판단에 사용할 PoC 설계 문서다.

## 배경

#11의 추가 배포 채널 전략에서 Homebrew는 P1 후보로 정리됐다. 현재 `docs/distribution-channels.md`는 GitHub Release/tag, `templates/manifest.json`, migration guide를 canonical 기준으로 두고, Homebrew를 npm CLI나 release asset을 설치하는 wrapper 계층으로 설명한다.

v0.2.0 GitHub Release/tag와 npm CLI 기준이 정리된 뒤에는 macOS 설치 발견성과 업데이트 경험을 개선할 수 있는 Homebrew 접근을 구체화할 필요가 있다. 다만 formula가 manifest, migration guide, 작업지시자 승인 게이트를 우회하면 Hyper-Waterfall의 배포 기준이 갈라지므로 책임 경계와 보류 조건을 먼저 고정한다.

## 범위

### 포함

- Homebrew formula/tap PoC의 사용자 문제, 비목표, 성공 기준 정리
- tap 저장소, 본 저장소 formula, npm package wrapper, release asset wrapper 접근 비교
- version, checksum, GitHub Release/tag, npm package 참조 방식 비교
- `hyper-waterfall --version`과 `hyper-waterfall doctor --repo .` smoke 검증 경로 정의
- 공개 tap 배포 전 승인 지점, 운영 비용, 보류 조건 정리
- 문서 grep, `git diff --check`, 산출물 존재 확인

### 제외

- v0.2.0 GitHub Release/tag 생성
- npm publish 실행
- Homebrew tap 공개 배포
- 자동 release pipeline 구현
- Docker image 구현
- Codex/Claude plugin 구현
- 승인 없는 formula 또는 tap 저장소 생성

## 설계 방향

- Homebrew PoC 문서는 `docs/homebrew-formula-tap-poc.md` 신규 문서로 둔다.
- `docs/distribution-channels.md`는 Homebrew 후보의 상위 전략 문서로 유지하고, 필요하면 신규 PoC 문서로 연결하는 최소 수정만 한다.
- PoC의 기본 후보는 npm package wrapper와 release asset wrapper를 비교한 뒤, canonical 기준과 운영 비용이 가장 낮은 쪽을 추천하는 방식으로 정리한다.
- formula는 CLI 실행 수단으로만 설명하고, manifest나 migration guide를 formula 내부에서 재정의하지 않는다.
- 실제 public tap 배포 여부, checksum 확정, release pipeline 자동화는 별도 승인 지점으로 남긴다.

### 통합 검증

- Homebrew가 해결하는 사용자 문제와 비목표가 명확하다.
- PoC에서 검증할 설치 경로와 smoke 명령이 명확하다.
- formula가 canonical 기준을 대체하지 않고 CLI 실행 수단으로만 동작한다는 경계가 드러난다.
- tap 저장소, 본 저장소 formula, npm package wrapper, release asset wrapper 접근의 장단점이 비교된다.
- 실제 public 배포 여부는 별도 승인 지점으로 남는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **canonical 기준 혼선**: formula는 GitHub Release/tag, manifest, migration guide를 대체하지 않고 CLI 설치 wrapper로만 정의한다.
- **공개 배포 오인**: 문서 제목과 본문에서 PoC 설계임을 분명히 하고, public tap 배포와 npm publish는 별도 승인으로 남긴다.
- **운영 비용 과소평가**: version, checksum, release asset, npm package, macOS smoke 검증 비용을 접근안 비교 표에 포함한다.
- **후속 pipeline 범위 확장**: 자동 release pipeline은 제외 범위로 고정하고 후속 이슈 후보로만 남긴다.
