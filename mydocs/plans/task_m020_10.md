# task_m020_10.md - npm CLI MVP로 init/update/doctor 제공 수행계획서

GitHub Issue: [#10](https://github.com/postmelee/hyper-waterfall/issues/10)
마일스톤: M020

## 목적

Hyper-Waterfall의 canonical 배포 기준은 GitHub Release/tag, `templates/manifest.json`, migration guide로 유지하되, 사용자가 같은 프로토콜을 더 쉽게 실행할 수 있도록 npm CLI MVP를 추가한다.

이번 작업의 핵심은 `npx hyper-waterfall init`, `npx hyper-waterfall update`, `npx hyper-waterfall doctor`가 적용 저장소의 manifest/version/symlink/template 상태를 읽고, 파일 변경 전 판단 결과와 다음 조치를 사람에게 보여주는 얇은 실행 계층을 만드는 것이다.

## 배경

#7에서 GitHub Release/tag, manifest, version 기록, migration guide 기준이 도입됐고, #8에서 framework lifecycle과 core task workflow의 경계가 정리됐다. #9에서는 기존 적용 저장소 업데이트를 Hyper-Waterfall 버전 업데이트 PR로 전환하는 규칙과 PR 본문 구조가 정리됐다.

이제 M020 로드맵의 마지막 실행 채널로 npm CLI MVP를 추가할 수 있다. 다만 CLI는 배포 프로토콜을 대체하거나 사용자 저장소를 자동으로 덮어쓰는 도구가 아니라, release manifest와 version 기록을 읽어 `init`, `update`, `doctor` 판단 결과를 재현 가능하게 출력하는 편의 계층이어야 한다.

## 범위

### 포함

- npm 패키지 메타데이터와 `bin` 엔트리 초안 추가
- Node.js 표준 라이브러리 기반 CLI MVP 구조 추가
- `init`, `update`, `doctor` 명령의 help, 입력 옵션, 최소 실행 경로 구현
- `templates/manifest.json` 파싱과 필수 필드 검증
- 적용 저장소의 `.hyper-waterfall/version.json` 존재 여부와 파싱 결과 확인
- `init`에서 신규 적용 판단 결과와 placeholder/version 생성 계획 출력
- `update`에서 현재 version, 목표 release/tag, migration guide, manifest diff 후보 출력
- `doctor`에서 manifest source/target, symlink, template, version 상태 점검 결과 출력
- README에 npm 사용 예시를 "편의 실행 채널"로 추가
- 로컬 검증 명령 또는 npm script 추가

### 제외

- npm publish 자동화
- Homebrew, Docker, plugin 배포
- GitHub Release 생성 자동화
- GitHub Actions 기반 업데이트 봇
- 사용자 저장소 파일의 무승인 자동 덮어쓰기
- 완전한 자동 충돌 병합 또는 PR 생성 자동화
- 외부 저장소에 대한 실제 업데이트 수행

## 설계 방향

- CLI는 canonical 배포 단위가 아니라 GitHub Release/tag와 manifest/migration 기준을 실행하기 쉽게 만드는 보조 채널로 정의한다.
- MVP는 외부 npm dependency 없이 Node.js 표준 라이브러리로 구현해 패키지 설치와 검증 부담을 낮춘다.
- 기본 명령은 `hyper-waterfall <command> [options]`로 두고, `npx hyper-waterfall init|update|doctor` 사용을 README에 노출한다.
- `init`과 `update`는 실제 파일 적용 전 판단 결과를 출력하고, 작업지시자 승인과 일반 task workflow로 전환해야 하는 지점을 명확히 보여준다.
- `doctor`는 적용 저장소의 상태를 진단하되, 수정 제안은 출력만 하고 자동 수정하지 않는다.
- manifest의 update policy와 #8/#9에서 정한 출력 필드명을 재사용해 문서와 CLI 결과가 갈라지지 않게 한다.
- `templates/` 안의 `{REPO_SLUG}` 같은 placeholder는 구현이나 검증 중 실제 저장소 값으로 치환하지 않는다.
- CLI 코드와 README 문구가 #7~#9의 lifecycle 설명과 충돌하면 문서 원칙을 우선하고 CLI 범위를 줄인다.

## 잠정 단계

- **Stage 1 — npm 패키지와 CLI 골격 추가**
  - `package.json`, executable `bin`, command dispatcher, help 출력을 추가한다.
  - `init`, `update`, `doctor` 명령이 최소 입력 검증과 help를 반환하는지 확인한다.
- **Stage 2 — manifest/version 읽기와 판단 결과 출력 구현**
  - manifest 파서, version state reader, migration guide 탐색, manifest diff 후보 요약을 구현한다.
  - `init`과 `update`가 #8/#9 문서의 판단 결과 필드와 같은 이름으로 출력하는지 확인한다.
- **Stage 3 — doctor 진단과 비파괴 안전장치 보강**
  - `doctor`가 version 파일, manifest source/target, symlink, template 경로, placeholder 상태를 점검한다.
  - 자동 수정이나 덮어쓰기를 하지 않는 안전장치와 경고 문구를 검증한다.
- **Stage 4 — README와 로컬 검증 정리**
  - README에 `npx hyper-waterfall init`, `update`, `doctor` 예시를 추가하고 canonical 배포 기준과의 관계를 짧게 설명한다.
  - package metadata, CLI smoke, manifest parse, `git diff --check`를 통합 검증한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `node bin/hyper-waterfall.js --help`
  - `node bin/hyper-waterfall.js init --help`
  - `node bin/hyper-waterfall.js update --help`
  - `node bin/hyper-waterfall.js doctor --help`
  - `npm pkg get name bin`
- Stage 2
  - `node bin/hyper-waterfall.js init --repo . --manifest templates/manifest.json --dry-run`
  - `node bin/hyper-waterfall.js update --repo . --manifest templates/manifest.json --from v0.1.0 --to v0.2.0 --dry-run`
  - `ruby -rjson -e 'JSON.parse(File.read("templates/manifest.json"))'`
  - `rg -n '현재 version|목표 release/tag|manifest diff|자동 적용 가능|수동 확인 필요|conflict|승인 요청' src README.md docs templates/mydocs/manual`
- Stage 3
  - `node bin/hyper-waterfall.js doctor --repo . --manifest templates/manifest.json`
  - `git diff --check`
  - `rg -n 'overwrite|자동 수정|덮어쓰기|승인' src README.md docs templates/mydocs/manual`
- Stage 4
  - `npm test` 또는 확정된 smoke 검증 script
  - `node bin/hyper-waterfall.js --version`
  - `rg -n 'npx hyper-waterfall (init|update|doctor)|canonical|GitHub Release|manifest|migration' README.md docs templates/mydocs/manual`
  - `git status --short`
  - `git diff --check`

### 통합 검증

- `npx` 또는 로컬 `node bin/hyper-waterfall.js` 경로로 `init`, `update`, `doctor` MVP를 실행할 수 있다.
- CLI가 `templates/manifest.json`을 읽고 필수 필드 누락을 진단한다.
- CLI가 `.hyper-waterfall/version.json` 존재 여부와 파싱 가능 여부를 진단한다.
- `init`과 `update`는 파일 적용 전 판단 결과와 승인 요청을 출력한다.
- `doctor`는 상태 점검 결과를 출력하되 사용자 수정 파일을 자동 덮어쓰지 않는다.
- README가 npm을 편의 실행 채널로 설명하고 GitHub Release/tag + manifest + migration guide를 canonical 기준으로 유지한다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **CLI가 canonical 기준처럼 보일 위험**: README나 help 문구가 npm을 배포 원천처럼 설명하면 #7~#9의 protocol과 충돌한다. CLI는 편의 실행 채널이고 기준은 release manifest와 migration guide임을 반복해서 둔다.
- **무승인 파일 변경 위험**: `init`과 `update`가 실제 파일을 적용하면 하이퍼-워터폴 승인 게이트를 우회할 수 있다. MVP는 기본적으로 판단 결과 출력과 dry-run 중심으로 두고, 쓰기 동작은 제외한다.
- **MVP 범위 과대화**: 충돌 병합, PR 생성, publish 자동화까지 포함하면 한 task의 검증 단위가 커진다. 이번 task는 로컬 실행 가능한 thin CLI와 진단 결과에 집중한다.
- **문서와 출력 필드 불일치**: #8/#9 문서의 판단 결과 필드와 CLI 출력이 달라지면 운영자가 다시 해석해야 한다. CLI 출력명은 기존 문서 용어를 재사용한다.

## 승인 요청 사항

- npm CLI를 canonical 배포 단위가 아니라 GitHub Release/tag + manifest + migration guide를 실행하기 쉽게 하는 편의 채널로 두는 방향에 동의?
- MVP에서 `init`, `update`, `doctor`는 기본적으로 판단 결과와 승인 요청을 출력하고, 사용자 저장소 파일 자동 덮어쓰기는 제외하는 범위에 동의?
- 위 4단계 진행안과 예상 변경 파일 범위로 구현계획서를 작성해도 되는지 승인 요청한다.

승인되면 `task_m020_10_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
