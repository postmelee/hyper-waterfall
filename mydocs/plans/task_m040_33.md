# npm publish 실행과 post-publish 검증 수행계획서

GitHub Issue: [#33](https://github.com/postmelee/hyper-waterfall/issues/33)
마일스톤: M040

## 목적

`hyper-waterfall@0.2.0`을 npm registry에 실제 publish하고, publish 이후 registry 조회와 `npx` 실행 smoke로 배포 상태를 확인한다.

이번 task는 M040의 첫 실행 항목이다. npm publish가 완료되어야 Homebrew formula smoke와 후속 배포 채널 검증이 실제 설치 경로를 기준으로 진행될 수 있다.

## 배경

Task #21에서 `docs/releases/v0.2.0-npm-publish.md`에 publish 전 검증 절차와 승인 게이트가 정리됐다. Task #30에서 `v0.2.0` GitHub Release/tag 생성이 완료된 상태를 전제로, 이제 npm registry publish와 publish 후 검증을 별도 task로 추적한다.

참고 문서와 기준은 다음과 같다.

- `docs/releases/v0.2.0-npm-publish.md`
- `docs/distribution-channels.md`
- `package.json`
- `templates/manifest.json`

## 범위

### 포함

- `v0.2.0` GitHub Release/tag, `package.json` version, `templates/manifest.json` 기준 정합성 재확인
- `npm test`, `npm pack --dry-run`, local tarball install smoke 재실행
- `npm whoami`와 `npm view hyper-waterfall@0.2.0 version`으로 publish 계정과 기존 registry 상태 확인
- 작업지시자의 별도 명시 승인 후 `npm publish` 실행
- publish 후 `npm view`와 `npx hyper-waterfall@0.2.0` smoke 검증
- 최종 보고서에 publish 승인 지점, publish commit, npm `gitHead`, registry URL, smoke 결과 기록

### 제외

- Homebrew formula 작성과 public tap 배포
- Codex/Claude plugin packaging 또는 배포 후보 생성
- Docker image 관련 작업
- 자동 release pipeline 구현
- npm 2FA OTP 또는 maintainer 비밀값 기록
- 동일 version이 이미 존재할 때의 강행 publish 또는 unpublish

## 설계 방향

- `docs/releases/v0.2.0-npm-publish.md`의 승인 게이트를 publish 전 체크리스트의 기준으로 사용한다.
- npm package는 canonical 배포 단위가 아니라 GitHub Release/tag, `templates/manifest.json`, migration guide를 실행하기 쉽게 하는 채널로 기록한다.
- `npm publish`는 수행계획서 승인만으로 실행하지 않고, publish 직전 검증 통과 후 작업지시자의 별도 명시 승인을 다시 받는다.
- npm registry에 `hyper-waterfall@0.2.0`이 이미 존재하면 publish를 중단하고 상태 조사와 보고로 전환한다.
- tarball, smoke 대상 저장소, npm install 산출물은 `/private/tmp` 아래 임시 디렉터리에 만들고 repository에는 남기지 않는다.

## 잠정 단계

- **Stage 1 — publish 전 기준 정합성 재확인**
  - `v0.2.0` tag/release, package version, manifest version, publish 후보 commit, npm 계정과 기존 registry 상태 확인
  - 동일 version이 이미 존재하면 publish 단계로 가지 않고 상태 조사로 전환
- **Stage 2 — package와 local tarball smoke 재검증**
  - `npm test`, `npm pack --dry-run`, local tarball install smoke, CLI `--version`/`--help`, 필요 시 `init --dry-run` 확인
  - publish 전 package 산출물이 문서 기준과 일치하는지 확인
- **Stage 3 — publish 승인 게이트와 npm publish 실행**
  - Stage 1-2 결과를 근거로 작업지시자에게 `npm publish` 별도 승인을 요청
  - 승인 후 `npm publish` 실행, OTP가 필요하면 작업지시자가 직접 입력 가능한 방식으로 진행
- **Stage 4 — post-publish registry/npx 검증과 최종 보고**
  - `npm view`와 `npx hyper-waterfall@0.2.0` 검증 결과를 기록
  - 최종 보고서에 승인 지점, publish commit, npm `gitHead`, registry URL, smoke 결과, 후속 Homebrew 선행 조건 충족 여부를 정리

## 검증 계획

### 단계별 검증

- Stage 1
  - `git rev-parse HEAD`
  - `git tag --list v0.2.0`
  - `gh release view v0.2.0`
  - `node -p "require('./package.json').version"`
  - `node -e "const m=require('./templates/manifest.json'); console.log(m.frameworkVersion, m.release && m.release.plannedTag)"`
  - `npm whoami`
  - `npm view hyper-waterfall@0.2.0 version`
- Stage 2
  - `npm test`
  - `npm pack --dry-run`
  - `npm pack --pack-destination "$PACK_DIR"`
  - `npm install "$PACK_DIR/hyper-waterfall-0.2.0.tgz" --prefix "$SMOKE_DIR"`
  - `"$SMOKE_DIR/node_modules/.bin/hyper-waterfall" --version`
  - `"$SMOKE_DIR/node_modules/.bin/hyper-waterfall" --help`
  - `"$SMOKE_DIR/node_modules/.bin/hyper-waterfall" init --repo "$REPO_DIR" --dry-run`
- Stage 3
  - 작업지시자의 `npm publish` 별도 명시 승인 확인
  - `npm publish`
- Stage 4
  - `npm view hyper-waterfall@0.2.0 version`
  - `npm view hyper-waterfall@0.2.0 dist.tarball`
  - `npm view hyper-waterfall@0.2.0 gitHead`
  - `npm view hyper-waterfall dist-tags --json`
  - `npx hyper-waterfall@0.2.0 --version`
  - `npx hyper-waterfall@0.2.0 --help`

### 통합 검증

- `hyper-waterfall@0.2.0`이 npm registry에서 조회된다.
- `npx hyper-waterfall@0.2.0 --version`이 `0.2.0`을 출력한다.
- `npx hyper-waterfall@0.2.0 --help`가 `init`, `update`, `doctor`와 canonical 기준 안내를 출력한다.
- publish 전후 검증 결과와 승인 지점이 최종 보고서에 남아 있다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **동일 version 재배포 불가**: `0.2.0`을 잘못 publish하면 같은 version으로 덮어쓸 수 없다. publish 전 local tarball smoke와 별도 승인 게이트로 대응한다.
- **registry에 이미 존재하는 version**: `hyper-waterfall@0.2.0`이 이미 존재하면 publish를 중단하고 실제 상태를 조사해 보고한다.
- **npm 계정 또는 2FA 문제**: `npm whoami`나 publish 시 OTP가 필요할 수 있다. 계정명과 OTP는 문서에 비밀값을 남기지 않고, 작업지시자가 직접 입력 가능한 흐름으로 진행한다.
- **canonical 기준 drift**: npm `gitHead`가 GitHub Release/tag와 어긋나면 후속 채널 검증 기준이 흔들린다. publish 전후 commit과 `gitHead`를 보고서에 기록한다.
- **네트워크/registry 일시 장애**: npm registry 조회나 `npx` smoke가 실패할 수 있다. 실패 시 명령, 시각, stderr 요약을 기록하고 재시도 여부를 별도 확인한다.

## 승인 요청 사항

- 위 범위와 단계 구성으로 Task #33을 진행해도 되는지 승인 요청한다.
- Stage 1-2 검증이 통과해도 `npm publish`는 별도 명시 승인 전에는 실행하지 않는다.
- 동일 version이 이미 registry에 있으면 publish를 중단하고 상태 조사로 전환하는 방침을 승인 요청한다.

승인되면 `task_m040_33_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
