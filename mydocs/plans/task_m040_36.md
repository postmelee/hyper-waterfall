# Task #36 수행계획서 - Codex/Claude plugin 공통 배포 원칙 정리

GitHub Issue: [#36](https://github.com/postmelee/hyper-waterfall/issues/36)
마일스톤: M040

## 목적

Codex plugin과 Claude plugin을 각각 만들기 전에 두 채널이 공유해야 할 배포 원칙, canonical 참조 경계, hook guardrail 정책, packaging 검증 checklist를 먼저 고정한다.

이번 task의 핵심 결과는 도구별 plugin 구현 task(#37-#40)가 공통으로 따를 원칙 문서와 check matrix다. plugin bundle이 `templates/manifest.json`, migration guide, `templates/mydocs/skills`, `templates/mydocs/manual`, npm CLI의 책임을 복제하거나 새 진실 원천으로 만들지 않도록 책임 경계를 명확히 한다.

## 배경

M040은 Docker를 제외하고 npm, Homebrew, Codex, Claude 채널의 v0.2.0 최종 배포 가능 상태를 만드는 마일스톤이다. #33에서 npm publish, #35에서 Homebrew public tap, #46에서 Homebrew core 제출 보류 판단이 완료됐고, plugin track은 #36의 공통 원칙 정리 뒤 #37-#40에서 도구별 packaging 검증과 배포 후보 smoke로 이어진다.

현재 `docs/distribution-channels.md`는 Codex plugin과 Claude plugin을 P3 후보로 두고, 둘 다 canonical protocol을 발견하거나 실행하기 쉽게 하는 계층으로 제한한다. 다만 실제 packaging 검증에 들어가기 전에는 다음 항목이 아직 충분히 고정돼 있지 않다.

- plugin bundle에 포함할 파일과 링크 또는 fallback으로 둘 파일의 기준
- `templates/mydocs/skills`, `templates/mydocs/manual`, `docs/agent-entrypoint.md`, npm CLI의 책임 경계
- Codex/Claude hook을 사용할 경우 공통 정책과 도구별 adapter를 나누는 기준
- hook이 작업지시자 승인을 자동 대체하지 않고 guardrail로만 동작한다는 표현
- 도구별 packaging 검증 task가 재사용할 check matrix

- 공통 canonical 참조 원칙 문서화

- 필요 시 `docs/distribution-channels.md`, `docs/agent-entrypoint.md`, `README.md`에 공통 원칙 문서 링크 또는 요약 반영

## 설계 방향

- 새 원칙 문서는 plugin 배포 track의 기준 문서로 둔다. 후보 파일명은 `docs/plugin-distribution-principles.md`로 한다.
- canonical 기준은 계속 GitHub Release/tag, `templates/manifest.json`, migration guide, `docs/agent-entrypoint.md`, npm CLI 출력 계약이다. plugin은 발견성, 실행 편의, guardrail 보조 계층으로만 설명한다.
- Skill과 manual 본문은 `templates/mydocs/skills`와 `templates/mydocs/manual`을 진실 원천으로 둔다. plugin bundle이 필요한 경우에도 별도 수정본을 만들지 않고 release 기준 복사, symlink, packaged snapshot, 또는 런타임 참조 중 어떤 방식을 쓰는지 명확히 표시한다.
- hook은 승인 게이트를 대신하지 않는다. 승인 여부처럼 사람 맥락이 필요한 판단은 block/allow 자동화 대상에서 제외하고, 브랜치명, 이슈 번호, 계획서/보고서 존재 여부, 위험 명령 시도처럼 객관적으로 판별 가능한 조건만 후보로 둔다.
- Codex와 Claude의 hook payload, event 이름, 차단 semantics가 다를 가능성이 높으므로, 공통 정책 규칙과 도구별 adapter를 분리하는 방향을 우선 검토한다.
- 공식 문서 확인 결과는 확인일을 남기고, 불확실하거나 spec이 바뀔 수 있는 부분은 도구별 packaging task의 재확인 항목으로 넘긴다.
- 이번 task에서 core Skill이나 manual 본문을 실제로 바꾸지 않는 것을 기본값으로 둔다. 필요성이 발견되면 별도 승인 항목으로 분리한다.

### 통합 검증

- Codex와 Claude plugin이 공통으로 따라야 할 canonical 참조 원칙이 문서화된다.
- plugin이 `templates/manifest.json`, migration guide, Skill 본문을 별도 진실 원천으로 복제하지 않는 기준이 명확하다.
- hook이 Hyper-Waterfall 승인 게이트를 대체하지 않고 보조 guardrail로만 동작한다는 원칙이 명확하다.
- 공통 hook 정책 후보와 도구별 adapter 분리 기준이 정리된다.
- 도구별 packaging 검증 이슈가 사용할 수 있는 checklist가 생긴다.
- 공식 문서 또는 현재 도구 사양 확인 결과와 불확실성이 기록된다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **도구 사양 변동**: Codex와 Claude plugin/hook 사양은 바뀔 수 있다. 확인일과 URL을 남기고, #37-#40에서 재확인하도록 한다.
- **canonical 기준 drift**: plugin bundle에 Skill, manual, manifest, migration guide를 독립 복제하면 release 기준이 갈라진다. 문서에서 포함/참조/fallback 기준을 명시하고 별도 진실 원천 생성을 금지한다.
- **hook 과신**: hook이 승인 여부를 자동 판단하면 하이퍼-워터폴 핵심 규칙을 훼손한다. 승인 자동화 금지와 객관 상태 기반 guardrail 원칙을 분리해 적는다.
- **도구별 차이 은폐**: Codex와 Claude의 packaging/hook semantics가 다를 수 있다. 공통 정책과 adapter를 분리하고, 불확실성은 도구별 packaging 검증 task로 넘긴다.
- **문서 변경 범위 확산**: 원칙 정리 중 core Skill이나 manual 변경 필요성이 발견될 수 있다. 이번 task의 기본 범위에서는 본문 변경을 제외하고, 필요 시 승인 요청 사항으로 분리한다.
