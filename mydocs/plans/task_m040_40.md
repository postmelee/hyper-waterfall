# Task #40 수행계획서 - Claude plugin 배포 후보 생성과 설치 smoke

GitHub Issue: [#40](https://github.com/postmelee/hyper-waterfall/issues/40)
마일스톤: M040

## 목적

Claude plugin packaging 검증 결과를 실제 배포 후보 산출물과 설치 smoke 기록으로 전환한다. 목표는 Claude Code 사용자가 Hyper-Waterfall 진입점과 core Skill 흐름을 plugin을 통해 발견할 수 있는지 확인하고, public 배포 가능 여부 또는 보류 사유를 문서화하는 것이다.

이번 task는 Claude plugin이 Hyper-Waterfall의 새 진실 원천이 되지 않도록 제한한다. 배포 후보는 canonical 문서와 Skill을 재정의하지 않고, `AGENTS.md`, `CLAUDE.md`, `.claude/skills`, `docs/agent-entrypoint.md`, npm CLI dry-run으로 이어지는 얇은 discovery layer로 만든다.

## 배경

#39는 Claude plugin packaging 구조를 검증했고, #40 최소 후보를 hook 없는 thin wrapper Skill 기반 bundle로 인계했다. #39의 핵심 결론은 `.claude-plugin/plugin.json`, `skills/hyper-waterfall/SKILL.md`, README, CHANGELOG를 포함한 local directory 후보를 먼저 만들고, `claude plugin validate`와 directory `--plugin-dir` smoke를 우선한다는 것이다.

#36의 공통 원칙은 plugin을 canonical source가 아닌 발견/실행 계층으로 정의한다. 따라서 이번 task에서도 `templates/manifest.json`, migration guide, `templates/mydocs/skills`, `templates/mydocs/manual`을 plugin 내부에서 새 절차처럼 재작성하지 않는다. Hook guardrail은 승인 자동화 위험이 있으므로 기본 후보에서 제외하고, 필요 시 별도 승인 항목으로 분리한다.

main에는 #37 최종 산출물이 반영되어 있어 Codex plugin 쪽 packaging 원칙도 참고 가능하다. 다만 #38 Codex plugin 배포 후보 생성과 설치 smoke 이슈는 아직 open 상태이므로, #40은 Claude 채널 산출물에만 집중하고 Codex 채널의 최종 smoke 결과와의 비교는 #41에서 다시 확인한다.

## 범위

### 포함

- Claude plugin 배포 후보 디렉터리와 metadata 작성
- hook 없는 thin wrapper Skill 작성
- README/CHANGELOG 또는 동등한 설치·smoke 안내 작성
- Claude Code local validation과 load/discovery smoke 시도 및 결과 기록
- `.claude/skills`, `docs/agent-entrypoint.md`, npm CLI fallback 경로 확인
- public 배포 가능 여부, 보류 사유, 대체 설치 경로 문서화

### 제외

- Codex plugin 배포 후보 생성과 smoke
- Docker, Homebrew, npm publish 변경
- core Skill 본문을 Claude 전용으로 fork하는 작업
- `hooks/hooks.json` 또는 hook script 기본 포함
- 작업지시자 별도 승인 없는 public marketplace 배포
- GitHub Release/tag 생성 또는 배포 자동화

## 설계 방향

- Claude plugin 후보는 `plugins/claude/hyper-waterfall/` 아래에 둔다. 이 디렉터리 자체가 local plugin root가 되고, 내부에 `.claude-plugin/plugin.json`과 `skills/`를 둔다.
- Skill은 thin wrapper로 시작한다. wrapper는 Hyper-Waterfall 절차를 요약해 새 규칙처럼 만들지 않고, 저장소의 `AGENTS.md`, `CLAUDE.md`, `.claude/skills`, `docs/agent-entrypoint.md`, npm CLI dry-run을 읽게 안내한다.
- Hook은 이번 최소 후보에서 제외한다. `PermissionRequest` 자동 allow, 승인 자동화, persistent permission update는 #36/#39 원칙과 충돌하므로 포함하지 않는다.
- 현재 local Claude Code version이 #39 기준 `2.1.111`이면 zip plugin smoke는 보류한다. directory plugin validation과 가능한 non-destructive smoke를 우선하고, zip 또는 public 배포가 필요하면 별도 승인 게이트를 둔다.
- 산출물은 release snapshot이 아니라 source-managed candidate로 둔다. 실제 release asset, marketplace 등록, zip 배포는 smoke 결과와 작업지시자 승인 후 별도 판단한다.

## 리스크

- **Claude Code version 한계**: local version이 zip plugin smoke 요구 기준보다 낮으면 directory validation만 가능할 수 있다. zip/public smoke는 보류 사유로 기록하거나 별도 승인 후 도구 업그레이드를 진행한다.
- **Thin wrapper discoverability 한계**: core Skill snapshot을 포함하지 않으면 오프라인 환경에서 개별 Skill 본문 발견성이 약할 수 있다. release snapshot이 필요하면 별도 승인 항목으로 분리한다.
- **#38 미완료 상태**: Codex plugin 배포 후보 smoke가 아직 open이므로 Codex/Claude 최종 비교는 #41에서 수행한다.
- **Public 배포 권한과 정책 불확실성**: marketplace 또는 registry 공개 배포가 즉시 가능하지 않을 수 있다. 승인 없는 public 배포는 실행하지 않고 대체 설치 경로를 남긴다.
- **Canonical drift**: plugin README나 wrapper Skill이 core Skill 절차를 재작성하면 별도 진실 원천이 생긴다. wrapper는 링크와 진입 지시 중심으로 제한한다.

## 승인 요청 사항

- #40을 hook 없는 thin wrapper Skill 기반 Claude plugin 후보로 진행하는 방향을 승인할지 확인한다.
- 후보 디렉터리를 `plugins/claude/hyper-waterfall/`로 두는 방향을 승인할지 확인한다.
- public 배포는 이번 task 안에서도 별도 승인 전에는 실행하지 않고, 먼저 local validation/load/discovery smoke와 보류 판단을 남기는 방향을 승인할지 확인한다.

승인되면 `task_m040_40_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
