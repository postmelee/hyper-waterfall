# #69 중국어 간체 locale pack 작성 수행계획서

GitHub Issue: [#69](https://github.com/postmelee/hyper-waterfall/issues/69)
마일스톤: M050

## 목적

이번 task의 목적은 Hyper-Waterfall 하네스를 중국어 간체 사용자도 자신의 저장소에 선택 언어로 적용할 수 있도록 `zh-CN` locale pack 본문을 작성하는 것이다.

중국어 간체 pack은 단순 번역 파일 묶음이 아니라 적용 저장소의 에이전트 운영 규칙, GitHub Issue/PR template, `mydocs/` 생성 문서 template, manual, Skill이 같은 언어 경험으로 이어지게 하는 source여야 한다. 동시에 승인 게이트, GitHub Issue 기준 추적, Stage 보고, 최종 보고, PR 게시, placeholder 보존 같은 Hyper-Waterfall 절차 계약은 #68에서 작성한 `en`/`ko` pack과 같은 의미를 유지한다.

## 배경

M050 정책은 초기 지원 locale을 `en`, `ko`, `zh-CN`으로 정했다. #67은 `templates/locales/{locale}/...` mirror 구조와 manifest `localization` 계약을 추가했고, #68은 `en` 기본 pack과 `ko` 원문 보존 pack을 각각 43개 파일로 작성했다.

현재 `templates/manifest.json`은 `supportedLocales`에 `zh-CN`을 포함하지만, 실제 `templates/locales/zh-CN/**` source는 아직 없다. 따라서 `doctor`/`init --dry-run`은 기본 `en` fallback source는 찾을 수 있지만, 선택 locale이 `zh-CN`일 때 사용할 source는 아직 채워지지 않은 상태다. #69는 이 누락을 해소하고, 후속 #70의 locale 선택 workflow 연결과 #71의 다국어 smoke 검증이 실제 `zh-CN` source를 대상으로 진행될 수 있게 만든다.

## 범위

### 포함

- `templates/locales/zh-CN/**` 중국어 간체 locale pack 작성
- #68의 `en`/`ko`와 같은 43개 파일 범위 유지
- GitHub Issue/PR template, `mydocs/_templates`, folder README, manual, Skill의 중국어 간체 표현 정렬
- 중국어 간체 용어 대응표와 번역 판단 메모 작성
- placeholder, branch pattern, filename pattern, command/code block, GitHub 식별자 보존 검증
- `templates/manifest.json`, `docs/localization.md`, `test/cli-smoke.test.js`가 #69 완료 상태와 어긋나는 경우 최소 수정
- `README.zh-CN.md` 또는 `docs/agent-entrypoint.md`의 `zh-CN` 안내가 locale pack 상태와 어긋나는 경우 최소 정합성 확인 및 보정

### 제외

- 중국어 번체 `zh-TW` 또는 기타 중국어권 locale 작성
- 중국어 외 추가 locale 작성
- CLI locale 선택 기능 구현 또는 `.hyper-waterfall/version.json` locale 저장 위치 확정
- 신규 적용/update workflow 실행 연결
- locale별 smoke 검증과 기존 한국어-only 적용 저장소 migration guide 작성
- 릴리스 생성, release checksum 확정, 배포 패키징
- README 다국어 구조 재개편

## 설계 방향

- `zh-CN` pack 경로는 #67 manifest의 `sourcePattern`과 같은 `templates/locales/zh-CN/...` mirror 구조를 따른다.
- 파일 범위는 #68에서 확정한 `en`/`ko` 43개 파일과 동일하게 맞춘다. manifest가 directory source를 가리키므로 일부 파일만 번역하면 mixed locale 위험이 생긴다.
- `en` pack을 기본 비교 기준으로 삼고, `ko` pack은 원문 의미 확인 기준으로 함께 사용한다.
- `Issue`, `PR`, `label`, `milestone`, `commit`, `branch`, `Stage`, `Skill`처럼 GitHub/도구 식별자와 로컬 절차 용어는 중국어 자연어 안에서도 식별성을 잃지 않게 처리한다.
- `{REPO_SLUG}`, `{BASE_BRANCH}`, `{RELEASE_BRANCH}`, `{PR_TEMPLATE_PATH}`, `{issue}`, `{stage}`, `{sha}`, `{head_sha}`, `local/task{N}`, `publish/task{N}` 같은 구조적 계약은 번역하거나 치환하지 않는다.
- `작성 언어` 정책의 중국어 기준 표현은 "使用此仓库选择的 Hyper-Waterfall locale" 의미를 유지한다.
- manual/Skill은 `requiresSemanticReview: true` 대상이므로 절차 의미 보존을 우선한다. 문체보다 승인 조건, 금지 사항, cleanup 조건, fallback 조건이 약해지지 않는지 검토한다.
- 전체 초기 지원 locale이 모두 채워지는 첫 task이므로, Stage 4에서 manifest availability 문구와 smoke test 기대값을 실제 결과에 맞춰 조정할지 판단한다. 단 #70/#71이 남아 있으므로 적용 workflow 완료처럼 과장하지 않는다.

### 통합 검증

- `en`, `ko`, `zh-CN` locale pack 파일 수와 manifest localized entry source 존재 여부가 기대값과 일치한다.
- `zh-CN` pack의 placeholder, branch pattern, filename pattern, command/code block이 번역 또는 치환되지 않는다.
- manual/Skill의 절차 의미가 `en`/`ko` 기준과 충돌하지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **절차 의미 drift**: 중국어 간체 번역에서 승인 게이트나 작업지시자 승인 조건이 약해질 수 있다. `en`/`ko` 대표 파일과 수동 대조하고, 흔들리는 용어는 용어 대응표에 남긴다.
- **도구 식별자 번역 과다**: Issue, PR, branch, label, milestone 같은 플랫폼 용어를 과도하게 번역하면 실제 GitHub UI와 어긋날 수 있다. 식별자는 유지하고 필요한 설명만 중국어로 쓴다.
- **mixed locale pack**: directory source가 일부만 존재하면 적용 결과가 혼합 언어가 된다. #68과 동일한 43개 파일 범위를 맞추고 Stage 4에서 source 누락 0개를 검증한다.
- **availability 표현 과장**: `zh-CN` pack이 작성되어도 #70 workflow 연결과 #71 smoke/migration이 남아 있다. manifest/docs 문구는 pack source 준비 상태와 workflow 완료 상태를 구분한다.
- **문서 범위 확장**: README 다국어 구조나 적용 workflow를 함께 고치면 #69 범위를 넘는다. 불일치 보정만 하고 구조 변경은 후속 이슈로 남긴다.

## 승인 요청 사항

- #69를 위 범위와 4단계로 진행한다.
- `zh-CN` pack은 #68의 `en`/`ko`와 같은 43개 파일 범위로 작성한다.
- manual/Skill은 전체 포함하되 `requiresSemanticReview: true` 기준으로 대표 파일 의미 보존 검증을 수행한다.

- CLI locale 선택 구현, workflow 연결, migration guide, release checksum 확정은 각각 #70, #71 또는 release 범위로 유지한다.
