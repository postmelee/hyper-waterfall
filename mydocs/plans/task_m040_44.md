# Task #44 수행계획서

GitHub Issue: [#44](https://github.com/postmelee/hyper-waterfall/issues/44)
마일스톤: M040

## 목적

manual 문서가 특정 이슈, PR, 릴리즈, 일회성 검증 결과를 계속 누적하는 장소가 되지 않도록, `manual/` 문서의 역할과 분리 기준을 명확히 한다.

이번 task는 alhangeul-macos PR #207에서 추가된 manual 문서 중립성 정책의 취지를 가져오되, Hyper-Waterfall 프레임워크 저장소에 일반적으로 적용 가능한 규칙으로 재정리한다. 특정 적용 저장소, macOS 앱, `mydocs/release/` 같은 현재 구조에 없는 경로는 그대로 가져오지 않는다.

## 배경

현재 `templates/mydocs/manual/document_structure_guide.md`는 `mydocs/` 폴더 역할, 문서 파일명, 중앙 템플릿, GitHub 플랫폼 템플릿, 배포 manifest, Agent Skills 위치 정책을 정의한다. 또한 `manual/` 상세 규칙에서 특정 task의 단계 보고, 완료 보고, 오늘할일 상태표를 두지 않는다고 이미 설명한다.

하지만 manual 문서가 장기 기준과 일회성 사건 기록을 어떻게 구분해야 하는지는 아직 별도 정책으로 충분히 드러나지 않는다. 이 경계가 흐려지면 릴리즈 검토, migration 판단, 특정 실패 사례, stage 결과가 manual 본문에 누적되어 문서가 비대해지고, 다음 작업자가 반복 적용 가능한 규칙과 사건 기록을 구분하기 어려워진다.

## 범위

### 포함

- `templates/mydocs/manual/document_structure_guide.md`에 manual 문서 중립성 또는 책임 경계 정책을 추가한다.
- manual 본문에 둘 내용과 분리해야 할 내용을 본 저장소 구조에 맞게 정리한다.
- 특정 사건 기록의 분리 위치를 `docs/releases/`, `docs/migrations/`, `mydocs/working/`, `mydocs/report/`, `mydocs/tech/`, `mydocs/troubleshootings/` 기준으로 안내한다.
- README와 관련 manual의 링크, 문서 구조 설명, 중복 여부를 확인하고 필요한 최소 보강만 수행한다.

### 제외

- alhangeul-macos 전용 표현, macOS 앱 특화 실패 예시, 특정 deprecation/notarization 사례의 직접 이식.
- `mydocs/release/` 같은 현재 저장소 구조에 없는 신규 폴더 도입.
- 실제 release 생성, migration guide 작성, 배포 채널 구현.
- manual 정책과 무관한 문서 리팩터링.
- 수행계획서 승인 전 `document_structure_guide.md`, README, AGENTS.md 등 운영 문서 본문 수정.

## 설계 방향

- 정책의 1차 위치는 `document_structure_guide.md`로 둔다. 이 문서가 폴더 역할과 문서 산출물 경계를 정의하므로, manual 문서의 중립성 기준도 같은 문맥에서 찾을 수 있어야 한다.
- `AGENTS.md`에는 상세 정책을 중복하지 않는다. `AGENTS.md`는 매 턴 적재되는 정책·제약·인덱스이므로, 상세 기준을 넣으면 프롬프트가 비대해지고 manual과 drift가 생길 수 있다.
- 새 manual 문서는 만들지 않는다. 이번 목적이 manual 비대화를 막는 것이므로, 독립 문서를 추가하기보다 기존 문서 구조 매뉴얼 안에 짧고 반복 가능한 기준을 둔다.
- 원문 PR의 분리 위치는 본 저장소에 맞게 일반화한다. 릴리즈 기록은 `docs/releases/`, migration/update 판단 근거는 `docs/migrations/`, 내부 task 산출물은 `mydocs/working/` 또는 `mydocs/report/`, 조사 근거는 `mydocs/tech/`, 재발 방지 기록은 `mydocs/troubleshootings/`로 안내한다.

## 잠정 단계

- **Stage 1 — manual 중립성 정책 반영**
  - `document_structure_guide.md`에 manual 본문에 둘 내용, 분리할 내용, 분리 위치, 링크 방식 기준을 추가한다.
  - 본 저장소 구조와 맞지 않는 `mydocs/release/` 같은 경로가 들어가지 않았는지 확인한다.
- **Stage 2 — 관련 문서 정합성 보강**
  - README와 관련 manual에서 문서 구조 설명, 책임 경계, 링크가 새 정책과 충돌하지 않는지 확인한다.
  - 필요한 경우 README 또는 `task_workflow_guide.md`에 최소 문구만 보강한다.
- **Stage 3 — 통합 검증과 최종 정리**
  - 문서 검색, symlink 검증, diff 공백 검사를 수행한다.
  - 단계 보고서와 최종 보고서에서 실제 변경 범위, 검증 결과, 잔여 리스크를 정리한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `rg -n 'Manual 문서|manual|중립|특정 이슈|특정 PR|docs/releases|docs/migrations|mydocs/release' templates/mydocs/manual/document_structure_guide.md`
  - `git diff --check`
- Stage 2
  - `rg -n 'document_structure_guide|manual/|문서 구조|docs/releases|docs/migrations' README.md templates/mydocs/manual`
  - README와 관련 manual의 중복/충돌 여부 수동 확인
- Stage 3
  - `rg -n 'manual|중립|특정 이슈|릴리즈|troubleshooting|migration|release' templates/mydocs/manual README.md AGENTS.md docs`
  - `test -L mydocs/manual && test -L mydocs/skills`
  - `git status --short`
  - `git diff --check`

### 통합 검증

- manual 문서에 특정 사건 기록을 누적하지 않는 기준이 명확히 드러난다.
- 특정 이슈/PR/stage/release 검증 결과/장애 기록의 분리 위치가 본 저장소 구조에 맞게 안내된다.
- `AGENTS.md`에 상세 정책이 중복되지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **정책 과잉 문서화**: manual 비대화를 막기 위한 정책 자체가 장황해질 수 있다. 원칙, 분리 대상, 분리 위치, 링크 방식만 남기고 사례는 일반화한다.
- **경로 정책 불일치**: 원문 PR의 `mydocs/release/`를 그대로 가져오면 현재 저장소 구조와 맞지 않는다. 본 저장소의 `docs/releases/`와 `docs/migrations/` 기준으로 바꾼다.
- **AGENTS.md 중복**: 상세 기준을 `AGENTS.md`에 넣으면 매 턴 프롬프트가 비대해질 수 있다. 필요한 경우에도 필수 참조 문서 인덱스 수준만 유지한다.
- **README 보강 범위 확대**: README는 사용자 진입 문서이므로 세부 정책을 길게 넣지 않는다. 링크 또는 한 문장 보강이 필요한지 Stage 2에서 판단한다.

## 승인 요청 사항

- manual 문서 중립성 정책의 1차 반영 위치를 `templates/mydocs/manual/document_structure_guide.md`로 두는 방향.
- `AGENTS.md`에는 상세 정책을 추가하지 않고, 필요 시 기존 인덱스 수준을 유지하는 방향.
- `mydocs/release/`를 만들지 않고 릴리즈 기록은 `docs/releases/`, migration/update 판단 근거는 `docs/migrations/`로 안내하는 방향.
- 위 잠정 단계 3개로 구현계획서를 작성하는 방향.

승인되면 `task_m040_44_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
