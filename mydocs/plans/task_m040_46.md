# Task #46 수행계획서 - Homebrew core 등재 가능성 평가와 제출 준비

GitHub Issue: [#46](https://github.com/postmelee/hyper-waterfall/issues/46)
마일스톤: M040

## 목적

`hyper-waterfall`을 아무 tap 지정 없이 `brew install hyper-waterfall`로 설치할 수 있는지 판단하기 위해 Homebrew core 등재 가능성을 평가한다.

이번 task의 핵심 결과는 Homebrew core 제출 가능/보류 판단, 현재 `hyper-waterfall@0.2.0` formula와 core 요구사항의 gap 목록, 제출을 진행할 경우 필요한 사전 준비 항목과 승인 게이트다. core 제출이 시기상조라면 `brew install postmelee/tap/hyper-waterfall` public tap 경로를 기본 안내로 유지하는 근거를 문서화한다.

## 배경

Task #35에서 `postmelee/homebrew-tap` public tap과 `postmelee/tap/hyper-waterfall` 설치 경로가 공개됐고, `brew audit --new --formula`, `brew install --build-from-source`, `hyper-waterfall --version`, `hyper-waterfall doctor --repo .`, `brew test` smoke가 통과했다.

다만 public tap은 사용자에게 tap namespace를 포함한 설치 명령을 요구한다. 첫 설치부터 `brew install hyper-waterfall` 단독 명령을 지원하려면 Homebrew 기본 저장소인 `homebrew/core` 등재가 필요하다. Homebrew core는 별도 acceptability, notability, formula style, platform support, reviewer process 기준을 적용하므로 public tap 통과 결과를 그대로 core readiness로 간주할 수 없다.

## 범위

### 포함

- Homebrew 공식 문서 기준의 `homebrew/core` acceptability와 제출 절차 확인
- `postmelee/homebrew-tap`의 현재 `hyper-waterfall@0.2.0` formula와 core 요구사항 대조
- self-submitted software의 notability 기준과 현재 repository 지표 확인
- npm tarball wrapper formula가 Homebrew core에 적합한지 검토
- macOS/Linux 지원 범위, `depends_on "node"`, `test do`, audit/test 명령의 core 제출 관점 점검
- `brew install hyper-waterfall` 단독 명령을 달성하기 위한 선택지 정리
- core 제출 가능/보류 판단과 제출 전 checklist 문서화
- public tap 안내를 유지해야 하는 경우의 사용자-facing 설명 정리

### 제외

- 작업지시자 승인 없는 Homebrew core PR 생성
- Homebrew core reviewer 또는 외부 maintainer와의 승인 없는 커뮤니케이션
- `postmelee/homebrew-tap` 삭제, force push, history rewrite
- npm package 재배포, GitHub Release/tag 재생성
- formula 자동 갱신 pipeline 구현
- Codex/Claude plugin packaging 작업

## 설계 방향

- 판단 문서는 Homebrew core 제출 여부를 결정하는 근거 문서로 작성한다. 단순 조사 로그가 아니라 "진행 가능", "조건부 가능", "보류" 중 하나의 결론과 다음 게이트를 남긴다.
- 공식 문서 확인일을 기록하고, 시간이 지나면 재검토가 필요하다는 점을 명시한다.
- `brew install hyper-waterfall` 단독 명령은 public tap만으로 첫 설치 보장이 불가능하다는 전제를 검증하고 문서화한다.
- 현재 public tap formula는 출발점으로 삼되, core 제출용 formula style과 audit 기준은 별도로 대조한다.
- notability나 acceptability가 부족하면 core PR을 만들지 않고, public tap 경로를 기본 설치 안내로 유지하는 쪽을 명확히 선택한다.
- 외부 제출, 외부 커뮤니케이션, public tap 변경은 별도 승인 게이트 뒤로 둔다.

- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **Homebrew 정책 변동**: core acceptability와 notability 기준은 바뀔 수 있다. 확인일과 공식 문서 링크를 남기고, 제출 직전 재확인 대상으로 둔다.
- **외부 제출의 공개성**: Homebrew core PR은 외부 공개 커뮤니케이션이다. 이번 task에서는 승인 없이 PR을 만들지 않고 준비 항목까지만 정리한다.
- **notability 부족 가능성**: self-submitted software 기준을 충족하지 못하면 core 제출은 거절될 가능성이 높다. 이 경우 public tap 경로를 기본 설치 안내로 유지한다.
- **tap과 core의 기준 혼동**: public tap smoke 통과가 core acceptability 통과를 의미하지 않는다. 두 기준을 문서에서 분리한다.
- **검증 환경 제한**: local Homebrew 명령이 환경 문제로 실행되지 않을 수 있다. 실행 불가 시 #35 검증 결과와 현재 공식 기준 대조를 분리해서 기록한다.
