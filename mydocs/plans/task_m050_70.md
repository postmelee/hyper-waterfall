# 적용·업데이트 workflow에 locale 선택 절차 연결 수행계획서

GitHub Issue: [#70](https://github.com/postmelee/hyper-waterfall/issues/70)
마일스톤: M050

## 목적

신규 적용과 기존 업데이트 workflow에서 locale 선택, 기본값, fallback, 기존 locale 보존, 명시적 locale 전환 요청이 같은 규칙으로 판단되도록 연결한다.

이번 task의 목표는 `en`, `ko`, `zh-CN` locale pack이 준비된 상태를 실제 적용 경험으로 연결하는 것이다. AI coding tool, lifecycle 문서, npm CLI dry-run, plugin 안내가 서로 다른 locale 규칙을 말하지 않게 하고, `.hyper-waterfall/version.json`에 선택 locale을 어떻게 기록하고 읽을지 확정한다.

## 배경

#65에서 다국어 정책을 정했고, #67에서 manifest의 localization 계약과 lifecycle 판단 결과 형식을 추가했다. #68과 #69에서는 `en`, `ko`, `zh-CN` locale pack 본문을 채웠다.

현재 문서와 CLI는 locale 판단 항목을 이미 출력하지만, 일부 문서에는 "실제 locale 선택 저장 위치와 workflow 실행 구현은 #70 범위"라고 남아 있다. 또한 `init`은 명시 locale 요청을 받을 CLI 옵션이 없고, `update`는 version state에 locale이 없으면 `unknown`으로 보고만 한다. 따라서 사용자가 신규 적용 시 언어를 고르고, 기존 적용 저장소가 업데이트 중 언어를 임의로 바꾸지 않는 흐름을 완성해야 한다.

## 설계 방향

- 신규 적용 기본 locale은 `templates/manifest.json`의 `localization.defaultLocale`인 `en`으로 유지한다.
- CLI는 dry-run 판단 채널로 유지한다. `init`과 `update`가 파일을 직접 적용하지 않고, 파일 변경 전 승인 요청에 필요한 locale 판단을 더 정확히 출력하게 한다.
- `init --locale <locale>`은 작업지시자의 명시 언어 요청으로 보고, 지원 여부와 source 존재 여부를 먼저 판단한다. 옵션이 없으면 기본 locale을 선택 후보로 제시한다.
- `update`는 `.hyper-waterfall/version.json`의 기존 locale을 기본값으로 보존한다. 명시적 locale 전환 요청이 있으면 전환 후보와 승인 필요성을 별도 항목으로 보고한다.
- version state의 locale 기록은 top-level `locale` 필드를 기본 계약으로 둔다. 기존 호환을 위해 `selectedLocale` 또는 `localization.locale` 읽기는 유지하되, 새 문서와 manifest 예시는 `locale`을 기준으로 한다.
- 선택 locale source가 누락되면 조용히 fallback을 적용하지 않고, manifest의 `missingLocalePolicy`와 entry별 `fallbackLocale` 기준으로 누락과 fallback 후보를 보고한다.
- manual이나 applied template에 반영되는 문구는 `templates/`의 canonical 파일과 `templates/locales/{locale}/...` 대응 파일을 함께 확인한다.

## 잠정 단계

- **Stage 1 — locale 선택·저장 계약 확정**
  - 신규 적용 기본값, 명시 요청, unsupported locale, fallback 후보, 기존 update 보존, locale 전환 요청 기준을 문서화한다.
  - `.hyper-waterfall/version.json`의 top-level `locale` 기록 계약과 기존 필드 읽기 호환성을 확정한다.
  - 검증 관점: #65-#69 산출물과 충돌하지 않는지 확인한다.
- **Stage 2 — CLI dry-run locale 판단 연결**
  - `init`과 `update`에 locale 요청/보존 판단 출력이 실제 옵션과 version state 읽기 결과에 연결되도록 구현한다.
  - unsupported locale, source 누락, fallback 후보, update locale 전환 요청이 사용자-facing 출력에 드러나게 한다.
  - 검증 관점: CLI smoke test로 `en`, `ko`, `zh-CN`, unsupported locale, version state locale 보존을 확인한다.
- **Stage 3 — lifecycle·plugin·manual 문서 정합성 반영**
  - `docs/agent-entrypoint.md`, `docs/lifecycle/*`, `docs/localization.md`, distribution/plugin 문서의 #70 보류 표현을 확정 표현으로 바꾼다.
  - 적용 저장소에 복사되는 manual과 locale mirror 문서가 같은 locale 저장/보존 규칙을 말하도록 정렬한다.
  - manual 변경이 발생하면 README의 "핵심 SKILL"·"도입 후 작업 흐름" 표와 `templates/mydocs/manual/task_workflow_guide.md`의 "SKILL 호출 표시 안내"를 함께 검토·갱신한다.
  - 검증 관점: `rg`로 #70 보류 문구와 서로 다른 locale 규칙이 남지 않았는지 확인한다.
- **Stage 4 — 통합 검증과 최종 정리**
  - CLI 통합 명령과 문서 검색 검증을 수행하고 stage 산출물을 정리한다.
  - 남은 #71 범위인 smoke matrix와 migration guide 항목을 최종 보고서에 명시한다.
  - 검증 관점: `npm test`, `git diff --check`, 주요 CLI dry-run 명령이 통과해야 한다.

### 통합 검증

- 신규 적용 사용자는 CLI 또는 agent entrypoint에서 언어 선택, 기본 영어 적용, unsupported locale 처리, fallback 후보를 이해할 수 있어야 한다.
- 기존 적용 저장소는 update 과정에서 기존 locale이 임의로 바뀌지 않아야 한다.
- locale 전환은 명시 요청과 별도 승인 항목으로 표시되어야 한다.
- CLI/plugin/문서 흐름이 서로 다른 locale 규칙을 안내하지 않아야 한다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **write mode 오해**: CLI가 실제 적용까지 수행하는 것처럼 보이면 lifecycle 승인 게이트와 충돌한다. 이번 task는 dry-run 판단 출력 연결로 제한하고 문서에 비파괴 경계를 유지한다.
- **version state 호환성**: 기존 적용 저장소에는 locale 필드가 없을 수 있다. 기존 update는 `unknown`을 허용하되 새 계약은 top-level `locale`을 기준으로 문서화한다.
- **locale 전환의 부수 효과화**: update 중 locale 전환을 자동 처리하면 사용자 문서 언어가 섞일 수 있다. 전환은 별도 승인 항목으로 분리한다.
- **manual 변경 동기화 비용**: manual 본문을 바꾸면 README와 task workflow 안내까지 정합성을 확인해야 한다. Stage 3에서 이 요구사항을 별도 검증한다.
- **#71 범위 침범**: 실제 다국어 smoke matrix와 migration guide 작성은 #71 범위다. 이번 task는 workflow 연결과 판단 출력까지만 완료한다.

## 승인 요청 사항

- 신규 적용 CLI의 명시 locale 요청 옵션을 `--locale <locale>`로 두는 방향
- 기존 update에서 `.hyper-waterfall/version.json`의 top-level `locale`을 기본 저장 계약으로 두고, 기존 호환 읽기는 유지하는 방향
- update 중 locale 전환 요청은 자동 변경이 아니라 별도 승인 항목으로 보고하는 방향
- 이번 task에서 CLI는 dry-run 판단 연결까지만 다루고 실제 파일 적용 write mode는 제외하는 범위
- 위 잠정 단계와 예상 변경 파일 범위

승인되면 `task_m050_70_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
