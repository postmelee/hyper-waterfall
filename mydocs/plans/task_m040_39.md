# Task #39 수행계획서 - Claude plugin packaging 검증

GitHub Issue: [#39](https://github.com/postmelee/hyper-waterfall/issues/39)
마일스톤: M040

## 목적

Claude Code 사용자가 Hyper-Waterfall 절차를 plugin 경로로 발견하고 실행할 수 있는지 packaging 관점에서 검증한다. 이번 task는 Claude plugin 또는 Claude Code plugin의 최신 구조, `.claude/skills`와 canonical Skill의 관계, hook 배포 가능성과 한계를 확인해 #40의 배포 후보 생성과 설치 smoke 전에 판단 근거를 남긴다.

핵심 결과는 Claude plugin packaging 검증 기록과 guardrail 적용성 분류다. plugin은 `templates/mydocs/skills`, `templates/mydocs/manual`, `docs/agent-entrypoint.md`, npm CLI, release manifest의 별도 진실 원천이 되지 않는다는 #36 공통 원칙을 유지한다.

## 배경

M040은 Docker를 제외하고 npm, Homebrew, Codex, Claude 채널의 v0.2.0 최종 배포 가능 상태를 만드는 마일스톤이다. #36에서 Codex/Claude plugin 공통 배포 원칙과 packaging check matrix가 정리됐고, #39는 그중 Claude plugin packaging 검증 단계다.

현재 #37 Codex plugin packaging 검증은 별도 worktree에서 병렬 진행 중이다. 따라서 #39는 #36 산출물을 기준으로 Claude 쪽 검증을 먼저 진행하고, #37 산출물이 main 또는 접근 가능한 보고서로 가용해지면 마지막 정합성 검토에서 참고한다. #37의 미완료 산출물을 전제로 #39 범위를 확장하지 않는다.

## 범위

### 포함

- Claude Code plugin 또는 Claude plugin packaging 공식/현재 사양 확인
- `.claude-plugin/plugin.json`, plugin root component 배치, `skills/`, `hooks/hooks.json`, local plugin test 방식 검토
- `.claude/skills`, `templates/.claude/skills`, `CLAUDE.md`, `docs/agent-entrypoint.md`가 canonical Skill과 연결되는 방식 확인
- Claude plugin에서 core Skill을 release snapshot, thin wrapper, runtime reference 중 어떤 방식으로 다룰 수 있는지 검토
- `PreToolUse`, `PermissionRequest`, `PostToolUse`, `UserPromptSubmit`, `Stop`, `InstructionsLoaded`, `FileChanged` 등 이슈 본문에 명시된 hook/lifecycle 후보의 실제 지원 여부와 guardrail 적합성 분류
- hook이 작업지시자 승인을 자동 대체하지 않는다는 경계 확인
- canonical drift, hook disable 가능성, local/user/project/managed scope 충돌, false positive, 사후 side effect 한계 기록
- 필요한 경우 local 검증용 scratch package skeleton 또는 후보 구조를 검토하되, #40에서 만들 배포 후보와 구분
- #36 공통 원칙과 가능한 경우 #37 결과를 기준으로 Claude-specific 보완점 정리

### 제외

- Claude plugin public 배포 실행
- #40에서 다룰 배포 후보 bundle 생성과 설치 smoke 완료 처리
- Codex plugin packaging 구현 또는 검증
- Docker image 관련 작업
- core Skill 본문을 Claude 전용으로 fork하는 작업
- hook 구현 코드 작성과 hook 정책 강제 적용
- hook을 통한 작업지시자 승인 자동 대체
- npm publish 또는 Homebrew tap 배포

## 설계 방향

- Claude plugin은 canonical source가 아니라 agent-facing 발견/실행 계층이라는 #36 원칙을 따른다.
- 검증 결과는 기술 기록으로 남긴다. 후보 파일명은 `mydocs/tech/task_m040_39_claude_plugin_packaging.md`로 한다.
- 공식 문서 확인 결과에는 확인일, URL, 확인한 항목, Hyper-Waterfall 해석을 함께 기록한다.
- 이슈 본문에 있는 hook/lifecycle 이벤트는 실제 공식 사양에서 지원 여부를 다시 확인한다. 지원되지 않거나 명칭이 다른 이벤트는 그대로 구현 후보로 취급하지 않고 보류/불일치로 기록한다.
- packaging 후보는 drift 방지를 우선한다. core Skill을 plugin에 포함해야 한다면 release snapshot 또는 thin wrapper로 표시하고, 편집 원천은 계속 `templates/mydocs/skills`로 둔다.
- hook은 deterministic command hook과 객관 상태 검사 중심으로 분류한다. 승인 여부처럼 대화 맥락이 필요한 판단은 자동 block/allow 대상에서 제외한다.
- #37 산출물은 이미 main에 병합됐거나 보고서가 명확히 확인되는 경우에만 cross-check 입력으로 사용한다. 별도 worktree의 진행 중 변경은 #39에서 수정하거나 의존하지 않는다.
- core Skill 또는 manual 본문 변경 필요성이 발견되면 이번 task에서 임의 수정하지 않고 승인 요청 또는 후속 이슈 후보로 분리한다.

### 통합 검증

- Claude plugin 또는 Claude Code packaging 후보 구조가 확인된다.
- `.claude/skills`가 canonical Skill 본문과 drift 없이 동작하는 기준이 정리된다.
- `hooks/hooks.json`을 포함할 경우 필요한 구조, 지원 이벤트, matcher 범위, decision control이 확인된다.
- Hyper-Waterfall guardrail에 적합한 Claude hook 후보와 적용하지 말아야 할 자동 승인 범위가 구분된다.
- plugin 미설치 또는 hook 비활성 상태에서도 `CLAUDE.md`, `.claude/skills`, `docs/agent-entrypoint.md`, npm CLI 경로로 fallback할 수 있음이 설명된다.
- #40에서 public 또는 local 배포 후보를 만들기 전에 필요한 보류 조건과 전제조건이 남는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **Claude 사양 변동**: plugin/hook 공식 사양은 바뀔 수 있다. 확인일과 URL을 남기고, #40에서 배포 후보를 만들 때 재확인하도록 한다.
- **#37 병렬 진행 영향**: Codex packaging 결과가 아직 main에 없을 수 있다. #39는 #36 원칙을 기준으로 진행하고, #37 결과는 가용한 산출물만 참고한다.
- **Skill drift**: plugin bundle에 core Skill 본문을 복제하면 `templates/mydocs/skills`와 갈라질 수 있다. release snapshot 또는 thin wrapper 여부를 명확히 표시한다.
- **hook 과신**: hook이 승인 여부를 자동 판단하면 하이퍼-워터폴 핵심 규칙을 훼손한다. 객관 상태 검증과 위험 명령 guardrail로 제한한다.
- **scope 충돌**: Claude user/project/local/managed 설정과 plugin hook이 merge될 때 중복 또는 충돌이 생길 수 있다. 충돌 가능성을 문서화하고 #40 smoke 조건으로 넘긴다.
- **문서 변경 범위 확산**: 검증 중 core Skill, manual, README 변경 필요성이 발견될 수 있다. 이번 task의 기본 범위는 기술 기록과 필요한 cross-reference로 제한하고, 절차 본문 변경은 별도 승인으로 분리한다.
