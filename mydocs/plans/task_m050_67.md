# 수행계획서

GitHub Issue: [#67](https://github.com/postmelee/hyper-waterfall/issues/67)
마일스톤: M050

## 목적

Hyper-Waterfall 적용 대상 저장소에 설치되는 `templates/` 사용자-facing 파일을 다국어 locale pack으로 확장할 수 있도록 파일 배치, manifest 표현 방식, 신규 적용과 기존 업데이트 판단 기준을 확정한다.

이번 task는 #65의 다국어 적용 정책과 #66의 README 다국어 진입 문서를 이어받아, 후속 #68 영어 locale pack 작성과 #69 중국어 간체 locale pack 작성이 따를 구조적 기준을 만든다. 실제 locale pack 본문 전체 작성은 후속 task로 남기고, 이번 task에서는 구조와 판단 계약을 고정한다.

## 배경

#65에서 기본 locale은 `en`, 기존 한국어 보존 locale은 `ko`, 중국어 간체는 `zh-CN`으로 정했다. 또한 placeholder, branch, filename pattern, command, code block은 언어와 무관한 구조적 계약으로 보존하고, 누락 locale fallback은 조용히 성공 처리하지 않고 적용/update 보고에 드러내야 한다고 정의했다.

현재 신규 적용과 기존 업데이트는 `docs/agent-entrypoint.md`, `docs/lifecycle/adoption.md`, `docs/lifecycle/update.md`, `templates/manifest.json`을 기준으로 동작한다. 하지만 `templates/manifest.json`은 아직 locale별 source, fallback, locale 선택 보존, locale pack 적용 가능 여부를 표현하지 않는다. 따라서 후속 locale pack 작성 전에 manifest와 lifecycle 문서가 같은 기준을 갖도록 정렬해야 한다.

## 범위

### 포함

- `templates/manifest.json`의 locale 지원 표현 방식 설계와 최소 필드 반영
- locale pack 디렉터리 또는 파일 배치 구조 확정
- 신규 적용 시 locale 선택, 기본값, fallback, 누락 보고 기준 문서화
- 기존 업데이트 시 기존 locale 보존, locale 전환, locale별 diff 판단 기준 문서화
- 기존 단일 template 파일과 locale pack의 canonical 관계 정의
- placeholder 보존과 checksum/diff 판단 시 locale별 source를 어떻게 다룰지 정리

### 제외

- `en`, `ko`, `zh-CN` locale pack 본문 전체 작성
- `templates/` 전체 파일의 실제 번역
- CLI 구현 변경
- GitHub Release 생성
- migration guide 본문 작성
- 새 label 생성

## 설계 방향

- 기본 locale은 #65 정책대로 `en`으로 둔다.
- locale pack은 canonical 절차를 새로 만들지 않고, `templates/manifest.json`과 lifecycle 문서가 가리키는 설치 source 계층으로만 둔다.
- 기존 한국어 template 원문은 후속 #68에서 `ko` 기준 자료로 보존할 수 있게 위치와 관계를 명확히 한다.
- manifest는 기계 판독 가능한 최소 계약을 제공하되, 실제 적용/update workflow 연결은 #70에서 완성할 수 있도록 과도한 실행 로직을 넣지 않는다.
- 선택 locale 파일이 없을 때는 `en` fallback 후보를 제시하되, 누락과 혼합 locale 상태를 보고서/판단 결과에 명시하는 방향을 유지한다.
- `{REPO_SLUG}`, `{BASE_BRANCH}`, `{PR_TEMPLATE_PATH}` 등 placeholder는 locale 파일에서도 번역하거나 실제 값으로 치환하지 않는다.

## 잠정 단계

- **Stage 1 — 현 구조 inventory와 locale 대상 분류**
  - 현재 `templates/manifest.json`, lifecycle 문서, `templates/` 파일 목록을 조사하고 locale 대상/비대상/후속 보류 대상을 정리한다.
  - 검증 관점: #65 정책과 충돌하는 한국어 고정 문구, locale pack 후보, manifest 표현 공백이 식별되어야 한다.

- **Stage 2 — locale pack 구조와 manifest 계약 반영**
  - locale pack 디렉터리 또는 source 표현 구조를 확정하고 `templates/manifest.json`에 최소 locale 계약을 반영한다.
  - 검증 관점: JSON이 유효하고, 후속 #68/#69가 어떤 파일 위치와 필드를 따라야 하는지 명확해야 한다.

- **Stage 3 — adoption/update 문서 기준 정렬**
  - 신규 적용과 기존 업데이트 문서에 locale 선택, 기본값, 기존 locale 보존, fallback, 누락 보고 기준을 정리한다.
  - 검증 관점: 파일 변경 전 판단 결과에 locale 관련 항목이 포함되어야 하며, #70 workflow 구현 범위를 침범하지 않아야 한다.

- **Stage 4 — 통합 정합성 검증과 보고**
  - README/진입점/정책/manifest/lifecycle 문서의 용어와 링크를 검증하고 최종 보고서를 작성한다.
  - 검증 관점: manifest JSON, locale 키워드 검색, diff check, 후속 이슈 경계가 모두 통과해야 한다.

- `templates/manifest.json`이 JSON으로 파싱된다.

- placeholder와 구조적 계약 보존 규칙이 #65 정책과 충돌하지 않는다.
- 후속 #68/#69가 따를 locale pack 파일 배치와 manifest 표현 기준이 명확하다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **manifest schema 과설계**: #70 workflow 구현 전에 지나치게 세부 실행 로직을 manifest에 넣으면 후속 작업이 경직된다. 이번 task는 기계 판독 가능한 최소 구조와 판단 기준으로 제한한다.
- **canonical 원천 분리**: locale별 문서가 별도 절차 원천처럼 보이면 유지보수 기준이 흔들린다. `en`과 구조적 계약을 기준으로 절차 의미를 보존한다고 명시한다.
- **기존 한국어 원문 손실**: #68 이전에 한국어 template을 이동하거나 덮어쓰면 dogfooding 원문이 손상될 수 있다. 이번 task는 구조만 만들고 전체 본문 이동/번역은 제외한다.
- **fallback이 조용한 품질 저하로 오해될 가능성**: 누락 locale fallback은 성공이 아니라 보고 대상임을 adoption/update 문서에 반복해서 명시한다.
- **후속 이슈 범위 침범**: locale pack 본문 작성은 #68/#69, workflow 연결은 #70, migration guide는 #71로 남긴다.

## 승인 요청 사항

- #67을 locale pack 본문 작성이 아니라 구조, manifest 표현, adoption/update 판단 기준 확정 task로 진행하는 방향
- 예상 변경 파일의 공식 문서 위치 판단
- manifest에는 최소 locale 계약만 반영하고, 실제 workflow 실행 구현은 #70으로 남기는 방향
- `en`, `ko`, `zh-CN` 본문 전체 작성과 migration guide 작성을 제외하는 범위

승인되면 `task_m050_67_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
