# task_m020_26.md - v0.2.0 manifest release 상태와 checksum 확정 수행계획서

GitHub Issue: [#26](https://github.com/postmelee/hyper-waterfall/issues/26)
마일스톤: M020

## 목적

`v0.2.0` GitHub Release/tag를 만들기 직전에 `templates/manifest.json`의 release 상태와 checksum 정책을 확정한다. Release가 Hyper-Waterfall의 canonical 배포 기준이 되므로, release manifest가 `planned` / `pending-release` 상태로 고정되는 문제를 피하거나, 보류가 필요하면 그 이유와 후속 확정 조건을 명확히 남긴다.

이번 task는 실제 tag 생성이나 GitHub Release publish가 아니라, 그 직전의 manifest/status/checksum 기준을 확정하는 작업이다.

## 배경

PR #25로 `docs/releases/v0.2.0.md`가 `main`에 merge되어 release 전 체크리스트와 release notes 초안은 준비됐다. 현재 `templates/manifest.json`은 `frameworkVersion: 0.2.0`, `plannedTag: v0.2.0`, `baselineTag: v0.1.0`, `release.status: planned`, checksum `pending-release` 상태다.

GitHub Release가 적용 저장소 업데이트의 기준점이 되려면 release manifest가 안정적인 기준을 제공해야 한다. 따라서 release 생성 직전에 `release.status`를 바꿀지, checksum 값을 산출해 채울지, 또는 일부 checksum을 보류할 수밖에 없는지 검토하고 문서화해야 한다. 메인 worktree에는 기존 미추적 파일 `DESIGN.md`, `README_draft.md`, `opus_report.md`가 있으므로 이번 task는 `/private/tmp/hyper-waterfall-task26` 분리 worktree에서 진행한다.

## 설계 방향

- 먼저 checksum을 어떤 단위로 계산할 수 있는지 확인한다. 파일 checksum은 source 경로별 SHA-256으로 계산 가능한지, directory checksum은 재현 가능한 archive/hash 방식이 필요한지 분리한다.
- release manifest가 실제 release 기준이 되도록 가능한 경우 `release.status`를 release 직전 상태로 조정한다. 단, 상태값 이름은 기존 문서와 CLI가 오해하지 않는 방향으로 정한다.
- checksum을 확정하면 산출 명령과 대상 범위를 `docs/releases/v0.2.0.md`에 남긴다.
- checksum을 보류하는 항목이 있으면 보류 이유와 release 이후 또는 tooling 보강 이후 확정 조건을 명시한다.
- tag/release 생성 명령은 이번 task의 산출물이 아니라 후속 승인 단계로 남긴다.

## 잠정 단계

- **Stage 1 — manifest checksum 산출 가능성 조사**
  - `templates/manifest.json`의 checksum 대상과 파일/디렉터리 항목을 분류하고, 재현 가능한 checksum 산출 방법을 결정한다.
  - 검증 관점: checksum 산출 방식이 JSON parse와 파일 경로 기준으로 설명 가능한지 확인한다.
- **Stage 2 — release 상태와 checksum 정책 반영**
  - 결정된 정책에 따라 `templates/manifest.json`과 `docs/releases/v0.2.0.md`를 수정한다.
  - 검증 관점: `planned`, `released`, `pending-release`, `checksum` 표현이 release 생성 직전 기준과 충돌하지 않는지 확인한다.
- **Stage 3 — 문서 정합성과 release 전 검증 확정**
  - README, migration guide, distribution strategy와 release 준비 문서의 기준 표현을 대조하고, 실제 tag/release 생성 전 검증 결과를 정리한다.
  - 검증 관점: tag/release 미생성 상태를 확인하고, 실제 생성은 승인 게이트로 남겼는지 확인한다.

## 리스크

- **directory checksum 재현성**: directory 항목은 파일 순서, 권한, symlink 처리에 따라 checksum이 달라질 수 있다. 산출 방식을 확정하지 못하면 무리하게 값을 채우지 않고 보류 조건을 문서화한다.
- **release.status 의미 혼선**: tag 생성 전 `released`로 바꾸면 실제 release와 문서 상태가 어긋날 수 있다. 상태값 변경 시점과 의미를 문서에 명확히 적는다.
- **canonical 기준 불안정**: release manifest가 `pending-release`를 너무 많이 남기면 적용 저장소 업데이트 기준으로 약해질 수 있다. 보류는 최소화하고 이유를 명확히 남긴다.
- **승인 없는 배포 실행**: 이번 task에서 tag/release/npm publish는 실행하지 않는다.
