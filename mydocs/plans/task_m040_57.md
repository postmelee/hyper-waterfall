# Task #57 수행계획서 - 신규 적용 시 manifest strict 범위 제한 명시

GitHub Issue: [#57](https://github.com/postmelee/hyper-waterfall/issues/57)
마일스톤: M040

## 목적

Hyper-Waterfall 신규 적용 단계에서 manifest가 정의한 운영 파일과 작업 기억 구조만 설치하도록 범위를 명확히 제한한다. 적용 대상 저장소의 제품 코드, 제품 문서, 아키텍처, 로드맵, API 계약, 예제, 스키마 같은 프로젝트 고유 산출물은 신규 적용 중 생성하지 않고 별도 task 후보로만 분리하게 만든다.

이번 task의 핵심은 `/docs`의 의미를 대상 저장소에 규정하는 것이 아니다. `docs/**`는 적용 중 생성/수정 금지 예시로만 다루고, 대상 저장소의 일반 문서 구조는 Hyper-Waterfall이 선점하지 않는다.

## 배경

rhwp-mcp 첫 적용 사례에서 "저장소 생성"과 "Hyper-Waterfall 적용"이 한 번에 수행되면서 `docs/architecture.md`, `docs/mcp-tools.md`, `docs/app-integration.md`, `docs/roadmap.md` 같은 제품 고유 문서가 같이 생성됐다. 이 문서들은 Hyper-Waterfall 운영 매뉴얼이 아니므로 `mydocs/manual`로 이동하게 강제하는 것도 맞지 않다.

현재 `docs/agent-entrypoint.md`는 신규 적용 절차와 manifest 기반 후보 보고를 설명하지만, manifest 외 산출물을 만들면 안 된다는 strict 제한이 충분히 기계적으로 표현돼 있지 않다. `templates/manifest.json`도 적용 대상 파일 목록은 제공하지만, 에이전트가 allowlist 밖을 생성하지 말아야 한다는 적용 모드와 금지 예시를 별도 필드로 드러내지 않는다.

참고 산출물:

- `docs/agent-entrypoint.md`
- `templates/manifest.json`
- `templates/mydocs/manual/document_structure_guide.md`
- `README.md`
- rhwp-mcp 초기 적용 결과

## 범위

### 포함

- `docs/agent-entrypoint.md`에 신규 적용 strict 범위 제한과 manifest 외 생성 금지 규칙 추가
- `templates/manifest.json`에 신규 적용 모드, strict manifest 여부, 금지 대상 예시를 표현하는 구조 추가
- `templates/mydocs/manual/document_structure_guide.md`의 lifecycle/신규 적용 설명이 strict 범위와 충돌하지 않게 보정
- 필요 시 `README.md`의 적용 절차 설명 정합성 확인과 최소 수정
- `/docs`의 의미를 정의하지 않고 금지 예시로만 등장하는지 확인

### 제외

- 대상 저장소의 일반 문서 구조 정의
- `/docs`를 제품/사용자/배포 문서 위치로 규정하는 문구 추가
- rhwp-mcp 저장소 수정 또는 기존 rhwp-mcp 문서 이동
- 신규 CLI 구현
- 신규 적용 자동화 도구 구현
- 제품 코드, 예제, 스키마 생성
- 제품 고유 문서를 `mydocs/manual`에 생성하도록 강제하는 규칙 추가

## 설계 방향

- 신규 적용 절차는 "무엇을 설치하는가"와 "무엇을 만들지 않는가"를 함께 표현한다. 설치 대상은 manifest allowlist와 version state, symlink로 제한한다.
- `docs/**`, `src/**`, `examples/**`, `schemas/**`, `package.json`, `tsconfig.json` 등은 경로 의미를 정의하지 않고 신규 적용 중 생성/수정 금지 예시로만 둔다.
- 프로젝트 고유 산출물이 필요해 보여도 신규 적용 중 파일을 만들지 않는다. 적용 판단 결과의 보류/별도 task 후보로 기록하고, 실제 작성은 적용 완료 후 별도 GitHub Issue와 Hyper-Waterfall task 흐름으로 넘긴다.
- `mydocs/manual`은 반복 적용되는 운영 절차와 기준만 담는다. 제품 아키텍처, 로드맵, API 계약을 `mydocs/manual`로 유도하지 않는다.
- manifest 필드는 사람이 읽는 설명과 기계적 판단을 둘 다 지원하되, 기존 `files[]` 구조와 update policy 의미를 깨지 않는다.

- Stage 1

- Stage 3
  - `rg -n "docs/.*제품|제품/사용자/배포 문서|mydocs/manual.*아키텍처|mydocs/manual.*로드맵" docs/agent-entrypoint.md templates/mydocs/manual/document_structure_guide.md README.md`

  - `git diff --check`

### 통합 검증

- 신규 적용 절차가 manifest 외 파일/디렉터리 생성 금지를 명확히 표현한다.
- `/docs`는 의미가 정의되지 않고, 적용 중 생성/수정 금지 예시로만 등장한다.
- 제품 고유 산출물은 Hyper-Waterfall 적용 이후 별도 task로 다뤄야 한다는 원칙이 경로 특정 없이 표현된다.
- manifest JSON이 파싱 가능하다.
- README, 매뉴얼, entrypoint, manifest 간 설명이 충돌하지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **대상 저장소 구조 선점**: `/docs`를 설명하려다 대상 저장소의 일반 문서 구조를 규정할 수 있다. 금지 예시와 의미 정의를 분리한다.
- **manual 오염**: 제품 아키텍처나 API 계약을 `mydocs/manual`로 유도하면 운영 매뉴얼 경계가 흐려진다. 제품 고유 산출물은 별도 task 후보로만 남긴다.
- **manifest 호환성**: 새 strict 필드가 기존 lifecycle 판단 도구와 충돌할 수 있다. 기존 `files[]`와 update policy 구조를 유지하고 additive 필드로만 추가한다.
- **README 과수정**: README의 적용 절차 설명을 크게 다시 쓰면 M040 최종 정합성 감사 범위와 겹칠 수 있다. 필요한 경우 짧은 보정만 한다.
- **검증 공백**: 현재 CLI가 strict 필드를 강제하지 않으면 문서 규칙에 머물 수 있다. 이번 task에서는 구현을 제외하고, 필요하면 후속 task 후보로 기록한다.
