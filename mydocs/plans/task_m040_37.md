# Task #37 수행계획서 - Codex plugin packaging 검증

GitHub Issue: [#37](https://github.com/postmelee/hyper-waterfall/issues/37)
마일스톤: M040

## 목적

Codex plugin을 public 배포 후보로 만들기 전에 현재 Codex plugin packaging 사양, bundle metadata, Skill 노출 방식, canonical 문서 참조 방식, fallback 경로, hook guardrail 적용 가능성을 검증한다.

이번 task의 핵심 결과는 Codex plugin bundle을 실제로 만들거나 배포하는 것이 아니라, #38의 "Codex plugin 배포 후보 생성과 설치 smoke"로 넘어갈 수 있는지 판단할 수 있는 검증 기록과 go/no-go 기준이다. 특히 plugin이 `templates/mydocs/skills`, `templates/mydocs/manual`, `templates/manifest.json`, migration guide, `docs/agent-entrypoint.md`, npm CLI를 새 진실 원천으로 복제하지 않는 구조를 확인한다.

## 배경

M040은 Docker를 제외하고 npm, Homebrew, Codex, Claude 채널의 v0.2.0 최종 배포 가능 상태를 만드는 마일스톤이다. #36에서 Codex/Claude plugin 공통 배포 원칙과 `docs/plugin-distribution-principles.md`가 작성됐고, #37은 그 원칙을 Codex plugin packaging 사양에 대입해 검증하는 첫 도구별 작업이다.

#36 산출물은 Codex plugin을 Hyper-Waterfall의 canonical source가 아니라 발견성과 실행 편의를 높이는 얇은 계층으로 제한했다. 동시에 Codex plugin hook은 기본 안전장치가 아니라 opt-in guardrail이며, 작업지시자 승인 여부를 자동 판정하지 않는다는 원칙을 세웠다.

이번 task에서는 OpenAI Codex plugin/hooks 공식 문서를 최신 기준으로 재확인하고, #36의 공통 check matrix 중 Codex packaging 행을 실제 파일 구조와 검증 항목으로 좁힌다. 공식 사양이 바뀌었거나 로컬 환경에서 install/load smoke가 불가능하면 그 한계를 명시하고 #38 진입 조건으로 넘긴다.

## 범위

### 포함

- 최신 Codex plugin packaging 공식 사양 확인과 확인일 기록
- `.codex-plugin/plugin.json` 후보 구조, metadata, component path 규칙, bundle root 제약 검토
- Codex plugin에서 Hyper-Waterfall Skill과 manual을 발견하게 하는 방식 검증
- core Skill 포함 방식 후보 비교: release snapshot, thin wrapper, 링크 안내, npm CLI/agent entrypoint 참조
- `templates/manifest.json`, migration guide, `docs/agent-entrypoint.md`, npm CLI를 복제하지 않는 canonical 참조 방식 검증
- fallback 경로 검증: plugin이 없거나 canonical 파일을 읽지 못할 때 AGENTS/manual/CLI로 복구하는 안내
- Codex plugin hook packaging 가능성 확인: `hooks/hooks.json`, manifest `hooks` field, feature flag, 경로 규칙
- `PreToolUse`, `PermissionRequest`, `PostToolUse`, `UserPromptSubmit`, `Stop`, `SessionStart` 중 Hyper-Waterfall guardrail에 적합한 이벤트 분류
- hook 적용 한계 기록: 일부 shell/tool path 미포착, 비대상 도구, 사후 side effect 되돌림 불가, 승인 자동화 불가
- 필요 시 로컬 검증용 skeleton의 파일 트리를 문서화하거나 임시 skeleton으로 static validation
- Codex 환경에서 install/load smoke가 가능한지 조사하고, 가능 여부와 #38 인계 조건 기록
- public 배포 전 go/no-go와 필요한 후속 작업 기록

### 제외

- Codex plugin public 배포 실행
- #38 범위인 배포 후보 bundle 생성과 설치 smoke 완료
- Claude plugin packaging 구현 또는 검증
- Docker image 관련 작업
- core Skill이나 manual 본문을 Codex 전용으로 fork하는 작업
- hook 구현 코드 작성
- hook을 통한 작업지시자 승인 자동 대체
- npm publish, Homebrew tap 배포, GitHub Release/tag 생성

## 설계 방향

- `docs/plugin-distribution-principles.md`를 상위 원칙으로 두고, #37은 Codex plugin에 특화된 packaging 검증 결과를 `mydocs/tech/` 기술 노트와 단계 보고서에 남긴다.
- Codex plugin 후보 구조는 `.codex-plugin/plugin.json`과 plugin root 내부 component path를 기준으로 검토한다. 실제 배포 후보 파일을 repository root에 고정하는 일은 #38 승인 전에는 하지 않는다.
- core Skill은 별도 편집 원천으로 만들지 않는다. plugin이 Skill을 노출해야 한다면 release snapshot 또는 thin wrapper 중 어느 방식이 Codex 사양과 canonical 원칙에 맞는지 비교한다.
- manual, manifest, migration guide는 plugin에 복제하지 않고 release URL, repository path, `docs/agent-entrypoint.md`, npm CLI dry-run 경로로 참조하는 것을 우선한다.
- hook은 필수 안전장치가 아니라 opt-in 보조 guardrail로 문서화한다. feature flag가 필요하거나 event coverage가 제한되면 public 배포 전 보류 조건으로 기록한다.
- hook event 분류는 block/warn/report-only 후보까지만 한다. 승인 여부 자동 판정, 자동 allow, 자동 파일 수정, 자동 revert는 설계 대상에서 제외한다.
- install/load smoke가 현재 Codex 환경이나 plugin tooling 제약 때문에 불가능하면 우회 구현을 만들지 않고 #38의 명시 인계 조건으로 남긴다.
- 공식 문서 확인 결과는 확인일을 남기고, 사양이 불명확한 부분은 "확정", "추론", "재확인 필요"로 분리한다.

- Codex plugin bundle에 필요한 metadata와 component path 규칙이 확인된다.

- public 배포 전 go/no-go, #38 인계 조건, 보류 조건이 기록된다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **Codex 사양 변동**: plugin과 hook 사양은 바뀔 수 있다. 공식 문서 확인일과 URL을 기록하고, #38에서 배포 후보 생성 전 다시 확인한다.
- **canonical drift**: plugin bundle이 Skill, manual, manifest, migration guide를 별도 본문으로 복제하면 release 기준이 갈라진다. 포함/참조/fallback 분류로 복제와 참조를 분리한다.
- **hook 과신**: hook은 작업지시자 승인 여부를 알 수 없다. 승인 자동화는 금지하고, 객관 상태 확인과 위험 명령 guardrail 후보로만 제한한다.
- **local smoke 제약**: 현재 Codex 환경에서 plugin install/load smoke가 불가능할 수 있다. 이 경우 우회 설치를 만들지 않고 #38의 환경 조건으로 남긴다.
- **범위 확산**: packaging 검증 중 실제 plugin bundle 생성이나 hook 구현 요구가 생길 수 있다. 이번 task에서는 검증과 판단 기록으로 제한하고, 실제 배포 후보는 #38로 넘긴다.
