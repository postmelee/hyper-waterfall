# Task M040 #60 수행 계획서

GitHub Issue: [#60](https://github.com/postmelee/hyper-waterfall/issues/60)
마일스톤: M040

## 목적

Hyper-Waterfall 적용 저장소에서 제품/사용자/API/아키텍처/로드맵 문서를 생성할 때, `docs/` 같은 특정 폴더명을 프레임워크가 강제하지 않고 대상 프로젝트가 명시적으로 공식 문서 루트를 선택하도록 정책을 강화한다.

이번 task는 `mydocs/`의 작업 기억 체계와 대상 프로젝트의 공식 문서 표면을 분리하고, 문서 생성 task가 수행계획서 단계에서 문서 위치 판단을 남기도록 템플릿과 매뉴얼을 보강하는 것이 목적이다.

## 배경

`rhwp-mcp` 첫 제품 설계 task에서 `docs/*.md`가 생성되면서, 해당 문서가 `mydocs/manual`이나 `mydocs/tech`가 아니라 `docs/`에 있어야 하는지 검토했다. 결론은 적용 직후 manifest 외 산출물을 만드는 것은 금지하되, 별도 Issue와 승인 후 공식 프로젝트 기준 문서를 만들 수 있다는 것이었다.

다만 `alhangeul-macos`의 `docs/`는 GitHub Pages와 Sparkle appcast 배포 표면으로 쓰이며, 일반적인 제품 기준 문서 폴더 선례로 일반화하기 어렵다. 반대로 `edwardkim/rhwp`는 역사적 원형 사례로 `mydocs` 안에 제품 문서와 작업 기억이 섞인 면이 있다. 따라서 프레임워크 템플릿은 특정 이름을 강제하기보다 문서 위치 판단을 강제해야 한다.

현재 `templates/mydocs/manual/document_structure_guide.md`에는 `manual/`을 "사용자/개발자 문서"로 설명하는 표현이 남아 있어 대상 프로젝트 제품 문서와 Hyper-Waterfall 운영 매뉴얼을 혼동시킬 수 있다. 계획서와 최종 보고서 템플릿에도 문서 위치 판단을 명시적으로 확인하는 항목이 없다.

## 범위

### 포함

- `templates/mydocs/manual/document_structure_guide.md`에 공식 문서 루트와 `mydocs/` 경계 정책 추가
- `mydocs/manual` 설명을 Hyper-Waterfall 운영 절차와 반복 규칙 중심으로 좁힘
- `templates/mydocs/_templates/task_plan.md`에 문서 위치 판단 섹션 추가
- `templates/mydocs/_templates/task_impl_plan.md`에 문서 위치 판단 확인 항목 추가
- `templates/mydocs/_templates/final_report.md`에 문서 위치 검증 항목 추가
- `docs/agent-entrypoint.md`의 신규 적용 중 제품 문서 보류 규칙 보강
- `templates/AGENTS.md`, `templates/CLAUDE.md`에 문서 위치 판단 승인 규칙 추가
- `README.md`의 문서 구조 설명을 새 정책과 맞게 보강

### 제외

- `templates/manifest.json`에 `docs/`, `specs/`, `site/`, `website/`, `adr/` 같은 공식 문서 루트를 추가하지 않는다.
- 신규 적용 시 공식 문서 루트를 자동 생성하지 않는다.
- Hyper-Waterfall이 대상 프로젝트의 공식 문서 루트 이름을 기본값으로 강제하지 않는다.
- `rhwp-mcp`, `alhangeul-macos`, `edwardkim/rhwp` 저장소 파일은 수정하지 않는다.
- 이번 task에서 실제 대상 프로젝트용 제품 문서 템플릿을 새로 만들지 않는다.

## 설계 방향

- Hyper-Waterfall은 `mydocs/` 작업 기억 체계와 운영 매뉴얼을 제공하지만, 대상 프로젝트의 공식 문서 루트 이름은 고정하지 않는다.
- 대상 프로젝트의 공식 문서 루트는 `docs/`, `specs/`, `site/`, `website/`, `adr/`, `book/`, `wiki/` 등 저장소 성격에 맞게 별도 Issue에서 명시적으로 선택한다.
- 제품/사용자/기여자/외부 통합/API/아키텍처/로드맵 문서를 생성하거나 이동하는 task는 수행계획서에 위치 판단을 기록하고 승인받는다.
- `mydocs/manual`은 Hyper-Waterfall 운영 절차, 에이전트 규칙, 반복 적용되는 작업 기준을 담는다. 대상 프로젝트 제품 문서를 이 위치로 강제하지 않는다.
- `mydocs/tech`는 기술 조사, 구조/스펙 분석, 대안 비교, 설계 판단 근거, 아직 공식화되지 않은 초안을 담는다.
- 신규 적용 guardrail은 유지한다. 적용 단계에서는 manifest 외 제품 코드와 제품 문서를 만들지 않고, 필요성이 보이면 별도 task 후보로만 기록한다.

## 잠정 단계

- **Stage 1 — 문서 구조 정책 정리**
  - `document_structure_guide.md`에 공식 문서 루트와 `mydocs/` 경계 정책 추가
  - `manual/` 설명을 Hyper-Waterfall 운영 매뉴얼 중심으로 수정
  - 신규 적용 중 공식 문서 자동 생성을 금지하는 기존 정책 유지 확인
- **Stage 2 — 템플릿과 에이전트 진입 규칙 보강**
  - `task_plan.md`, `task_impl_plan.md`, `final_report.md`에 문서 위치 판단/검증 항목 추가
  - `docs/agent-entrypoint.md`, `templates/AGENTS.md`, `templates/CLAUDE.md`에 관련 guardrail 추가
- **Stage 3 — README 정합성 및 통합 검증**
  - README의 문서 구조 설명과 철학 설명을 새 정책과 맞게 보강
  - `rg` 기반 정합성 확인, `git diff --check`, manifest 비변경 확인
  - 최종 보고서 작성과 PR 준비

## 검증 계획

### 단계별 검증

- Stage 1
  - `rg -n "사용자/개발자 문서|공식 문서|문서 위치|mydocs/manual|mydocs/tech" templates/mydocs/manual/document_structure_guide.md`
  - `rg -n "docs/\\*\\*|manifest 외|제품 문서" templates/mydocs/manual/document_structure_guide.md`
- Stage 2
  - `rg -n "문서 위치|공식 문서|docs/|specs/|site/|website/|adr/" templates/mydocs/_templates docs/agent-entrypoint.md templates/AGENTS.md templates/CLAUDE.md`
  - `rg -n "docs/|specs/|site/|website/|adr/" templates/manifest.json`로 자동 생성 항목이 추가되지 않았는지 확인
- Stage 3
  - `rg -n "문서 위치|공식 문서|mydocs/|docs/" README.md`
  - `git diff --check`
  - `git status --short`

### 통합 검증

- 공식 문서 루트 이름이 `docs/`로 고정되지 않는다.
- `templates/manifest.json`에 공식 문서 루트 자동 생성 항목이 추가되지 않는다.
- 제품/사용자/API/아키텍처/로드맵 문서 생성 task가 계획서에서 위치 판단을 승인받도록 템플릿에 반영된다.
- `mydocs/manual`, `mydocs/tech`, 공식 문서 루트의 경계가 README와 매뉴얼에서 모순 없이 설명된다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **과도한 일반화**: `docs/`를 금지하거나 반대로 기본값으로 만들면 또 다른 오적용이 생긴다. 폴더명 대신 위치 판단 기준을 강제한다.
- **템플릿 장황화**: 계획서와 보고서 템플릿이 지나치게 무거워질 수 있다. 문서 변경이 있는 task에서만 채우도록 짧은 표 형식으로 둔다.
- **README 중복**: README와 매뉴얼에 같은 정책을 길게 반복하면 유지보수 비용이 커진다. README는 요약, 상세 기준은 `document_structure_guide.md`에 둔다.
- **기존 적용 저장소와의 차이**: `rhwp`, `alhangeul-macos`의 역사적 배치를 새 템플릿 기준으로 강제 변경하지 않는다. 새 정책은 이후 task와 신규 적용의 판단 기준으로만 둔다.

## 승인 요청 사항

- 공식 문서 루트 이름을 Hyper-Waterfall이 고정하지 않고, 대상 프로젝트가 별도 task에서 선택하도록 하는 방향
- 문서 생성/수정 task에서 수행계획서에 문서 위치 판단을 기록하고 승인받도록 템플릿을 보강하는 방향
- `mydocs/manual`을 Hyper-Waterfall 운영 절차와 반복 규칙으로 좁히고, 대상 프로젝트 제품 문서 위치로 쓰지 않는 방향
- `templates/manifest.json`에는 `docs/` 또는 다른 공식 문서 루트를 추가하지 않는 제외 범위

승인되면 `task_m040_60_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
