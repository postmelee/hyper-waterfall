# 다국어 배포 진입점 보강과 v0.3.0 release readiness 수행계획서

GitHub Issue: [#79](https://github.com/postmelee/hyper-waterfall/issues/79)
마일스톤: M050

## 목적

M050 다국어 산출물을 배포 직전 사용자 경험까지 연결한다. README 3종에서 지원 언어, 언어별 적용 프롬프트, CLI dry-run 경로를 바로 확인할 수 있게 하고, English/Chinese 사용자가 진입 문서와 lifecycle 문서를 따라갈 때 한국어-only 문서로 갑자기 떨어지지 않도록 정리한다.

또한 v0.3.0 배포 전 `npx hyper-waterfall@0.3.0 ... --locale ... --dry-run` 안내가 package, manifest, release/migration 문서 상태와 모순되지 않게 맞춘다. 실제 npm publish, GitHub Release/tag 생성, Homebrew tap 갱신은 이번 task의 범위에서 제외하고 별도 release 실행 단계로 남긴다.

## 배경

#65-#71에서 다국어 정책, README 다국어화, locale pack 구조, `en`/`ko`/`zh-CN` 본문, locale 선택 workflow, smoke/migration guide를 완료했다. 기능 관점에서는 M050 목표가 거의 충족됐지만, 배포 전 README를 검토한 결과 다음 release-facing 보강이 필요하다는 결론이 나왔다.

- README 3종은 존재하지만 Quick Start가 한국어 중심의 `docs/agent-entrypoint.md`로 이어진다.
- English/Chinese README에 한국어 anchor 링크가 일부 남아 있다.
- README 상단에서 지원 locale, 기본 locale, fallback, 언어별 적용 프롬프트, `--locale` dry-run 명령이 즉시 보이지 않는다.
- npm registry의 현재 latest는 `0.2.0`이며, v0.3.0 다국어 배포 전 package/manifest/release 문서 정합성 확인이 필요하다.
- `npm pack --dry-run` 기준 locale pack은 tarball에 포함되므로, version과 문서 정합성만 맞추면 `npx hyper-waterfall@0.3.0` 배포 경로는 기술적으로 가능하다.

## 범위

### 포함

- `README.md`, `README.ko.md`, `README.zh-CN.md` 상단 Quick Start 근처 언어 지원 안내 보강
- 언어별 적용 프롬프트 예시 추가
- `npx hyper-waterfall@0.3.0 init --repo . --locale <locale> --dry-run` 형태의 CLI 판단 명령 안내 추가
- `docs/agent-entrypoint.en.md`, `docs/agent-entrypoint.zh-CN.md` 추가
- 필요한 범위의 lifecycle localized 문서 추가
  - `docs/lifecycle/adoption.en.md`, `docs/lifecycle/update.en.md`, `docs/lifecycle/update_pr.en.md`
  - `docs/lifecycle/adoption.zh-CN.md`, `docs/lifecycle/update.zh-CN.md`, `docs/lifecycle/update_pr.zh-CN.md`
- English/Chinese README의 한국어 anchor 링크를 localized template/manual 문서 또는 언어 중립 링크로 교체
- v0.3.0 배포 readiness 관점에서 `package.json`, `templates/manifest.json`, release/migration 문서의 version/status 문구 정합성 검토 및 필요한 보정
- `npm pack --dry-run`, `npm test`, locale 관련 grep/smoke 검증

### 제외

- 실제 `npm publish`
- GitHub Release/tag 생성
- Homebrew tap formula 갱신 또는 public tap 배포
- CLI를 승인 없는 자동 파일 설치기로 변경
- M050 외 신규 locale 추가
- locale pack 전체 재번역 또는 의미 변경

## 설계 방향

- README는 사용자가 처음 보는 release-facing 문서로 보고, 상단에 짧은 언어 지원 블록을 둔다. 상세 다국어 정책은 `docs/localization.md`와 lifecycle 문서로 연결한다.
- 언어별 적용 프롬프트는 복사 가능한 한 줄로 제공한다. 기본 프롬프트는 AI coding tool에 적용 판단을 요청하는 경로로 두고, CLI 예시는 dry-run 판단 명령임을 명확히 쓴다.
- `docs/agent-entrypoint.md`는 기존 한국어 진입점으로 유지하고, English/Chinese README는 각각 localized entrypoint로 연결한다. 한국어 README는 기존 문서를 계속 사용할 수 있다.
- lifecycle localized 문서는 한국어 원문과 같은 절차 계약을 유지하는 번역/정렬 문서로 작성한다. 새 정책을 추가하기보다 기존 `adoption`, `update`, `update_pr`의 의미를 언어별로 접근 가능하게 만드는 데 집중한다.
- `templates/locales/{locale}/mydocs/manual`과 `templates/locales/{locale}/mydocs/skills`는 이미 존재하는 localized 운영 문서이므로, README의 깊은 manual 링크는 가능한 한 해당 locale pack 문서로 향하게 한다.
- v0.3.0 readiness는 실제 배포 실행이 아니라 배포 전 파일 상태 정렬이다. package/manifest/release 문서가 v0.3.0을 가리키더라도 publish와 tag 생성은 별도 승인 단계로 남긴다는 점을 문서에 명시한다.

## 잠정 단계

- **Stage 1 — 다국어 진입점/링크 baseline 감사**
  - README 3종, `docs/agent-entrypoint.md`, lifecycle 문서, locale pack manual 링크, package/manifest version 상태를 점검한다.
  - English/Chinese README에 남은 한국어 anchor와 한국어-only 진입 경로를 목록화한다.
  - 검증 관점: 수정해야 할 링크, 진입 문서, release readiness 항목이 stage 보고서에 정리되어 있다.
- **Stage 2 — README 언어 지원과 적용 프롬프트 보강**
  - README 3종 상단에 지원 locale, 기본 locale, fallback, 언어별 AI 적용 프롬프트, CLI dry-run 명령을 짧게 추가한다.
  - README의 `npx` 안내가 dry-run 판단 명령이며 실제 적용은 승인 workflow를 거친다는 점을 명확히 한다.
  - 검증 관점: 세 README에서 언어 지원과 `--locale` 명령을 바로 검색할 수 있다.
- **Stage 3 — localized entrypoint/lifecycle 문서 추가**
  - English/Chinese entrypoint와 lifecycle 문서를 추가하고 링크를 정렬한다.
  - 한국어 원문의 절차 계약, placeholder, approval gate, manifest/update 기준을 유지한다.
  - 검증 관점: localized 문서의 링크 대상이 존재하고, 구조적 계약이 번역되지 않았다.
- **Stage 4 — v0.3.0 release readiness 정렬**
  - `package.json`, `templates/manifest.json`, release/migration 문서의 version/status 문구를 v0.3.0 배포 직전 상태와 맞춘다.
  - `npm pack --dry-run` 기준 README, docs, templates/locales가 tarball에 포함되는지 확인한다.
  - 검증 관점: 실제 publish/tag/Homebrew 갱신은 제외하면서도 `npx hyper-waterfall@0.3.0` 안내와 package 상태가 모순되지 않는다.
- **Stage 5 — 통합 검증과 최종 보고**
  - 테스트, pack dry-run, grep/link 검증, diff 검증을 실행한다.
  - 남은 release 실행 절차와 별도 승인 필요 항목을 최종 보고서에 정리한다.
  - 검증 관점: PR 전 작업 tree가 정리되고 #79 수용 기준이 모두 추적된다.

## 리스크

- **localized 문서 중복 관리 비용**: `docs/lifecycle/*.en.md`, `*.zh-CN.md`를 추가하면 원문 변경 시 동기화 비용이 생긴다. 이번 task에서는 README와 entrypoint가 직접 따라가는 최소 lifecycle 문서만 추가하고, 적용 후 운영 manual은 기존 `templates/locales`를 사용한다.
- **v0.3.0 배포 상태 혼동**: package/manifest를 v0.3.0으로 정렬하면 실제 publish가 된 것으로 오해할 수 있다. README와 release readiness 문서에 publish/tag/Homebrew 갱신은 별도 단계임을 명시한다.
- **CLI 역할 오해**: 현재 CLI는 자동 설치기가 아니라 lifecycle 판단 보고 도구다. `npx` 예시는 dry-run 판단 명령으로 표기하고, 실제 파일 변경은 approval workflow로 전환한다.
- **한국어 잔존 grep의 false positive**: README의 언어 선택 링크나 고유명사는 의도적으로 다국어 문자가 남을 수 있다. 검증에서는 의도한 잔존과 한국어-only anchor를 구분한다.
- **release 범위 확장**: npm publish, GitHub Release/tag, Homebrew tap 갱신을 이번 task에 포함하면 승인 게이트와 검증 범위가 커진다. 이번 task는 배포 직전 readiness까지만 다룬다.

## 승인 요청 사항

- README 3종 상단에 언어 지원 블록, 언어별 AI 적용 프롬프트, `npx ... --locale ... --dry-run` 예시를 추가하는 방향
- `docs/agent-entrypoint.en.md`, `docs/agent-entrypoint.zh-CN.md`와 English/Chinese lifecycle 문서를 공식 `docs/` 아래 추가하는 방향
- English/Chinese README의 깊은 manual 링크를 localized template/manual 문서 또는 언어 중립 링크로 정리하는 방향
- package/manifest/release 문서를 v0.3.0 배포 직전 상태로 정렬하되 실제 publish/tag/Homebrew 갱신은 제외하는 방향
- 위 5단계 Stage 분할과 예상 변경 파일 범위

승인되면 `task_m050_79_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
