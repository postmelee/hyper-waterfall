# 수행계획서

GitHub Issue: [#13](https://github.com/postmelee/hyper-waterfall/issues/13)
마일스톤: M020

## 목적

PR 본문의 `검증` 섹션을 실행 명령 나열에서 벗어나, 리뷰어가 검증 목적·방법·결과·근거·한계를 빠르게 판단할 수 있는 구조로 강화한다.

이번 task는 하이퍼-워터폴 PR 템플릿과 PR 작성 규칙을 함께 정렬해, 이후 내부 task PR이 검증 결과를 주제별로 압축하고 재현 가능한 근거를 남기도록 만드는 것을 목표로 한다.

## 배경

PR #12의 `검증` 섹션은 실제 명령을 많이 남겼지만, 각 명령이 어떤 수용 기준을 보장하는지와 실행 결과가 무엇이었는지 분리해 읽기 어렵다. 반면 rhwp PR #642는 로컬 검증, e2e 검증, GitHub Checks, 수동 검증 자료를 주제별로 나누고 결과를 체크해 리뷰어가 merge 판단 근거를 빠르게 확인할 수 있다.

React, VS Code, GitLab MR 사례도 PR/MR 본문에서 "어떻게 테스트했는가", "어떻게 로컬에서 검증하는가", "자료는 무엇인가"를 남기도록 유도한다. 이 작업은 그 방향을 하이퍼-워터폴 템플릿과 SKILL 절차에 맞게 반영한다.

## 범위

### 포함

- `.github/pull_request_template.md`의 `검증` 섹션 강화
- `templates/.github/pull_request_template.md`의 동일 변경 반영
- `templates/mydocs/manual/pr_process_guide.md`의 내부 task PR 작성 규칙 갱신
- `templates/mydocs/skills/task-final-report/SKILL.md`의 PR 본문 생성 기준 갱신
- SKILL/매뉴얼 변경에 따른 README와 `templates/mydocs/manual/task_workflow_guide.md` 동기화
- placeholder 보존과 dogfooding 위치/진실 원천 일치 검증

### 제외

- PR 생성 자동화 구현
- GitHub Actions 추가 또는 CI 정책 변경
- 새 label, milestone 생성
- 과거 PR 본문 일괄 수정
- 외부 기여 PR 검토 문서 템플릿 개편

## 설계 방향

- PR 본문의 `검증`은 `자동 검증`, `수동/시나리오 검증`, `CI/원격 검증`, `검증 한계`로 나눈다.
- 자동 검증과 수동 검증은 각각 `주제 / 검증 방법 / 결과 / 근거` 형식의 표를 사용한다.
- CI/원격 검증은 check 이름, 결과, run 링크 또는 확인 시점을 남기는 구조로 안내한다.
- 실행하지 않은 항목은 체크박스로 남기지 않고 `검증 한계` 또는 `남은 리스크`로 옮기도록 명시한다.
- 긴 로그를 PR 본문에 붙이지 않고, 핵심 출력·통과 개수·확인한 조건만 요약하도록 안내한다.
- `.github/` dogfooding 파일과 `templates/.github/` 진실 원천 파일은 구조를 맞추되, `templates/` 안의 `{REPO_SLUG}` placeholder는 보존한다.
- 매뉴얼과 SKILL을 함께 수정하므로 README의 핵심 SKILL/도입 후 작업 흐름 설명과 `task_workflow_guide.md`의 SKILL 호출 표시 안내를 같은 PR에서 확인하고 필요한 문구를 보정한다.

## 잠정 단계

- **Stage 1 — PR 템플릿 검증 섹션 개편**
  - `.github/pull_request_template.md`와 `templates/.github/pull_request_template.md`의 검증 목차를 강화한다.
  - 두 파일의 구조 일치와 placeholder 보존을 검증한다.
- **Stage 2 — PR 작성 규칙과 최종 보고 SKILL 동기화**
  - `pr_process_guide.md`와 `task-final-report/SKILL.md`에 새 검증 섹션 작성 기준을 반영한다.
  - PR 본문 생성 규칙이 명령 나열이 아니라 결과 요약을 요구하는지 검증한다.
- **Stage 3 — 상위 안내 동기화와 통합 검증**
  - README와 `task_workflow_guide.md`의 관련 설명을 확인하고 필요한 문구를 보정한다.
  - 전체 변경의 링크, placeholder, diff whitespace, 문서 간 규칙 일치를 검증한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `diff -u .github/pull_request_template.md templates/.github/pull_request_template.md`에서 `{REPO_SLUG}` 차이를 제외한 검증 섹션 구조가 일치하는지 확인
  - `grep -nE '자동 검증|수동/시나리오 검증|CI/원격 검증|검증 한계' .github/pull_request_template.md templates/.github/pull_request_template.md`
- Stage 2
  - `grep -nE '주제 / 검증 방법 / 결과 / 근거|검증 한계|실행하지 않은' templates/mydocs/manual/pr_process_guide.md templates/mydocs/skills/task-final-report/SKILL.md`
  - `git diff --check`
- Stage 3
  - `grep -nE 'task-final-report|PR 본문|검증 결과|SKILL 호출' README.md templates/mydocs/manual/task_workflow_guide.md`
  - `grep -n '\\{REPO_SLUG\\}' templates/.github/pull_request_template.md`

### 통합 검증

- PR 템플릿 두 벌의 검증 섹션 구조가 일치한다.
- `templates/` 안의 `{REPO_SLUG}` placeholder가 보존된다.
- `pr_process_guide.md`, `task-final-report/SKILL.md`, README, `task_workflow_guide.md`의 설명이 서로 충돌하지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **PR 본문 과밀화**: 표 구조가 너무 길어지면 리뷰 화면 가독성이 떨어질 수 있다. 각 표는 핵심 결과 요약만 담고, 긴 로그는 단계 보고서나 최종 보고서 링크로 넘긴다.
- **문서 간 규칙 불일치**: PR 템플릿만 바꾸면 SKILL이 예전 방식으로 PR 본문을 만들 수 있다. PR 템플릿, 매뉴얼, SKILL, README, task workflow 설명을 같은 task에서 맞춘다.
- **placeholder 손상**: dogfooding 파일과 template 원본을 함께 수정하면서 `{REPO_SLUG}`를 실제 저장소 값으로 치환할 위험이 있다. Stage별 grep으로 보존 여부를 확인한다.

## 승인 요청 사항

- 위 범위와 3단계 진행안으로 구현계획서를 작성해도 되는지 승인 요청한다.
- `README.md`와 `templates/mydocs/manual/task_workflow_guide.md`는 SKILL/매뉴얼 변경 동기화 범위에 포함해 필요한 문구만 최소 수정하는 방향으로 승인 요청한다.

승인되면 `task_m020_13_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
