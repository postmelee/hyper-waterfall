# Task M050 #65 수행 계획서

GitHub Issue: [#65](https://github.com/postmelee/hyper-waterfall/issues/65)
마일스톤: M050

## 목적

Hyper-Waterfall을 다국어 사용자가 자신의 저장소에 자신의 언어로 적용할 수 있게 하기 위한 언어 정책과 locale 구조를 먼저 고정한다.

이번 task는 README 번역이나 locale pack 작성 자체를 수행하지 않는다. 대신 M050 전체 이슈가 공통으로 참조할 기본 언어, 초기 지원 locale, fallback, canonical 원천, placeholder 보존, 번역 동기화, 한국어 고정 정책 대체 방향을 문서화한다.

## 배경

M050의 목표는 사용자가 Hyper-Waterfall 하네스를 적용할 때 README, 진입 프롬프트, 설치 템플릿, 생성 문서가 선택 언어로 정렬되는 것이다. 현재 저장소에는 `templates/AGENTS.md`의 "모든 문서는 한국어 작성", `_templates`의 "작성 언어: 한국어", `todo` Skill의 한국어 고정 규칙처럼 다국어 적용 목표와 충돌할 수 있는 문구가 있다.

README만 다국어화하면 외부 사용자는 프로젝트를 이해할 수 있지만, 실제 적용 저장소에 복사되는 템플릿과 작업 산출물은 여전히 한국어 기준으로 생성된다. 따라서 후속 #66~#71이 같은 기준으로 움직일 수 있도록 언어 정책과 locale 구조를 먼저 설계해야 한다.

## 범위

### 포함

- 기본 언어와 초기 지원 locale 정의: `en`, `ko`, `zh-CN`
- locale 코드와 파일명/디렉터리 명명 규칙 설계
- 사용자-facing 문서와 에이전트 운영 규칙 문서의 분리 기준 정의
- canonical 원천, fallback, 번역 동기화 원칙 정리
- `{REPO_SLUG}`, `{BASE_BRANCH}` 같은 placeholder 보존 규칙 정리
- 승인 게이트, 브랜치, 이슈, PR 규칙의 의미 보존 기준 정리
- 현재 한국어 고정 문구 inventory 작성
- 후속 #66~#71이 참조할 작업 경계와 의존성 정리

### 제외

- 실제 README 번역 작성
- 영어 또는 중국어 locale pack 전체 작성
- CLI, plugin, install/update workflow 구현 변경
- `templates/manifest.json`의 실제 locale 확장 구현
- GitHub label 또는 milestone 추가 생성
- 수행계획서 승인 전 코드, 매뉴얼, 템플릿 본문 변경

## 설계 방향

- M050의 기본 언어는 영어(`en`)로 둔다. 한국어는 기존 원문을 보존하는 `ko`, 중국어 간체는 `zh-CN`으로 명시한다.
- locale 코드는 BCP 47에 가까운 짧은 태그를 사용한다. 중국어는 모호한 `zh` 대신 `zh-CN`으로 시작하고, 번체가 필요하면 별도 `zh-TW` 후보로 둔다.
- 브랜치명, 파일명, placeholder, GitHub Issue/PR 번호, 명령어는 언어와 무관한 구조적 계약으로 유지한다.
- 사용자-facing 설명과 생성 문서 섹션명은 locale pack에서 번역 가능하게 두되, 승인 게이트와 단계 전환 조건의 의미는 locale 간 동일해야 한다.
- `templates/`의 placeholder 표기 보존 규칙은 기존 AGENTS 규칙을 유지한다. locale 설계에서도 placeholder를 번역하거나 치환하지 않는다.
- fallback은 선택 locale 파일이 없을 때 영어 기준으로 판단하되, 필수 템플릿 누락은 조용히 넘어가지 않고 적용/업데이트 보고에 드러내는 방향으로 설계한다.
- 실제 파일 배치와 manifest 확장은 #67에서 구현하므로, 이번 task는 구조 후보와 판단 기준을 문서화하는 데 집중한다.

## 리스크

- **정책과 구현 범위 혼합**: #65에서 실제 locale pack 구현까지 들어가면 후속 이슈 경계가 흐려진다. 이번 task는 정책과 inventory로 제한한다.
- **번역본 간 절차 의미 불일치**: 다국어 문서는 같은 승인 게이트를 다르게 표현할 위험이 있다. 구조적 계약과 사용자-facing 문구를 분리해 기준을 둔다.
- **placeholder 손상**: locale pack 설계 중 `{REPO_SLUG}` 같은 placeholder를 번역하거나 치환하면 적용 저장소에서 깨질 수 있다. 보존 규칙을 별도 수용 기준으로 둔다.
- **영어 기본화와 기존 한국어 dogfooding 충돌**: 이 저장소의 현재 문서는 한국어 중심으로 운영 중이다. 기본 언어 전환은 후속 #66~#70에서 단계적으로 처리하고, #65에서는 전환 정책만 고정한다.
- **fallback이 조용한 품질 저하를 만들 가능성**: 누락 locale을 영어로 대체할 수는 있지만, 필수 템플릿 누락은 보고서나 dry-run 결과에 명시하도록 설계한다.

## 승인 요청 사항

- #65를 실제 번역/구현이 아니라 다국어 정책과 locale 구조 설계 task로 제한하는 방향
- `docs/localization.md`를 M050 공통 정책 문서로 새로 만드는 방향
- `mydocs/tech/task_m050_65_language_inventory.md`에 현재 한국어 고정 문구와 후속 이슈 배정을 정리하는 방향
- 기본 언어를 `en`, 기존 한국어를 `ko`, 중국어 간체를 `zh-CN`으로 두는 초기 locale 기준
- placeholder, 브랜치명, 파일명, GitHub 번호, 명령어는 언어 독립 계약으로 유지하는 방향

승인되면 `task_m050_65_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
