# 다국어 적용 smoke 검증과 migration guide 작성 수행계획서

GitHub Issue: [#71](https://github.com/postmelee/hyper-waterfall/issues/71)
마일스톤: M050

## 목적

M050의 마지막 연결 작업으로, `en`, `ko`, `zh-CN` locale pack이 실제 신규 적용 판단과 기존 update 판단에서 재현 가능하게 검증되는지 확인한다. 또한 기존 한국어-only 적용 저장소가 M050 다국어 구조로 이동할 때 참고할 migration guide를 작성한다.

이번 task는 다국어 문서와 template을 추가하는 데서 끝나지 않고, 사용자가 Hyper-Waterfall 하네스를 자신의 저장소에 자신의 언어로 적용할 수 있다는 근거를 남기는 것이 목표다. smoke 결과, migration 기준, M050 종료 판단에 필요한 잔여 위험을 장기 문서로 정리한다.

## 배경

#65는 다국어 정책을 고정했고, #66은 README와 적용 진입 프롬프트를 다국어 기준으로 정렬했다. #67은 locale pack 구조와 manifest/update 기준을 추가했고, #68과 #69는 `en`, `ko`, `zh-CN` locale pack 본문을 채웠다. #70은 `.hyper-waterfall/version.json`의 top-level `locale` 저장 계약과 `init/update --locale` dry-run 판단 흐름을 연결했다.

#71은 이 선행 결과를 묶어 smoke 검증과 migration guide로 마무리해야 한다. 현재 CLI는 write mode가 아니라 dry-run 판단 보고만 제공하므로, 실제 외부 저장소 변경 대신 scratch directory 또는 dry-run 기반 재현 절차와 locale source 검증을 함께 사용한다.

## 범위

### 포함

- `en`, `ko`, `zh-CN` locale별 신규 적용 smoke 절차와 결과 문서화
- scratch directory 또는 dry-run 기반 적용 판단 재현 절차 정리
- locale별 placeholder 보존, 승인 게이트 의미, 생성 문서 template 언어, README 링크 정합성 확인
- `templates/manifest.json`과 locale pack source 정합성 검증
- 기존 한국어-only 적용 저장소가 M050 다국어 구조로 이동하는 migration guide 작성
- `locale`이 없는 기존 version state를 `unknown`으로 보고하고 `ko` 보존 또는 명시 전환을 승인받는 흐름 문서화
- M050 종료 판단에 필요한 잔여 위험과 후속 작업 후보 정리

### 제외

- 새 locale 추가
- 실제 외부 사용자 저장소 변경
- 승인 없는 GitHub Release/tag 생성
- npm publish, Homebrew/Docker/plugin 배포
- `init`/`update` write mode 구현
- locale pack 본문 전체 재번역
- 미완료 선행 이슈의 범위 대체 구현

## 설계 방향

- smoke 검증은 실제 파일 적용이 아니라 `init --locale <locale> --dry-run`, `update --locale <locale> --dry-run`, manifest source 검사, locale source 텍스트 검사로 구성한다.
- 장기 보관용 smoke 문서는 `docs/localization-smoke.md`에 둔다. `mydocs/working/` 보고서는 task 내부 진행 기록이고, M050 이후 maintainer와 에이전트가 재사용할 검증 matrix는 공식 `docs/`가 맞다.
- migration guide는 `docs/migrations/README.md`의 파일명 규칙을 따라 `docs/migrations/v0.2.0-to-v0.3.0.md` 후보로 작성한다. M050 milestone 설명의 목표 version은 `v0.3.0`이지만, release/tag 생성과 npm publish는 이번 task에서 수행하지 않는다.
- 기존 한국어-only 적용 저장소 migration은 "locale 기록이 없으면 `unknown`으로 보고한 뒤, 기존 한국어 운영 문맥을 보존하려면 `ko`를 선택 locale로 기록한다"는 흐름을 기본 시나리오로 둔다. `en` 또는 `zh-CN` 전환은 별도 승인 항목으로 설명한다.
- placeholder, branch, filename pattern, command, GitHub vocabulary는 번역하지 않는 구조적 계약으로 검증한다.
- README, lifecycle, plugin/distribution 문서는 최종 링크와 stale 표현만 확인하고, 새 중복 설명을 늘리지 않는다.

## 잠정 단계

- **Stage 1 — smoke matrix 설계와 baseline 검증**
  - `templates/manifest.json`, locale source, README/lifecycle 링크, #65-#70 산출물을 기준으로 smoke 항목을 확정한다.
  - `docs/localization-smoke.md`의 matrix 골격과 검증 기준을 작성한다.
  - 검증 관점: locale source 15개가 세 locale에 모두 존재하고, CLI dry-run이 각 locale을 지원 상태로 보고하는지 확인한다.
- **Stage 2 — locale별 smoke 실행과 결과 문서화**
  - `en`, `ko`, `zh-CN` 신규 적용 dry-run, unsupported locale fallback 후보, 기존 update 보존/전환 dry-run을 실행한다.
  - placeholder 보존, 승인 게이트 문구, 생성 문서 template 언어, 링크 정합성 검증 결과를 `docs/localization-smoke.md`에 채운다.
  - 검증 관점: smoke matrix가 세 locale 각각의 적용 가능 여부와 검증 결과를 기록한다.
- **Stage 3 — 기존 한국어-only migration guide 작성**
  - `docs/migrations/v0.2.0-to-v0.3.0.md`를 작성하고, 필요 시 `docs/migrations/README.md`에 locale migration 작성 기준을 보강한다.
  - `locale`이 없는 version state의 `unknown` 처리, `ko` 보존, 명시적 `en`/`zh-CN` 전환 승인, manual/Skill 충돌 가능성을 설명한다.
  - 검증 관점: migration guide 필수 섹션과 locale/version/manifest 근거가 모두 존재하는지 확인한다.
- **Stage 4 — 최종 링크 정합성 및 M050 종료 점검**
  - README, lifecycle, plugin/distribution, localization 정책에서 smoke/migration 링크와 stale 표현을 최종 확인한다.
  - 필요 시 `templates/manifest.json`의 availability note 또는 `test/cli-smoke.test.js`를 M050 완료 상태에 맞춘다.
  - 검증 관점: M050 종료 판단에 필요한 잔여 위험과 후속 작업 후보가 최종 보고서에 남는다.

- README, lifecycle, plugin/distribution 문서의 최종 링크가 깨지지 않는다.
- M050 milestone 종료 판단에 필요한 잔여 위험과 후속 작업 후보가 정리되어 있다.

## 리스크

- **write mode 부재로 인한 smoke 한계**: 현재 CLI는 dry-run만 제공한다. 실제 파일 복사 검증은 하지 않고, dry-run 출력과 locale source 검증을 결합해 한계를 명시한다.
- **v0.3.0 release 상태 혼동**: M050은 `v0.3.0` 목표 milestone이지만, 이번 task는 release/tag 생성과 npm publish를 제외한다. migration guide에는 release 상태와 보류 항목을 명확히 적는다.
- **한국어-only migration 모호성**: 기존 저장소가 한국어 문맥인지 기계적으로 확정하기 어렵다. version state에 `locale`이 없으면 `unknown`으로 보고하고 maintainer 승인으로 `ko` 보존 또는 전환을 결정하게 한다.
- **문서 중복 증가**: README에 smoke/migration 상세를 넣으면 진입 문서가 무거워진다. 상세는 `docs/localization-smoke.md`와 migration guide에 두고 README는 링크 정합성만 확인한다.
- **#71 범위 초과**: 새 locale 추가, write mode, release publish는 수행하지 않고 후속 작업 후보로만 기록한다.

## 승인 요청 사항

- smoke 결과 장기 문서를 `docs/localization-smoke.md`에 작성하는 방향
- 기존 한국어-only migration guide를 `docs/migrations/v0.2.0-to-v0.3.0.md`로 작성하는 방향
- smoke는 실제 외부 저장소 변경 없이 scratch directory와 CLI dry-run, locale source 검사로 수행하는 방향
- release/tag 생성, npm publish, write mode 구현을 #71 범위에서 제외하는 방향
- 위 4단계 Stage 분할과 예상 변경 파일 범위

승인되면 `task_m050_71_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
