# task_m020_24.md - v0.2.0 GitHub Release/tag 준비 수행계획서

GitHub Issue: [#24](https://github.com/postmelee/hyper-waterfall/issues/24)
마일스톤: M020

## 목적

`v0.2.0` GitHub Release/tag를 만들기 전에 확인해야 할 기준과 승인 지점을 문서로 고정한다. Hyper-Waterfall의 canonical 배포 단위가 GitHub Release/tag + `templates/manifest.json` + migration guide라는 원칙을 유지하면서, npm publish나 Homebrew PoC 전에 M020 release 준비 상태를 재현 가능하게 검증한다.

이번 task의 결과는 release/tag 생성 전 체크리스트, release notes 초안, manifest/migration/README/package version 정합성 점검이다. 실제 tag 생성과 GitHub Release publish는 작업지시자 승인 후 별도 명시 지시가 있을 때만 수행한다.

## 배경

현재 `package.json`의 version은 `0.2.0`이고, `templates/manifest.json`은 `frameworkVersion: 0.2.0`, `plannedTag: v0.2.0`, `baselineTag: v0.1.0`, checksum `pending-release` 상태를 가리킨다. `docs/migrations/v0.1.0-to-v0.2.0.md`도 release 생성 전 update protocol 기준 문서로 작성되어 있다.

작업 시작 시점인 2026-05-08 기준으로 `git tag --list --sort=version:refname`와 `gh release list --repo postmelee/hyper-waterfall --limit 20`는 출력이 없어, 로컬/원격 확인 가능한 Git tag와 GitHub Release가 아직 없음을 확인했다. 메인 worktree에는 작업지시자가 만든 것으로 보이는 미추적 파일 `DESIGN.md`, `README_draft.md`, `opus_report.md`가 있으므로 이번 task는 `/private/tmp/hyper-waterfall-task24` 분리 worktree에서 진행한다.

## 설계 방향

- release 준비 기준은 `docs/releases/v0.2.0.md` 같은 release 전용 문서에 모아, migration guide와 배포 채널 전략 문서가 체크리스트 세부 항목으로 과밀해지지 않게 한다.
- `templates/manifest.json`의 checksum은 실제 release/tag 생성 시점에 확정 가능한 값으로 남겨야 하므로, 이번 task에서 임의로 값을 채우지 않는다. 필요하면 `pending-release` 의미와 확정 조건을 문서에 명확히 적는다.
- migration guide는 기존 적용 저장소가 `v0.1.0 -> v0.2.0`으로 이동할 때의 판단 근거로 유지하고, release 준비 문서는 프레임워크 저장소 maintainer가 tag/release 전에 확인할 운영 체크리스트로 둔다.
- README와 `docs/distribution-channels.md`는 canonical 기준을 안내하는 허브 역할에 그치며, 상세 release notes와 체크리스트는 release 준비 문서로 연결한다.
- 실제 release/tag 생성 명령은 최종 보고 이후에도 자동 실행하지 않고, 작업지시자가 생성 시점을 명시 승인한 뒤 별도 명령으로 진행한다.

## 잠정 단계

- **Stage 1 — release 준비 기준과 체크리스트 작성**
  - `docs/releases/v0.2.0.md`를 신설해 release 전 확인 항목, release notes 초안, 승인 게이트, 보류 항목을 정리한다.
  - 검증 관점: GitHub Release/tag가 canonical 기준이고, 실제 생성은 승인 전 보류된다는 점이 문서에서 분명한지 확인한다.
- **Stage 2 — manifest, migration, 안내 문서 정합성 보강**
  - `templates/manifest.json`, `docs/migrations/v0.1.0-to-v0.2.0.md`, `README.md`, `docs/distribution-channels.md`, `package.json`의 version/tag/canonical 설명을 대조하고 필요한 최소 문서 보강을 적용한다.
  - 검증 관점: `0.2.0`, `v0.2.0`, `plannedTag`, `baselineTag`, `pending-release`, migration guide 설명이 서로 충돌하지 않는지 확인한다.
- **Stage 3 — release/tag 생성 승인 경계와 통합 검증**
  - tag/release 미생성 상태, checksum 확정 조건, npm publish와 후속 채널 의존 관계를 최종 점검하고 생성 승인 요청 항목을 정리한다.
  - 검증 관점: 이슈의 검증 명령을 통과하고, 실제 tag/release 생성 명령은 실행되지 않았음을 확인한다.

## 리스크

- **release checklist와 migration guide 역할 혼선**: migration guide는 적용 저장소 업데이트 판단 근거이고, release 준비 문서는 프레임워크 maintainer 체크리스트다. 두 문서의 역할을 명확히 분리한다.
- **checksum 확정 시점 오해**: release 전 `pending-release` 상태를 임의로 해소하면 실제 tag 기준 checksum과 어긋날 수 있다. 이번 task는 확정 조건만 적고 값은 생성 시점 승인 뒤 처리한다.
- **실제 배포 실행 오인**: release notes 초안과 체크리스트가 있어도 tag 생성, GitHub Release publish, npm publish는 실행하지 않는다. 최종 보고에서 별도 승인 요청으로 분리한다.
- **동시 작업 충돌**: 메인 worktree에 미추적 파일이 있으므로 분리 worktree에서 작업하고, 해당 파일은 건드리지 않는다.
