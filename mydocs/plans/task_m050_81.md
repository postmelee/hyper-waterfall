# v0.3.0 release 실행과 post-publish 검증 수행계획서

GitHub Issue: [#81](https://github.com/postmelee/hyper-waterfall/issues/81)
마일스톤: M050

## 목적

M050 다국어 적용 지원 산출물을 실제 `v0.3.0` 배포로 확정한다. release 전 검증을 통해 `package.json`, `templates/manifest.json`, README, localized docs, locale pack이 같은 배포 기준을 가리키는지 확인하고, 승인된 시점에 GitHub tag/Release와 npm publish를 실행한다.

배포 후에는 npm registry와 빈 임시 디렉터리 기준 `npx hyper-waterfall@0.3.0` smoke를 확인하고, release 실행 결과와 남은 후속 작업을 문서와 최종 보고서에 남긴다.

## 배경

#65-#71과 #79에서 M050의 locale 정책, README 다국어화, locale pack, localized lifecycle 문서, migration guide, v0.3.0 release readiness 정렬을 완료했다. 현재 `package.json`과 `templates/manifest.json`은 `0.3.0` / `v0.3.0`을 가리키지만, GitHub Release와 npm registry는 아직 `v0.2.0`까지만 존재한다.

`docs/releases/v0.3.0.md`에는 release readiness, npm tarball 체크리스트, release notes 초안, publish 후 검증 기준이 준비되어 있다. 이번 task는 그 체크리스트를 실제 배포 실행과 post-publish 검증으로 전환한다.

## 범위

### 포함

- `v0.3.0` release 전 기준 정합성 확인
- `npm test`, locale별 `init --dry-run`, `npm pack --dry-run` 검증
- `npm whoami`, npm registry 기존 version, GitHub Release/tag 기존 상태 확인
- 작업지시자의 별도 승인 후 `v0.3.0` Git tag push와 GitHub Release 생성
- 작업지시자의 별도 승인 후 `hyper-waterfall@0.3.0` npm publish
- publish 후 `npm view`, `npx --version`, `--help`, locale별 dry-run smoke 검증
- `docs/releases/v0.3.0.md`, `docs/distribution-channels.md` 등 release 상태 문서의 필요한 갱신
- 단계 보고서와 최종 보고서에 승인 지점, 대상 commit, registry URL, smoke 결과 기록

### 제외

- Homebrew tap formula 갱신 또는 public tap 배포
- Docker image 구현 또는 registry 배포
- Codex/Claude official plugin 배포
- 자동 release pipeline 구현
- write mode 자동 적용
- npm OTP, auth token, maintainer 비밀값 기록
- 같은 npm version이 이미 존재할 때의 강행 publish, unpublish, dist-tag 임의 수정

## 설계 방향

- canonical 배포 단위는 GitHub Release/tag이고, npm package는 canonical protocol을 실행하기 쉽게 하는 편의 실행 채널로 유지한다.
- release/tag/npm publish는 수행계획서 승인만으로 실행하지 않는다. Stage 1 preflight와 Stage 2 package smoke가 통과한 뒤 작업지시자의 별도 명시 승인을 다시 받는다.
- publish 대상은 `origin/main`의 검증된 release commit으로 고정한다. task 브랜치의 `mydocs/` 산출물은 release package 기준에 섞지 않는다.
- `npm publish`는 별도 release worktree 또는 명확한 clean checkout에서 실행해 task branch 변경이 package 산출물에 섞이지 않게 한다.
- `npm pack --dry-run` 결과에서 다국어 README, localized docs, `templates/locales` 포함 여부를 확인한다.
- 배포 후 문서 갱신은 실제 상태만 반영한다. Homebrew tap이 아직 `0.2.0`이면 npm/GitHub는 `0.3.0`, Homebrew는 후속 갱신 필요 상태로 분리해 기록한다.
- OTP나 token 같은 민감값은 채팅, 문서, 보고서에 기록하지 않는다. 2FA가 필요하면 작업지시자가 직접 입력 가능한 방식으로 진행한다.

## 잠정 단계

- **Stage 1 — release 기준 정합성 preflight**
  - `origin/main` commit, `package.json`, `templates/manifest.json`, migration guide, release readiness 문서, 기존 GitHub Release/tag, npm registry 상태를 확인한다.
  - publish 계정과 동일 version 존재 여부를 확인한다.
  - 검증 관점: 배포 대상 commit과 version 기준이 고정되고, 중복 release/publish 조건이 없거나 중단 조건이 명확히 보고된다.
- **Stage 2 — package와 locale smoke 검증**
  - `npm test`, locale별 `node bin/hyper-waterfall.js init --locale ... --dry-run`, `npm pack --dry-run`, 필요 시 local tarball smoke를 실행한다.
  - tarball에 README 3종, localized docs, `templates/locales`가 포함되는지 확인한다.
  - 검증 관점: npm publish 직전 package 산출물이 M050 다국어 배포 기준을 충족한다.
- **Stage 3 — release/tag와 npm publish 실행**
  - Stage 1-2 결과를 근거로 작업지시자에게 `git tag`/`gh release create`/`npm publish` 별도 승인을 요청한다.
  - 승인 후 `v0.3.0` Git tag push, GitHub Release 생성, npm publish를 실행한다.
  - 검증 관점: 외부 상태 변경 명령, 대상 commit, release URL, registry 1차 조회 결과가 stage 보고서에 기록된다.
- **Stage 4 — post-publish 검증과 문서 상태 갱신**
  - `npm view`, 빈 임시 디렉터리 기준 `npx hyper-waterfall@0.3.0 --version`/`--help`, locale별 dry-run을 검증한다.
  - `docs/releases/v0.3.0.md`, `docs/distribution-channels.md`, 필요 시 README 3종을 실제 publish 상태에 맞춘다.
  - 검증 관점: npm/GitHub는 `0.3.0` published 상태로, Homebrew 등 제외 채널은 후속 필요 상태로 분리되어 있다.
- **Stage 5 — 통합 검증과 최종 보고**
  - 전체 테스트, 문서 grep, diff 검증, 작업 tree 상태를 확인한다.
  - 최종 보고서에 release 실행 결과, post-publish smoke, 후속 Homebrew tap 갱신 후보를 정리한다.
  - 검증 관점: #81 수용 기준이 모두 추적되고 PR 게시 준비가 끝난다.

## 리스크

- **동일 version 재배포 불가**: `0.3.0`을 잘못 publish하면 같은 version으로 덮어쓸 수 없다. Stage 1-2 preflight와 별도 publish 승인 게이트로 대응한다.
- **release commit drift**: tag, GitHub Release, npm tarball이 서로 다른 commit을 대표하면 update 기준이 흔들린다. 대상 commit을 Stage 1에서 고정하고 Stage 3 보고서에 기록한다.
- **2FA 또는 계정 권한 문제**: npm publish 과정에서 OTP나 권한 확인이 필요할 수 있다. 민감값은 기록하지 않고, 작업지시자가 직접 처리 가능한 흐름으로 둔다.
- **네트워크/registry 지연**: publish 직후 registry나 npx 캐시가 즉시 반영되지 않을 수 있다. 실패 명령, 시각, stderr 요약을 남기고 재시도 여부를 별도 판단한다.
- **문서와 채널 상태 차이**: npm/GitHub는 `0.3.0`으로 갱신되더라도 Homebrew tap은 후속 작업 전까지 `0.2.0`일 수 있다. 배포 채널 문서에서 채널별 상태를 분리한다.
- **release 범위 확장**: Homebrew, Docker, plugin 배포까지 포함하면 승인 게이트와 외부 상태 변경 범위가 커진다. 이번 task에서는 명시적으로 제외하고 후속 이슈 후보로 남긴다.

## 승인 요청 사항

- 위 범위와 5단계 구성으로 Task #81을 진행해도 되는지 승인 요청한다.
- Stage 1-2 검증이 통과해도 `git tag`, `gh release create`, `npm publish`는 별도 명시 승인 전에는 실행하지 않는다.
- `origin/main`의 검증된 release commit 기준으로 tag/release/npm publish를 수행하고, task branch에는 실행 결과 문서와 보고서를 남기는 방침을 승인 요청한다.
- Homebrew tap formula 갱신, Docker, plugin 배포, 자동 release pipeline은 이번 task에서 제외하고 후속 작업 후보로 분리하는 방침을 승인 요청한다.

승인되면 `task_m050_81_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
