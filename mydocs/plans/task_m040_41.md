# Task #41 수행계획서 - 최종 배포 채널 정합성 감사와 문서 갱신

GitHub Issue: [#41](https://github.com/postmelee/hyper-waterfall/issues/41)
마일스톤: M040

## 목적

M040에서 진행한 npm, Homebrew, Codex plugin, Claude plugin 배포 채널 작업 결과를 한 번에 대조하고, 사용자에게 노출되는 README와 배포 문서가 실제 상태와 충돌하지 않게 정리한다.

이번 task의 완료 기준은 새 배포를 추가로 실행하는 것이 아니라, 이미 완료된 #33-#40, #52, #54의 검증 결과와 현재 외부 배포 상태를 기준으로 `planned`, `PoC`, `public`, `보류` 표현을 정확히 맞추는 것이다. Docker는 M040 범위에서 제외된 상태를 유지하고, plugin public 배포와 release asset은 #52/#54의 최종 판단을 반영해 후속 조건으로 분리한다.

## 배경

이슈 #41은 M040의 마지막 정합성 감사 task다. 선행 이슈 #33-#40은 모두 closed 상태이며, 추가 최종 판단 입력으로 지정된 #52와 #54도 closed 상태다.

확인된 선행 결과:

- #33: `hyper-waterfall@0.2.0` npm publish와 post-publish smoke 완료
- #34: Homebrew formula local tap smoke PoC 완료
- #35: `postmelee/tap/hyper-waterfall` public tap 배포와 설치 smoke 완료
- #36: Codex/Claude plugin 공통 배포 원칙 정리 완료
- #37-#40: Codex/Claude plugin packaging과 local/zip 후보 smoke 완료
- #52: Codex plugin repo-local discovery 확인, official public 배포는 보류
- #54: Claude plugin public-readiness와 zip 후보 smoke 확인, Official directory 제출과 release asset 게시는 #41에서 재결정

따라서 이번 task는 배포 채널별 실제 상태를 재확인한 뒤, 사용자 문서가 그 상태를 안정 채널처럼 과장하거나 아직 가능한 경로를 planned로 남기지 않도록 정리한다.

## 설계 방향

- 배포 상태는 외부 상태와 저장소 문서를 양방향으로 대조한다. 문서만 읽고 판단하지 않고, 가능한 채널은 `npm view`, `npx`, `brew`, `gh release`, `gh issue` 등으로 현재 상태를 확인한다.
- 사용자-facing 문서는 "사용자가 지금 실행할 수 있는 경로"와 "후속 검토 후보"를 명확히 분리한다.
- canonical 기준은 GitHub Release/tag + `templates/manifest.json` + migration guide로 유지한다. npm, Homebrew, plugin은 canonical protocol을 실행하거나 발견하기 쉽게 하는 채널로만 설명한다.
- `templates/manifest.json`의 release 상태가 실제 GitHub Release/tag 상태와 충돌하면, 변경 필요 여부를 Stage 1에서 판단하고 Stage 2 구현계획서에서 별도 변경 항목으로 고정한다.
- Codex/Claude plugin public 배포와 release asset 게시 여부는 #52/#54의 보류 조건과 작업지시자 승인 게이트를 유지한다. #41에서는 문서 정합성만 다루고 외부 공개 action은 실행하지 않는다.
- manual이나 SKILL 본문 변경이 필요하다고 판단되면 이번 task에서 즉시 고치지 않고 작업지시자에게 범위 변경 승인을 요청한다. 해당 변경은 README 핵심 SKILL 표와 `task_workflow_guide.md` 동시 갱신 의무가 있기 때문이다.

## 리스크

- **외부 상태 변동**: npm registry, Homebrew tap, GitHub Release, plugin directory 상태는 변경될 수 있다. Stage 1에서 실제 명령으로 현재 상태를 다시 확인하고 문서에는 확인 시점을 남긴다.
- **manifest 상태 충돌 가능성**: `templates/manifest.json`이 release 전 상태를 유지하고 있으면 실제 `v0.2.0` release/tag 상태와 충돌할 수 있다. 변경 필요 여부를 Stage 1에서 별도 판단한다.
- **plugin public 상태 과장**: #52/#54는 public 배포를 보류하거나 #41 재결정으로 넘겼다. 사용자 문서에서 plugin을 stable public channel처럼 표현하지 않는다.
- **공통 문서 변경 범위 확대**: README, release 문서, distribution 문서가 서로 참조해 작은 표현 수정이 여러 파일로 번질 수 있다. 사용자-facing 충돌 제거에 필요한 최소 변경으로 제한한다.
- **manual/SKILL 동시 갱신 의무**: manual이나 SKILL 본문 변경이 필요하면 README 핵심 SKILL 표와 workflow guide 동시 갱신 규칙이 작동한다. 이번 task의 기본 범위에서는 해당 변경을 제외한다.
- **네트워크 의존 검증**: npm, Homebrew, GitHub 조회는 네트워크나 인증 상태에 의존한다. 실패하면 실패 사유와 대체 확인 근거를 Stage 보고서에 분리해 기록한다.
