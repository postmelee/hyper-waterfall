# v0.1.0-to-v0.2.0 migration guide

이 문서는 Hyper-Waterfall을 이미 적용한 저장소가 `v0.1.0` baseline에서 M020 release인 `v0.2.0`으로 이동할 때 확인해야 할 항목을 정리한다. 최초 작성 시점에는 GitHub Release/tag가 아직 생성되지 않았지만, 2026-05-19 기준 `v0.2.0` GitHub Release/tag와 npm package는 공개 완료 상태다. 본 문서는 release 후에도 update protocol의 기준 문서로 유지한다.

## 대상 버전

| 항목 | 값 |
|---|---|
| 시작 version | `v0.1.0` |
| 목표 version | `v0.2.0` |
| canonical 배포 단위 | GitHub Release/tag |
| manifest 위치 | `templates/manifest.json` |
| release 상태 | `released` |
| 적용 저장소 version 기록 | `.hyper-waterfall/version.json` |
| checksum 상태 | file `ready`, root/directory `pending-release`, symlink `not-applicable` |

## 변경 요약

- Hyper-Waterfall 배포 기준을 프롬프트 요청만이 아니라 GitHub Release/tag + manifest + migration guide로 정의한다.
- `templates/manifest.json`을 추가해 적용 대상 파일, 대상 경로, update policy, checksum 상태를 기계가 읽을 수 있는 JSON으로 표현한다.
- 단일 파일 checksum은 source 파일 SHA-256 기준으로 확정하고, root/directory checksum은 공식 산식 확정 전까지 `pending-release`로 둔다.
- 적용 저장소의 현재 framework version 기록 위치를 `.hyper-waterfall/version.json`으로 정한다.
- migration guide 위치와 필수 섹션을 `docs/migrations/README.md`에 정의한다.
- lifecycle 판단, Hyper-Waterfall 버전 업데이트 PR 워크플로우, npm CLI가 참조할 공통 기준을 만든다.

## 추가 파일

| 경로 | 목적 | 적용 저장소 처리 |
|---|---|---|
| `templates/manifest.json` | release 포함 파일과 update policy 정의 | 프레임워크 원본으로 유지. 적용 저장소에는 직접 복사하기보다 update 도구가 참조 |
| `docs/migrations/README.md` | migration guide 작성 규칙 | 프레임워크 저장소 문서. 적용 저장소 업데이트 시 참고 링크로 사용 |
| `docs/migrations/v0.1.0-to-v0.2.0.md` | 이번 version 이동의 수동 확인 항목 | Hyper-Waterfall 버전 업데이트 PR 본문과 리뷰 근거로 사용 |
| `.hyper-waterfall/version.json` | 적용 저장소의 설치/업데이트 version 상태 | 적용 저장소에 신규 생성. 기존 파일이 있으면 보존 후 diff 확인 |

## 수정 파일

| 경로 | 변경 내용 | update policy 관점 |
|---|---|---|
| `templates/mydocs/manual/document_structure_guide.md` | 배포 manifest, version 기록, update policy 설명 추가 | `overwrite` 후보이지만 적용 저장소에서 manual을 수정했다면 충돌 확인 |
| `README.md` | 신규 적용 프롬프트와 기존 업데이트 lifecycle 안내를 짧게 연결 | `merge` 후보 |
| `docs/agent-entrypoint.md` | 신규 적용과 기존 업데이트 진입 절차, 판단 결과 형식 정의 | `merge` 후보 |
| `templates/mydocs/manual/git_workflow_guide.md` | GitHub Release/tag와 update protocol 관계 설명 | `overwrite` 후보, 로컬 수정 시 충돌 확인 |
| `templates/mydocs/manual/task_workflow_guide.md` | framework lifecycle 판단을 일반 타스크 흐름으로 전환하는 기준 설명 | `overwrite` 후보, 로컬 수정 시 충돌 확인 |

## 적용 저장소 수동 확인

적용 저장소를 `v0.2.0`으로 올릴 때 maintainer는 다음을 확인한다.

- `.hyper-waterfall/version.json`이 없으면 새로 생성한다.
- `.hyper-waterfall/version.json`이 이미 있으면 `frameworkVersion`, `releaseTag`, `source`, `updatedAt` 값을 확인하고 임의로 덮어쓰지 않는다.
- `AGENTS.md`와 `CLAUDE.md`는 프로젝트별 규칙을 담을 수 있으므로 manifest가 바뀌어도 자동 overwrite하지 않는다.
- `.github/ISSUE_TEMPLATE/task.yml`과 `.github/pull_request_template.md`는 저장소별 운영 방식에 맞게 수정됐을 수 있으므로 Hyper-Waterfall 버전 업데이트 PR에서 diff를 리뷰한다.
- `mydocs/manual/`과 `mydocs/skills/`가 symlink인지 복사본인지 확인한다. 본 프레임워크 저장소는 dogfooding을 위해 symlink를 쓰지만, 적용 저장소는 일반 복사본일 수 있다.
- `mydocs/_templates/`는 문서 출력 형식의 진실 원천이므로 기존 사용자 수정 여부를 확인한 뒤 update policy에 따라 처리한다.

## 충돌 가능성

| 영역 | 충돌 이유 | 기본 처리 |
|---|---|---|
| `AGENTS.md` | 프로젝트별 운영 규칙이 추가되는 파일 | `merge`, 수동 리뷰 |
| `CLAUDE.md` | Claude Code 진입 방식이 프로젝트별로 달라질 수 있음 | `merge`, 수동 리뷰 |
| `.github/` | 이슈/PR 템플릿은 저장소 운영 관습에 맞게 수정될 수 있음 | `merge`, 수동 리뷰 |
| `mydocs/manual/` | 사용자가 절차를 로컬 정책에 맞게 보강했을 수 있음 | directory checksum이 `pending-release`이므로 manifest diff와 파일 단위 diff를 수동 확인 |
| `mydocs/skills/` | Skill 본문은 프레임워크 절차 핵심이지만 로컬 수정 가능성 있음 | directory checksum이 `pending-release`이므로 manifest diff와 파일 단위 diff를 수동 확인 |
| `.agents/skills`, `.claude/skills` | 파일/폴더로 존재하면 symlink 생성과 충돌 | `symlink`, 기존 경로 확인 후 승인 필요 |
| `.hyper-waterfall/version.json` | 상태 파일이므로 기존 기록을 보존해야 함 | `preserve`, 누락 시 생성 |

## 검증

프레임워크 저장소에서 이번 migration guide 자체는 다음 명령으로 검증한다.

```bash
test -f docs/migrations/README.md
test -f docs/migrations/v0.1.0-to-v0.2.0.md
grep -nE '파일명 규칙|필수 섹션|대상 버전|추가 파일|수정 파일|수동 확인|충돌 가능성|검증' docs/migrations/README.md docs/migrations/v0.1.0-to-v0.2.0.md
grep -nE 'v0.1.0|v0.2.0|manifest|version|GitHub Release' docs/migrations/v0.1.0-to-v0.2.0.md
git diff --check
```

적용 저장소 Hyper-Waterfall 버전 업데이트 PR에서는 다음 확인을 추가한다.

- 현재 `.hyper-waterfall/version.json`의 `frameworkVersion`과 목표 version 비교
- `templates/manifest.json` 기준 source/target 경로 존재 여부 확인
- 사용자 수정 파일과 manifest checksum 불일치 항목 확인
- 수동 확인 항목이 PR 본문에 남았는지 확인

## Hyper-Waterfall 버전 업데이트 PR 본문 반영 기준

이 migration guide를 근거로 Hyper-Waterfall 버전 업데이트 PR을 만들 때는 기존 PR 템플릿을 사용하되, 다음 항목을 반드시 남긴다.

| PR 본문 항목 | 이번 migration에서 채울 내용 |
|---|---|
| 현재 version | `v0.1.0` |
| 목표 release/tag | `v0.2.0` |
| migration guide | `docs/migrations/v0.1.0-to-v0.2.0.md` |
| manifest diff | `templates/manifest.json`, `.hyper-waterfall/version.json`, migration guide 추가와 관련 문서 수정 |
| 자동 적용 가능 | checksum이 `ready`이고 사용자 수정이 없는 file 대상. directory/root checksum은 `pending-release`이므로 자동 적용하지 않음 |
| 수동 확인 필요 | `AGENTS.md`, `CLAUDE.md`, `.github/`, `mydocs/_templates/`, 저장소별로 수정됐을 가능성이 있는 Manual/Skill |
| conflict | 사용자 수정 파일과 목표 release 변경이 같은 경로에 걸리는 항목 |
| 보류 | root/directory checksum 산식 확정, Homebrew core 제출, Docker image, plugin official public 배포, release automation |

## CLI 출력 기준

`v0.1.0 -> v0.2.0` init/update/doctor 출력은 이 문서를 근거로 다음 항목을 보여줘야 한다. 실제 파일 변경이나 PR 생성은 이 출력만으로 자동 수행하지 않고, 작업지시자 승인 후 일반 task 흐름으로 전환한다.

| 출력 항목 | 이번 migration 기준 |
|---|---|
| 현재 version | `.hyper-waterfall/version.json`의 `frameworkVersion`이 없거나 `v0.1.0` baseline인지 확인 |
| 목표 release/tag | `v0.2.0` 또는 M020에서 확정될 release tag |
| migration guide | `docs/migrations/v0.1.0-to-v0.2.0.md` |
| manifest diff | `templates/manifest.json` 추가, `.hyper-waterfall/version.json` 생성/보존, lifecycle 관련 문서 변경 |
| 자동 적용 가능 | checksum이 `ready`이고 대상 파일이 사용자 수정 없이 이전 manifest와 일치하는 file 후보. directory/root checksum은 `pending-release`이므로 수동 확인 |
| 수동 확인 필요 | `AGENTS.md`, `CLAUDE.md`, `.github/`, `mydocs/_templates/`, 로컬 수정 가능성이 있는 Manual/Skill |
| conflict | 사용자 수정 파일과 목표 release 변경이 같은 경로를 수정하는 경우, 또는 symlink 대상이 기존 파일/폴더와 충돌하는 경우 |
| 보류 | root/directory checksum 산식 확정, Homebrew core 제출, Docker image, Codex official public 배포, Claude Official directory 제출과 release asset 게시, release automation |
| 검증 | manifest 파싱, migration guide 필수 섹션 확인, version 기록 확인, file checksum 불일치 항목과 directory/root `pending-release` 항목 확인 |
| 승인 요청 | Hyper-Waterfall 버전 업데이트 PR 생성 여부와 PR에 포함할 범위 |

## 후속 작업

- #9 Hyper-Waterfall 버전 업데이트 PR 워크플로우는 이 migration guide의 수동 확인과 충돌 가능성 섹션을 PR 본문에 요약한다.
