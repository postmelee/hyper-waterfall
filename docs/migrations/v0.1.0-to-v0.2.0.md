# v0.1.0-to-v0.2.0 migration guide

이 문서는 Hyper-Waterfall을 이미 적용한 저장소가 `v0.1.0` baseline에서 M020 목표 release인 `v0.2.0`으로 이동할 때 확인해야 할 항목을 정리한다. 실제 GitHub Release/tag는 이 문서 작성 시점에 아직 생성하지 않았으며, 본 문서는 release 생성 전 update protocol의 기준 문서다.

## 대상 버전

| 항목 | 값 |
|---|---|
| 시작 version | `v0.1.0` |
| 목표 version | `v0.2.0` |
| canonical 배포 단위 | GitHub Release/tag |
| manifest 위치 | `templates/manifest.json` |
| 적용 저장소 version 기록 | `.hyper-waterfall/version.json` |
| checksum 상태 | `pending-release` |

## 변경 요약

- Hyper-Waterfall 배포 기준을 프롬프트 요청만이 아니라 GitHub Release/tag + manifest + migration guide로 정의한다.
- `templates/manifest.json`을 추가해 적용 대상 파일, 대상 경로, update policy, checksum 상태를 기계가 읽을 수 있는 JSON으로 표현한다.
- 적용 저장소의 현재 framework version 기록 위치를 `.hyper-waterfall/version.json`으로 정한다.
- migration guide 위치와 필수 섹션을 `docs/migrations/README.md`에 정의한다.
- 후속 `framework-install`, `framework-update`, update PR workflow, npm CLI가 참조할 공통 기준을 만든다.

## 추가 파일

| 경로 | 목적 | 적용 저장소 처리 |
|---|---|---|
| `templates/manifest.json` | release 포함 파일과 update policy 정의 | 프레임워크 원본으로 유지. 적용 저장소에는 직접 복사하기보다 update 도구가 참조 |
| `docs/migrations/README.md` | migration guide 작성 규칙 | 프레임워크 저장소 문서. 적용 저장소 업데이트 시 참고 링크로 사용 |
| `docs/migrations/v0.1.0-to-v0.2.0.md` | 이번 version 이동의 수동 확인 항목 | update PR 본문과 리뷰 근거로 사용 |
| `.hyper-waterfall/version.json` | 적용 저장소의 설치/업데이트 version 상태 | 적용 저장소에 신규 생성. 기존 파일이 있으면 보존 후 diff 확인 |

| 경로 | 변경 내용 | update policy 관점 |
|---|---|---|

| `README.md` | Stage 3에서 배포·업데이트 모델 설명 추가 예정 | `merge` 후보 |

| `templates/mydocs/manual/git_workflow_guide.md` | Stage 3에서 GitHub Release/tag와 update protocol 관계 보강 예정 | `overwrite` 후보, 로컬 수정 시 충돌 확인 |
| `templates/mydocs/manual/task_workflow_guide.md` | Stage 3에서 후속 install/update Skill 참조 위치 보강 예정 | `overwrite` 후보, 로컬 수정 시 충돌 확인 |

## 적용 저장소 수동 확인

적용 저장소를 `v0.2.0`으로 올릴 때 maintainer는 다음을 확인한다.

- `.hyper-waterfall/version.json`이 없으면 새로 생성한다.
- `.hyper-waterfall/version.json`이 이미 있으면 `frameworkVersion`, `releaseTag`, `source`, `updatedAt` 값을 확인하고 임의로 덮어쓰지 않는다.
- `AGENTS.md`와 `CLAUDE.md`는 프로젝트별 규칙을 담을 수 있으므로 manifest가 바뀌어도 자동 overwrite하지 않는다.
- `.github/ISSUE_TEMPLATE/task.yml`과 `.github/pull_request_template.md`는 저장소별 운영 방식에 맞게 수정됐을 수 있으므로 update PR에서 diff를 리뷰한다.
- `mydocs/manual/`과 `mydocs/skills/`가 symlink인지 복사본인지 확인한다. 본 프레임워크 저장소는 dogfooding을 위해 symlink를 쓰지만, 적용 저장소는 일반 복사본일 수 있다.
- `mydocs/_templates/`는 문서 출력 형식의 진실 원천이므로 기존 사용자 수정 여부를 확인한 뒤 update policy에 따라 처리한다.

## 충돌 가능성

| 영역 | 충돌 이유 | 기본 처리 |
|---|---|---|
| `AGENTS.md` | 프로젝트별 운영 규칙이 추가되는 파일 | `merge`, 수동 리뷰 |
| `CLAUDE.md` | Claude Code 진입 방식이 프로젝트별로 달라질 수 있음 | `merge`, 수동 리뷰 |
| `.github/` | 이슈/PR 템플릿은 저장소 운영 관습에 맞게 수정될 수 있음 | `merge`, 수동 리뷰 |
| `mydocs/manual/` | 사용자가 절차를 로컬 정책에 맞게 보강했을 수 있음 | manifest checksum 일치 시 update, 아니면 충돌 |
| `mydocs/skills/` | Skill 본문은 프레임워크 절차 핵심이지만 로컬 수정 가능성 있음 | manifest checksum 일치 시 update, 아니면 충돌 |
| `.agents/skills`, `.claude/skills` | 파일/폴더로 존재하면 symlink 생성과 충돌 | `symlink`, 기존 경로 확인 후 승인 필요 |
| `.hyper-waterfall/version.json` | 상태 파일이므로 기존 기록을 보존해야 함 | `preserve`, 누락 시 생성 |

## 검증

프레임워크 저장소에서 이번 migration guide 자체는 다음 명령으로 검증한다.

```bash
test -f docs/migrations/README.md
test -f docs/migrations/v0.1.0-to-v0.2.0.md
grep -nE '파일명 규칙|필수 섹션|대상 버전|추가 파일|수정 파일|수동 확인|충돌 가능성|검증' docs/migrations/README.md docs/migrations/v0.1.0-to-v0.2.0.md
grep -nE 'v0.1.0|v0.2.0|manifest|version|GitHub Release' docs/migrations/v0.1.0-to-v0.2.0.md
git diff --check
```

적용 저장소 update PR에서는 다음 확인을 추가한다.

- 현재 `.hyper-waterfall/version.json`의 `frameworkVersion`과 목표 version 비교
- `templates/manifest.json` 기준 source/target 경로 존재 여부 확인
- 사용자 수정 파일과 manifest checksum 불일치 항목 확인
- 수동 확인 항목이 PR 본문에 남았는지 확인

## 후속 작업

- #8 `framework-install` / `framework-update` Skill이 이 문서와 manifest를 참조하도록 구현한다.
- #9 update PR 워크플로우에서 이 migration guide의 수동 확인과 충돌 가능성 섹션을 PR 본문에 요약한다.
- #10 npm CLI MVP는 manifest와 `.hyper-waterfall/version.json`을 읽는 `init`, `update`, `doctor` 명령을 제공한다.
- 실제 `v0.2.0` GitHub Release/tag 생성은 M020 산출물이 안정화된 뒤 별도 승인으로 수행한다.
