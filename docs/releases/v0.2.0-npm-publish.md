# v0.2.0 npm publish 준비 문서

이 문서는 `hyper-waterfall@0.2.0` npm publish 전에 maintainer가 확인해야 할 체크리스트와 smoke 절차를 정리한다. 실제 `npm publish`, `git tag`, `gh release create`는 이 문서만으로 실행하지 않고, 작업지시자 승인 후 별도 단계에서 수행한다.

## 목적

`hyper-waterfall@0.2.0`은 Hyper-Waterfall npm CLI의 첫 public registry 배포 후보이다. npm package는 사용자가 `npx hyper-waterfall`로 lifecycle 판단을 실행하기 쉽게 하는 편의 실행 채널이며, canonical 기준은 계속 GitHub Release/tag + `templates/manifest.json` + migration guide다.

이번 publish 준비의 목표는 다음을 publish 전에 고정하는 것이다.

- package metadata와 포함 파일이 CLI 실행에 충분한지 확인한다.
- `npm pack` tarball 구성과 로컬 tarball smoke 절차를 재현 가능하게 남긴다.
- `v0.2.0` GitHub Release/tag와 `hyper-waterfall@0.2.0`의 순서와 정합성을 확인한다.
- rollback, unpublish, republish 리스크를 publish 승인 전에 검토한다.

## canonical 기준과 publish 순서

| 항목 | 기준 |
|---|---|
| canonical 배포 단위 | `v0.2.0` GitHub Release/tag |
| npm package version | `package.json`의 `0.2.0` |
| npm package 역할 | canonical protocol을 실행하기 쉽게 하는 편의 실행 채널 |
| publish 승인 | 작업지시자의 별도 명시 승인 필요 |

권장 순서:

1. Task #21 PR이 `main`에 merge된 상태에서 publish 후보 commit을 확정한다.
2. `v0.2.0` GitHub tag와 GitHub Release가 생성됐거나, 같은 maintenance window에서 생성될 것을 작업지시자가 명시 승인한다.
3. `package.json` version, `templates/manifest.json`의 `frameworkVersion`, `release.plannedTag`, migration guide 대상 version이 일치하는지 확인한다.
4. `npm pack --dry-run`과 로컬 tarball smoke를 통과시킨다.
5. 작업지시자가 `npm publish` 실행을 별도 승인하면 그때 publish한다.

`v0.2.0` GitHub Release/tag 없이 npm package만 먼저 publish하면 README의 `npx` 예시가 안정 경로처럼 보일 수 있다. 따라서 첫 publish는 GitHub Release/tag 기준과 같은 commit 또는 명시적으로 승인된 순서에서 수행한다.

## 로컬 tarball smoke 절차

아래 절차는 registry에 publish하지 않고 local tarball을 설치해 CLI entrypoint를 확인한다. 임시 디렉터리는 `/private/tmp` 아래에 만들고 repository에는 tarball 산출물을 남기지 않는다.

```bash
PACK_DIR="$(mktemp -d /private/tmp/hyper-waterfall-pack.XXXXXX)"
SMOKE_DIR="$(mktemp -d /private/tmp/hyper-waterfall-smoke.XXXXXX)"
npm pack --pack-destination "$PACK_DIR"
npm install "$PACK_DIR/hyper-waterfall-0.2.0.tgz" --prefix "$SMOKE_DIR"
"$SMOKE_DIR/node_modules/.bin/hyper-waterfall" --version
"$SMOKE_DIR/node_modules/.bin/hyper-waterfall" --help
```

기대 결과:

- [ ] `--version` 출력이 `0.2.0`이다.
- [ ] `--help` 출력에 `init`, `update`, `doctor`가 포함된다.
- [ ] help 하단에 GitHub Release/tag, `templates/manifest.json`, migration guide가 canonical 기준이라는 안내가 포함된다.

필요하면 다음 dry-run 명령까지 실행해 bundled manifest가 package 내부에서 접근 가능한지 확인한다.

```bash
REPO_DIR="$(mktemp -d /private/tmp/hyper-waterfall-target.XXXXXX)"
"$SMOKE_DIR/node_modules/.bin/hyper-waterfall" init --repo "$REPO_DIR" --dry-run
```

기대 결과:

- [ ] 신규 적용 판단 결과가 출력된다.
- [ ] `templates/manifest.json` 기준 적용 후보가 출력된다.
- [ ] 파일 변경 전 승인 요청이 출력된다.

## publish 전 승인 게이트

- [ ] `v0.2.0` GitHub Release/tag 생성 상태 또는 생성 순서가 별도 승인됐다.
- [ ] `npm test`가 통과했다.
- [ ] `npm pack --dry-run`이 통과했다.

- [ ] `npm view hyper-waterfall@0.2.0 version` 결과를 확인했다. 이미 같은 version이 존재하면 publish를 중단하고 상태를 먼저 조사한다.
- [ ] README의 `npx hyper-waterfall` 안내가 실제 publish 상태와 충돌하지 않는다.
- [ ] 작업지시자가 `npm publish` 실행을 명시 승인했다.

## publish 명령 후보

아래 명령은 승인 후 실행 후보일 뿐이다. Stage 구현이나 최종 보고 전에는 실행하지 않는다.

```bash
npm publish
```

npm 계정에 2FA가 필요한 경우 maintainer가 OTP 입력 가능 상태에서 실행한다. OTP를 채팅이나 문서에 기록하지 않는다.

## rollback, unpublish, republish 리스크

| 리스크 | 영향 | 대응 |
|---|---|---|
| 동일 version 재배포 불가 | `0.2.0`을 잘못 publish하면 같은 version으로 덮어쓸 수 없다. | publish 전 tarball smoke와 승인 게이트를 통과시킨다. 필요하면 `0.2.1` 후속 fix를 발행한다. |
| unpublish 제한 | public package를 unpublish하면 사용자와 automation이 깨질 수 있고 npm 정책 제한을 받을 수 있다. | unpublish를 기본 대응으로 보지 않는다. 심각한 경우 작업지시자와 별도 판단한다. |
| dist-tag 오배치 | 의도치 않게 `latest`가 잘못된 version을 가리킬 수 있다. | 첫 publish 후 `npm view hyper-waterfall dist-tags --json`으로 상태를 확인한다. |
| canonical 기준 drift | npm package가 GitHub Release/tag와 다른 commit을 대표하면 적용 저장소 업데이트 기준이 흔들린다. | publish 전 `v0.2.0` release/tag와 package version, manifest 기준을 대조한다. |
| README 상태 오인 | publish 전 `npx` 예시가 이미 사용 가능한 안정 경로처럼 보일 수 있다. | Stage 3에서 README와 배포 채널 문서를 정합성 검토한다. |
