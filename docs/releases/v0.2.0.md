# v0.2.0 GitHub Release/tag 상태 문서

이 문서는 Hyper-Waterfall `v0.2.0` GitHub Release/tag 생성 전 체크리스트와 생성 후 상태 감사를 함께 기록한다. 실제 `git tag`, `gh release create`, `npm publish`는 이 문서만으로 실행하지 않고, 작업지시자 승인 후 별도 task에서 수행했다.

## 목적

`v0.2.0`은 Hyper-Waterfall의 배포 기준을 GitHub Release/tag + `templates/manifest.json` + migration guide로 고정하는 M020 release다. 이 release가 생성되면 기존 적용 저장소는 목표 GitHub Release/tag의 manifest와 `docs/migrations/v0.1.0-to-v0.2.0.md`를 기준으로 업데이트 후보를 판단한다.

## checksum 산출 정책

`templates/manifest.json`의 checksum 상태값은 `v0.2.0` release 전후 정합성 감사 기준으로 다음처럼 해석한다.

| 상태 | 의미 | 이번 release 적용 |
|---|---|---|
| `ready` | 현재 repository의 source 파일 바이트를 SHA-256으로 계산했고 manifest 값과 일치한다. | `kind: file` 12개 |
| `pending-release` | release artifact 또는 공식 산식이 아직 확정되지 않아 값을 비워 둔다. | root checksum, `kind: directory` 3개 |
| `not-applicable` | checksum보다 다른 검증 기준이 우선인 항목이다. | `kind: symlink` 2개 |

`kind: file` checksum은 source 파일 내용을 그대로 SHA-256으로 계산한다. 줄 끝, 인코딩, 권한, 파일명은 별도 정규화하지 않고 repository에 저장된 파일 바이트를 기준으로 한다.

`kind: directory` checksum은 Stage 1에서 후보 digest를 계산했지만 공식 산식으로 채택하지 않았다. directory checksum을 확정하려면 포함 파일 범위, 정렬 기준, 빈 디렉터리, symlink, 권한, 파일명 인코딩 처리 방식을 먼저 문서화해야 한다.

root checksum은 manifest 전체나 release asset 전체 중 무엇을 대표하는지 아직 고정하지 않았다. manifest 내부 checksum 필드를 포함하면 자기참조 문제가 생기므로, 실제 release artifact checksum 또는 root checksum 정규화 규칙이 확정될 때까지 `pending-release`로 둔다.
