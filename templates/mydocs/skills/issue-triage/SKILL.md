# 평문 이슈 도입과 진행 추적

## 목적

사용자, 리뷰어, 외부 기여자가 이미 만든 평문 이슈를 내부 절차 문서로 다시 쓰지 않는다. 원본 보고는 그대로 두고, 워크플로우 메타데이터와 진행 추적용 댓글을 주변에 추가한다.

## 트리거

- 명시 호출만: 작업지시자가 "이슈 #N 도입", "이슈 추적 댓글 갱신", "Stage 완료 댓글 등록"처럼 명시 지시한 경우
- 본 SKILL을 직접 호출한 경우

## 사전 조건

- 추적 대상 GitHub Issue 번호가 존재
- `gh` CLI 인증 완료
- 평문 이슈 원본을 다시 쓰지 않는다는 점에 작업지시자가 동의

## 두 가지 댓글 레이어

이슈 진행 추적은 두 레이어로 나눈다. 영구 기록은 여전히 작업 문서, 커밋, PR 본문에 남는다. 이슈 댓글은 협업자와 미래 핸드오프를 위한 현재 상태 가시성 용도다.

### 1. 현재 상태 레이어 (추적 댓글 1개)

하이퍼-워터폴 추적 댓글 1개로 최신 상태를 보여준다.

- 평문 사용자 이슈는 원본 보고를 보존하기 위해 추적 댓글 형태를 우선한다.
- 정형 내부 이슈는 추적 섹션을 이슈 본문에 둘 수 있다.

권장 필드:

- 현재 상태
- 담당자
- 작업 브랜치 또는 draft PR
- 계획 문서 링크
- 현재 단계
- 단계 체크리스트
- 다음 액션
- 블로커

### 2. 이벤트 로그 레이어 (체크포인트 댓글)

다음 이벤트마다 새 체크포인트 댓글을 추가한다.

- 작업 도입
- 수행계획서 승인 요청
- 구현계획서 승인 요청
- Stage 완료
- 블로커 발견
- 핸드오프 필요
- 최종 보고서 작성
- PR 생성
- merge 후 정리 완료

추적 댓글만 갱신하면 가시성은 좋지만 알림과 감사 추적이 약하다. 체크포인트 댓글만 쌓으면 알림은 강하지만 한눈에 현재 상태를 보기 어렵다. 두 레이어를 함께 사용한다.

## 절차

1. 이슈 본문과 댓글 확인
   ```bash
   gh issue view {N} --repo {REPO_SLUG} \
     --json number,title,state,body,labels,assignees,milestone,comments
   ```
2. 처리 방향 판단: 작업화, 중복, 정보 부족, 보류 중에서 결정
3. 작업으로 도입할 경우
   - 담당자 지정 (필요 시)
   - 라벨, 마일스톤 갱신 (조회된 기존 라벨/열린 마일스톤만 사용)
   - 프로젝트 보드 상태 갱신 (사용 시)
4. 추적 댓글 등록 또는 갱신
   - 처음 도입이면 위 "현재 상태 레이어" 필드를 채워 새 댓글 작성
   - 이미 추적 댓글이 있으면 같은 댓글을 갱신해 최신 상태 유지
5. 이벤트 발생 시 체크포인트 댓글 추가 (이벤트 종류, 시각, 관련 문서 링크 포함)
6. 다음 단계(브랜치 생성, 작업 문서 작성)는 작업지시자 승인 후 `task-start` 또는 후속 SKILL로 분리

## 검증

- 도입 후 이슈에 추적 댓글 1개가 존재한다.
- 추적 댓글에 권장 필드(현재 상태, 담당자, 브랜치/PR, 계획서, 현재 단계, 다음 액션)가 채워져 있다.
- 이벤트 발생 후 해당 이벤트 체크포인트 댓글이 추가되어 있다.
- 라벨/마일스톤은 조회된 기존 항목 중에서만 골라 적용되었다.
- 평문 이슈의 원본 본문은 수정되지 않았다.

## 절대 하지 말 것

- 평문 이슈 원문을 지우거나 크게 다시 쓰기
- 수정·결정이 실제로 완료되기 전에 이슈 close
- 작업지시자 승인 없이 새 라벨, 새 마일스톤 생성
- 이슈를 처음 도입하면서 곧바로 브랜치 생성·작업 문서 작성까지 이어서 진행
- 이미 추적 댓글이 있는데 새 추적 댓글을 또 만들기 (같은 댓글 갱신 우선)
