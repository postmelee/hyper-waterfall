# 오늘할일 보드 작성

## 목적

`mydocs/orders/{yyyymmdd}.md`는 그날의 작업 보드 한 장이다. 계획서, 보고서, 기술 노트로 사용하지 않는다. 마일스톤별 작업 진행 상태를 한눈에 보여주고, 완료 시각을 기록하는 용도다.

## 트리거

- 작업지시자가 "오늘할일 작성", "orders 갱신", "오늘할일 마감"처럼 명시 지시한 경우
- `task-start`, `task-stage-report`, `task-final-report`, `pr-merge-cleanup` 절차에서 오늘할일을 갱신할 때 본 SKILL의 형식을 적용
- 본 SKILL을 직접 호출한 경우

## 산출물

- 갱신 대상: `mydocs/orders/{yyyymmdd}.md` 한 파일
- 작성 언어: 한국어
- 출력 형식 기준: `mydocs/_templates/orders.md`

## 표준 형식

중앙 템플릿 `mydocs/_templates/orders.md`를 우선 기준으로 사용한다. 아래 내용은 템플릿을 읽을 수 없는 상황을 위한 최소 fallback 형식이다.

- 파일명: `yyyymmdd.md`
- 제목: `# 오늘 할일 - YYYY년 M월 D일`
- 본문: 마일스톤 섹션 1개 이상, 필요 시 공통 운영 섹션 추가
- 각 마일스톤 섹션은 다음 표 형식 사용

```md
## M{마일스톤} — {마일스톤 이름}

| Issue | 타스크 | 상태 | 비고 |
|------|--------|------|------|
| #{번호} | {작업} | 예정 | {메모} |
| #{번호} | {작업} | 진행중 | {메모} |
| #{번호} | {작업} | 완료 | {메모} |
```

## 마일스톤 결정 규칙

작업을 적기 전에 어떤 마일스톤 섹션에 둘지 먼저 결정한다.

1. 작업지시자의 명시 지시
2. 명시 지시가 없으면 같은 날짜의 기존 `mydocs/orders/{yyyymmdd}.md`에 이미 잡혀 있는 마일스톤 구조 재사용
3. 같은 날짜 파일이 없으면 대상 저장소의 `README.md` 로드맵·릴리즈 표기에서 명시 마일스톤 라벨을 가져와 사용
4. 어느 쪽으로도 결정할 수 없으면 임의 마일스톤 이름을 만들지 말고 작업지시자에게 확인

주의:

- 로드맵 전환 표기와 마일스톤 이름을 구분한다. 예: `0.5 → 1.0`은 전환 표기이지 섹션 제목이 아니다.
- 작업지시자가 명확히 지정하지 않은 한, 미래 마일스톤에 작업을 배치하지 않는다.
- `README.md`가 한 마일스톤을 `(현재)`로 표시하면 모호한 작업의 디폴트 섹션으로 사용한다.

## 공통 운영 섹션

특정 마일스톤에 명확히 속하지 않는 교차 작업, 운영 작업, 분류 대기 작업은 잘못된 마일스톤에 강제로 넣지 않고 공통 섹션에 둔다.

```md
## 공통 — 운영 작업

| Issue | 타스크 | 상태 | 비고 |
|------|--------|------|------|
| 생성필요 | {작업} | 예정 | {메모} |
```

다음 작업에 사용한다.

- 마일스톤 정의 또는 로드맵 정리
- 릴리즈 또는 CI/CD 계획
- 저장소 전반의 운영 작업
- 분류 전 이슈 생성이 필요한 작업

## 백로그 (선택)

오늘의 주 작업은 아니지만 잊지 말아야 할 미해결 항목이 있을 때만 추가한다.

```md
## 백로그

| No | 내용 | 발견일 | 상태 | 비고 |
|----|------|--------|------|------|
| B-001 | {항목} | YYYY-MM-DD | 미해결 | {메모} |
```

## 작성 규칙

- GitHub Issue 번호가 있으면 `#번호` 형식으로 적는다.
- 아직 이슈가 없으면 임의 번호나 `미정` 대신 `생성필요`로 적는다.
- `타스크`는 한 줄 요약으로 유지한다.
- `비고`는 짧게 유지한다. 목적, 현재 단계, 검증 포인트, 연결 문서, 블로커를 우선한다.
- 상태를 `완료`로 변경하면 비고에 `완료: HH:mm` 형식으로 완료 시각을 추가한다.
- 기본 상태값: `예정`, `진행중`, `완료`, `보류`.
- 백로그가 없으면 백로그 섹션은 생략한다.
- 한 날짜에 여러 마일스톤이 동시에 진행되면 마일스톤별로 섹션을 분리한다.
- `mydocs/orders/`에 템플릿, 임시 메모, 날짜가 아닌 파일을 두지 않는다.

## 검증

- 파일명이 `yyyymmdd.md` 형식이다.
- 제목이 `# 오늘 할일 - YYYY년 M월 D일` 형식이다.
- 모든 작업 행이 `Issue | 타스크 | 상태 | 비고` 표 안에 있다.
- 상태값이 `예정`, `진행중`, `완료`, `보류` 중 하나다.
- 완료 작업의 비고에 `완료: HH:mm`이 들어 있다.

## 절대 하지 말 것

- 계획서, 보고서, 기술 노트 형식의 본문을 `orders/`에 작성
- 마일스톤이 모호하다는 이유로 임의 마일스톤 이름 생성
- 같은 날짜의 기존 마일스톤 구조를 무시하고 새 구조 강제 적용
- 작업지시자 승인 없이 미래 마일스톤에 작업 배치
- 이슈가 있는데도 `생성필요`로 두기

## 최소 템플릿

아래 예시는 `mydocs/_templates/orders.md`의 축약본이다. 새 날짜 파일을 만들 때는 중앙 템플릿을 먼저 확인한다.

```md
# 오늘 할일 - YYYY년 M월 D일

## M{마일스톤} — {마일스톤 이름}

| Issue | 타스크 | 상태 | 비고 |
|------|--------|------|------|
| #{번호} | {작업} | 예정 | {메모} |
```

```md
## 공통 — 운영 작업

| Issue | 타스크 | 상태 | 비고 |
|------|--------|------|------|
| 생성필요 | {작업} | 예정 | {메모} |
```
