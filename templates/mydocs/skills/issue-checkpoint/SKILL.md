# 이슈 진행 추적 댓글

## 목적

작업의 영구 기록은 작업 문서, 커밋, PR 본문에 남는다. GitHub Issue 댓글은 협업자와 미래 핸드오프를 위한 현재 상태 가시성 용도다. 본 SKILL은 이슈에 추적 댓글 1개와 이벤트별 체크포인트 댓글을 추가·갱신하는 절차에만 집중한다.

## 트리거

- 명시 호출만: 작업지시자가 "이슈 추적 댓글 갱신", "Stage 완료 댓글 등록", "이슈 체크포인트 작성"처럼 명시 지시한 경우
- `task-start`, `task-stage-report`, `task-final-report`, `pr-merge-cleanup` 절차에서 이슈 댓글을 추가·갱신할 때 본 SKILL의 형식을 적용
- 본 SKILL을 직접 호출한 경우

## 사전 조건

- 작업 대상 GitHub Issue 번호가 존재
- `gh` CLI 인증 완료

## 두 가지 댓글 레이어

### 1. 추적 댓글 (현재 상태 1개)

이슈에 추적 댓글 1개로 최신 상태를 보여준다. 한 번 만들고 같은 댓글을 갱신해 최신 상태로 유지한다.

권장 필드:

- 현재 상태
- 담당자
- 작업 브랜치 또는 draft PR
- 계획 문서 링크
- 현재 단계
- 단계 체크리스트
- 다음 액션
- 블로커

### 2. 체크포인트 댓글 (이벤트 로그)

다음 이벤트마다 새 체크포인트 댓글을 추가한다.

- 작업 시작
- 수행계획서 승인 요청
- 구현계획서 승인 요청
- Stage 완료
- 블로커 발견
- 핸드오프 필요
- 최종 보고서 작성
- PR 생성
- merge 후 정리 완료

추적 댓글만 갱신하면 가시성은 좋지만 알림과 감사 추적이 약하다. 체크포인트 댓글만 쌓으면 알림은 강하지만 한눈에 현재 상태를 보기 어렵다. 두 레이어를 함께 사용한다.

## 절차

1. 이슈 상태와 기존 추적 댓글 확인
   ```bash
   gh issue view {N} --repo {REPO_SLUG} \
     --json number,title,state,labels,assignees,milestone,comments
   ```
2. 추적 댓글 갱신
   - 추적 댓글이 없으면 위 "추적 댓글" 권장 필드를 채워 새로 작성한다.
   - 추적 댓글이 이미 있으면 같은 댓글 본문을 최신 상태로 갱신한다.
3. 이번 절차에서 발생한 이벤트가 있으면 체크포인트 댓글 추가
   - 이벤트 종류, 시각, 관련 문서 링크를 포함한다.
4. 작업지시자에게 갱신된 추적 댓글 URL과 새 체크포인트 댓글 URL 보고

## 검증

- 이슈에 추적 댓글이 1개만 존재한다 (중복 추적 댓글 없음).
- 추적 댓글 본문에 권장 필드(현재 상태, 담당자, 브랜치/PR, 계획서, 현재 단계, 다음 액션)가 채워져 있다.
- 이벤트가 발생한 절차였다면 해당 이벤트 체크포인트 댓글이 추가되어 있다.

## 절대 하지 말 것

- 이미 추적 댓글이 있는데 새 추적 댓글을 또 만들기 (같은 댓글 갱신 우선)
- 이슈 본문 자체를 수정·재작성
- 본 SKILL 안에서 이슈 close, 새 라벨 생성, 마일스톤 변경
- 이슈 등록·작업 시작·브랜치 생성 같이 다른 SKILL의 책임 범위까지 처리
